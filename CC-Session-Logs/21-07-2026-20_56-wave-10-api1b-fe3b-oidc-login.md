## Key Learnings
- **When a read can't be resolved read-only, check whether the value was available at WRITE time and stamp it forward** — often far cheaper than reshaping the read. (API-1b: the write-boundary succeeds exactly where API-1's read-boundary was refuted.)
- **A live-network dependency-audit gate (pip-audit/OSV) is a flakiness source** — it can go red on an unchanged tree when the advisory service blips or a new advisory lands. Mitigate with a retry, and diagnose "red on a docs-only commit" as transient before treating it as real.
- **A docs-linter that greps for a phrase must anchor on STRUCTURE (line-start), not substring** — prose describing the rule will otherwise match the rule's own trigger (the closure-check caught itself).
- **`(push)` and `(pull_request)` CI runs are SEPARATE required contexts on the same commit** — a stale failed `(push)` run can block a merge even when the fresh `(pull_request)` run is green; a new head SHA (or a re-run) clears it. `mergeable_state:"unstable"` was NOT reliably "mergeable" here.
- **Never put backticks / `$(…)` in a `git commit -m` message** (shell command-substitution); use `-F`.
- **The pre-ratification verifier + 4-finder review keep paying off:** the verifier reshaped API-1b (CLAIM 3 footprint) and FE-3b (the JWKS Docker fix) BEFORE implementation; the FE-3b 4-finder review caught a HIGH — the ratified demo login literally could not complete (nginx 404) — that all green gates missed.

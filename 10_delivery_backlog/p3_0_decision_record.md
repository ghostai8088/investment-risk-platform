## Part 2 — Decision detail

### OD-P3-0-A — P3 entry point → methodology + model-governance hardening (carrying the first number)
The P2-closeout readiness review evaluated six entry options (A…F). **VaR/ES first (F) is rejected** — it has the highest methodology, model-governance, and data-depth bar and depends on covariance + history that do not yet exist. A pure governance-only slice (methodology + registry, no number) is *safe* but unproven. So P3-1 = **the governance scaffolding PROVEN by the lowest-dependency reproducible number** (analytic sensitivities, OD-P3-0-B): it lands the methodology framework (OD-P3-0-C), the model_version-for-risk contract (OD-P3-0-D/E), the `RISK.*`/`risk.*` reservations (OD-P3-0-H/I), the `COMPONENT_KIND_CURVE` mint (OD-P3-0-G), and the snapshot+run+model_version+IA-result output contract (OD-P3-0-F/M) — and proves them end-to-end on a number that needs no new captured input. This de-risks every later subphase (they inherit a proven contract). The full sequence is the `p3_implementation_plan.md` subphase map.

### OD-P3-0-B — first computed risk output → analytic sensitivities
Analytic sensitivities (duration / DV01 / spread-duration) are a **closed-form, deterministic** function of the captured curve nodes + positions — reproducible to the bit (no seeded MC, no estimation window, no history). They are therefore the **most testable first risk number** (REQ-MKT-002: "greeks reproduce within ε; conventions declared") and exercise the full output contract (snapshot-pin the curve + positions, bind a run + a model_version carrying the convention methodology, write an IA result, lineage `snapshot→run→result`) without any unbuilt input. **NOT VaR/ES** (the highest bar; needs covariance + history). **NOT factor exposures** (need factor returns, not captured). The conventions a sensitivity declares (bump size, day-count, compounding, which curve) are **model/methodology choices** — hence a sensitivity run carries a `model_version` (OD-P3-0-E), not `code_version`-only.

### OD-P3-0-C — methodology documentation framework
The model registry already has `model_version.methodology_ref` (a pointer) but there is **no framework** for what it points to. P3-0 ratifies: the EXISTING governed methodology home **`05_analytics_methodologies/`** holding **one versioned doc per risk method/version**, each linked from `methodology_ref`. Required sections (the template P3-1 instantiates): purpose & applicability; inputs + data policy; formulas + numerical standards (rounding/quantization, day-count, compounding, bump convention); estimation window; assumptions (mirrored into `model_assumption`); limitations (mirrored into `model_limitation`, CTRL-014); validation / benchmark / reproduction tests (the acceptance ε per REQ-MKT-*); known limitations + scope-out. A risk method **may not ship without its methodology doc + a model-inventory entry** — basis: **REQ-MKT-001 acceptance** ("methodology doc + inventory entry") + **CTRL-002/CTRL-003** (REQ-MKT-002's own acceptance is "conventions declared", not a doc clause). P3-0 creates the **framework/template only** — no method's doc.

### OD-P3-0-D — model governance stance
The registry is a **ready referent** but its validation/approval columns are **non-enforcing** (reserved for P7). P3 stance: (1) **CTRL-003 inventory-before-use is load-bearing** — a model-driven run resolves its `model_version` through `assert_registered_model_version` (an unregistered version fails closed); (2) **`methodology_ref` is mandatory** on a risk `model_version`; (3) **`model_version_id` is required** whenever a model is involved (OD-P3-0-E); (4) **validation approval is NOT enforced in P3** — `validation_status` defaults `UNVALIDATED` and is recorded but does not gate execution (P7 owns the maker-checker validation workflow). This keeps the governance gate honest from the first risk number while explicitly deferring the *enforced* validation workflow to its ratified phase — recorded, not silent.

### OD-P3-0-E — `code_version` vs `model_version`
**`code_version`** anchors the deterministic algorithm (mandatory on every run — model-driven or not). **`model_version_id`** anchors the registered model + its methodology/assumptions/limitations. The dividing line is **whether a methodology/convention/estimation choice is applied**: if yes → `model_version` (mandatory + registered). Analytic sensitivities apply conventions (REQ-MKT-002 "conventions declared") → **model_version**. Factor/covariance/VaR/ES apply estimation → **model_version**. **`code_version`-ALONE** is reserved for a convention-free deterministic transform — the **only** precedent is the P2-3 `exposure_aggregate` rollup (`model_version_id` N/A-with-rationale; a sham model_version is forbidden). This rule prevents a risk number from ever escaping model governance by claiming "it's just arithmetic".

### OD-P3-0-F — snapshot / run binding (the P3 output contract)
Extends the AD-014/FW-RUN/TR-15 gate (realized at P2-3) verbatim to risk: **no official risk result row without** (a) a non-null `input_snapshot_id` pinning the consumed inputs, (b) a non-null `run_id` over a complete `calculation_run`, (c) `code_version` + `environment_id` on the run, (d) a registered `model_version_id` where a model applies, (e) captured assumptions/limitations, (f) a snapshot-only compute (no live `reconstruct_*`/`resolve_*` read — import-fenced, the P2-3 precedent) so the result is **reproducible under a later input correction**, and (g) IA-append-only result rows. The failure model mirrors P2-3 (pre-create refusal = zero run/result/audit; post-create FAILED = a committed FAILED run + `CALC.RUN_STATUS_CHANGE(outcome='failure')` + zero result rows). **The P2 snapshot/run controls are not weakened.**

### OD-P3-0-G — required snapshot component kinds
P2-3 minted `COMPONENT_KIND_FX` exactly when exposure needed to pin FX; P2-4/5/6 left price/curve/benchmark **readiness-only**. P3 follows: each risk method **additively mints the `COMPONENT_KIND_*` it consumes** (extend `SNAPSHOT_COMPONENT_KINDS` + `build_snapshot`; the `dataset_snapshot`/`component` tables are **unchanged**). P3-1 sensitivities mint **`COMPONENT_KIND_CURVE`** (pin the curve nodes + the positions, which are already pinnable). Later: `PRICE`/`FX` as a method consumes them, `BENCHMARK` for benchmark-relative, `RATING`/`VOLATILITY_SURFACE` once those inputs exist. **None minted in P3-0.**

### OD-P3-0-H — `RISK.*` audit family
A risk run **is** a `calculation_run`, so the run lifecycle **reuses the shipped `CALC.RUN_CREATE`/`CALC.RUN_STATUS_CHANGE`** (the P2-3 precedent — no new run-lifecycle audit code). For risk-specific signals a slice may need, **reserve a `RISK.*` family at a new EVT block (proposed EVT-220)**, defaulting — like P2-3's `EXPOSURE.*`/EVT-210 — to **NO result-create event** (results are run-tracked + lineaged). Activation is per-slice via R-07; **caller-side constants only; `audit/service.py` stays FROZEN**. P3-0 reserves the names + the EVT block; activates nothing.

### OD-P3-0-I — `risk.*` entitlements
**Reserve** `risk.view` + `risk.run` (the read + the governed-compute verb). Recipient sets (the `exposure.*` precedent): `.run` → `data_steward`/`risk_analyst_1l`/`platform_admin`; `.view` → `risk_analyst_1l`/`risk_manager_2l`/`data_steward`/`platform_admin` + **`auditor_3l`** (the governed-output read SoD established at P2-3). Deny-by-default `require_permission`. **No `bootstrap.py`/catalog change in P3-0** — the codes mint at the implementation slice (R-07).

### OD-P3-0-J — P3 result entities (roadmap only)
The canonical model already reserves `factor_return` (ENT-025), `risk_result` (ENT-027), `sensitivity`/`exposure_metric` (ENT-028), `scenario_definition` (ENT-029). P3 realizes: **sensitivity** → ENT-028 (P3-1); **factor_return** → ENT-025 (P3-2); **factor_exposure** → ENT-028 family or a net-new id (P3-3); **covariance_matrix** → **net-new canonical id** (P3-4, minted via Part-3 at that slice — not a free build choice); **var_result/ES** → ENT-027 (P3-5); **stress_result** → **ENT-030 `scenario_result`** (realize — the reserved "Run-tracked scenario outputs") + `scenario_definition` ENT-029 (P3-6). The only genuine net-new mints are `covariance_matrix` (P3-4) and a net-new `factor_exposure` if ENT-028 can't carry it (P3-3). **P3-0 records the mapping only — no entity, migration, model, or code.**

### OD-P3-0-K — captured-data gaps
Recorded honestly, per affected subphase: **`volatility_surface` (ENT-022) absent** → no options/vega risk, no options VaR (a prerequisite captured-data slice before any vol-sensitive risk); **adjusted/total-return prices absent** (P2-4 is RAW) → price-derived factor returns + equity total-return analytics need an adjusted series first; **`rating` assignments absent** → credit risk (REQ-CRD) is gated; **`benchmark_level`/`return` deferred** → return-based benchmark analytics gated (membership-based active risk is NOT). **DECISION: none blocks P3-1** (analytic rates/credit-spread sensitivities use the captured `curve`/`curve_point`); each is a named prerequisite for its later subphase. A captured-data gap is filled by a **P2-style captured-data slice** (a captured vendor input), NOT by computing the missing data inside a risk run.

### OD-P3-0-L — data-history requirements
Non-binding targets (3y/5y/10y/15–20y); **store as much clean history as available** (additive, no migration). **Mandatory:** each risk `model_version` declares its estimation window + data policy (frequency / lookback / decay-or-window / missing-data / outlier / adjustment) in `methodology_ref` + assumptions — so a result is interpretable + reproducible. **Not load-bearing for P3-1** (point-in-time analytic sensitivities); load-bearing at P3-2 (factor returns) + P3-4 (covariance).

### OD-P3-0-M — temporal class for risk results
Risk results are **IA TRUE append-only** (ENT-027; the `exposure_aggregate` precedent: run-bound, a re-run = new run + new rows, never an edit; `APPEND_ONLY_TABLES` + P0001 trigger + ORM guard). `scenario_definition` = **EV** (versioned saved assumptions, BR-8). `factor_return` = captured-or-derived (IA if a derived run-output / FR if a captured vendor series; decided at P3-2). "Append-only unless explicitly justified otherwise" (the user's requirement) is satisfied — the default is IA append-only, and any deviation is a recorded justification at the slice.

### OD-P3-0-N — deferrals
Stress/scenario (REQ-MKT-004, RTM-P5 — late P3 or P5); credit risk (REQ-CRD, needs ratings); counterparty PFE (REQ-CPT-002); the model-validation workflow enforcement (P7); performance attribution; reporting/dashboards/risk UI (separately planned frontend); limits/breach + ABAC enforcement (P6+). Recorded so the roadmap does not silently pull them forward.

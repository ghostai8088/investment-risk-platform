## Part 2 — Decision detail (the non-obvious ones)

### OD-P3-4-A/B — why factor sample covariance is the honest v1
The covariance families in reach are (i) **asset/instrument covariance** — blocked: instrument return history requires adjusted/total-return prices (P2-4 is RAW; the OD-P3-0-K register), and computing returns from raw prices inside a risk run is exactly the "compute the missing captured input inside the run" move P3-0 forbids; (ii) **decayed/shrunk estimators** — genuine estimation-methodology upgrades that deserve their own declared `model_version`s once the sample floor exists; (iii) **factor sample covariance** — computable TODAY from the P3-2 captured series with zero new inputs, PSD by construction, exactly reproducible, and precisely the substrate P3-5's parametric VaR needs. The v1 is (iii), with (i) and (ii) recorded as named prerequisites/extensions — the same honest-narrowing pattern as OD-P3-1-A and OD-P3-3-A. The slice title's "volatility estimation" is delivered by the matrix **diagonal** (variances), not by a separate vol model.

### OD-P3-4-C/D — ENT-051 minted; element rows, not a blob
OD-P3-0-J fixed `covariance_matrix` as the one remaining genuine net-new canonical mint (P3-3 declined its optional mint by realizing under ENT-028 — the fork closed as designed). The matrix is stored **relationally as upper-triangle element rows** (the platform stores atoms, not blobs: queryable per-pair, RLS-enforced per-row, lineage-addressable) — a JSON matrix payload would defeat per-element auditability and the append-only diff story. The binder-enforced `factor_id_1 <= factor_id_2` ordering (no CHECK — controlled-vocab/genericity rule) makes each unordered pair unique; symmetry is structural (the mirror element is the same row), so "symmetric within ε" is not even a test surface — it is exact.

### OD-P3-4-G — the window as registration identity (the P3-3 conflict lesson, extended)
OD-P3-0-L requires the `model_version` to declare its estimation window. If the window were a free per-run parameter, two runs of "the same v1 model" could use different windows — the declared-window governance would be decorative. If it were hard-coded, tests could not exercise realistic windows. The resolution: `window_observations` is a **registration argument recorded as a model_assumption and folded into the version-resolution identity** — re-registering the same `version_label` with a different window (or `code_version`) raises `ModelVersionConflictError` (409), the exact conflict semantics the 2026-07 review installed for `code_version`. A tenant wanting a different window mints a new `version_label` — a new immutable inventory identity, as it should be.

### OD-P3-4-I — the FACTOR_RETURN component: series-per-factor, FR-immutable content
The pin must make the run reproducible under later vendor supersede/correction (TR-09). Each pinned row is an immutable FR **version** (corrections close it out and insert successors — they never mutate it), so capturing the ordered window rows per factor gives byte-stable re-verification through per-row id re-reads — the `curve` header+nodes shape, with the `factor` head as the component target. One component per FACTOR (not per return row) keeps a 60-observation × 30-factor snapshot at 60 components (30 FACTOR + 30 FACTOR_RETURN), not 1,830. The kind is minted HERE because P3-4 is its first consumer — completing the OD-P3-2-G readiness note exactly as designed, and honoring the anti-over-mint rule that kept it out of P3-2/P3-3.

### OD-P3-4-N — why numpy, and why test-only
The standing rule ("external ground truth over self-consistency; dual-path verification for estimation methods") exists precisely for this slice: a Decimal kernel whose tests re-derive expectations with the same Decimal arithmetic proves only self-consistency. An eigenvalue check and an independent covariance implementation need a numerical library; `numpy` is the standard, and confining it to `requirements-dev.txt` + a shared-package import fence keeps the runtime pure-Decimal (QS-01) while making the verification honest. Hand-computed small-matrix references (computed at planning/review time, outside both implementations) remain the third leg.

### OD-P3-4-O — paying the duplication debt before compounding it
The 2026-07 review named three copy-clusters at their tipping points. P3-4 unavoidably needs a fourth snapshot builder and a fifth completeness gate; writing them as copies would be a deliberate regression against a filed finding. The R0 pre-step is deliberately minimal — two mechanical extractions with zero behavior change, protected by the full 970+ test suite and the now-complete CI PG coverage — and is a **separate commit** so the P3-4 implementation diff stays reviewable. The run-scaffold extraction stays deferred: it entangles failure-timing semantics (the safety-critical invariant) and is not required to avoid new duplication.

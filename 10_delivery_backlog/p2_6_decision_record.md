## Part 2 — Decision detail

### OD-P2-6-O — P3-conditionality resolution → P3 dependency CONFIRMED; capture inputs only
`p2_implementation_plan.md`:26 marks P2-6 "Benchmark / index **(conditional)**" and :113 "benchmark definitions/constituents + levels, **only if a P3 factor dependency is confirmed**." This is a ratified gate, not a build detail — proceeding is a governance decision. **RESOLVED (user sign-off 2026-06-29): the P3 dependency is CONFIRMED.** The confirmed dependency is the **P3 benchmark-relative / active-risk analytics + factor/risk-engine on-ramp**: active return + tracking error (relative performance/risk), the factor model (ENT-025 `factor_return`), and the risk engine (ENT-027 `risk_result`, ENT-028 `sensitivity`; REQ-MKT-001 VaR/ES, REQ-MKT-002 sensitivities) — all of which need **captured benchmark membership + weights, as-of**, to compute anything benchmark-relative. The load-bearing, correctly-grounded evidence: REQ-PUB-003's business purpose is "**Credit & relative risk inputs**", its Data is "...benchmark constituents", and its acceptance is exactly "**Inputs present & as-of for the risk engine**" (`requirements_backbone.md`:149); REQ-MKT-001's RTM row even lists CAP-3 as an input dependency. *(The earlier-draft citation of REQ-PPM-004's "Aggregation benchmark" was struck — review fold, product MED: that is REQ-PPM-004's QA **test-oracle** column, not a dependency on ENT-009 benchmark-index data; REQ-PPM-004 is basic exposure from positions/valuations/FX and consumes no benchmark constituents.)* So P2-6 lands the **as-of input foundation** the confirmed P3 work consumes, and **nothing more** — the analytics themselves (active return, active risk, tracking error, factor exposure, covariance, VaR/ES, stress, scenario, performance, benchmark-relative reporting) are **all out of P2-6** (the hard exclusions, §2 of the plan). This keeps the captured-data slice cleanly separated from the P3 compute it enables (the `price_point`→exposure, `curve`→discounting precedent: capture the input now, compute later).

### OD-P2-6-A — entity model → `benchmark` EV header + `benchmark_constituent` FR membership
The canonical model lists **ENT-009 `benchmark`** as "Benchmark definitions **and constituents**" — a single canonical entity spanning the index *identity* (code/name/family/currency/methodology) and its *membership* (the constituent instruments + weights, effective-dated). The cleanest realization is a **two-table header+detail**: a `benchmark` metadata header + a `benchmark_constituent` membership detail — both bookkept under **ENT-009** (the curve_point/`dataset_snapshot_component` precedent: a detail table bookkept under its parent canonical entity, **no new canonical id minted**, no canonical over-expansion; OQ-P2-6-A-id). The header+detail split mirrors `dataset_snapshot` (ENT-049 header) + `dataset_snapshot_component` (ENT-050 detail). Captured index **levels** (`benchmark_level`, FR) and **returns** (`benchmark_return`) are a **different shape** (a per-date time-series value, not membership) and a **net-new canonical ENT id** — **DEFERRED** (OD-P2-6-K). v1 scope = **metadata + constituents** (the user-confirmed bound). *(The governance amendment recording ENT-009 → `benchmark`/`benchmark_constituent` is Part 3; the `benchmark_constituent` ENT-id bookkeeping is OQ-P2-6-A-id below.)*

### OD-P2-6-B — temporal class → benchmark EV; constituents FR/bitemporal
A benchmark **definition** (name, index family, denomination currency, methodology label) is slowly-changing **reference data** — the canonical model classes ENT-009 **EV** and the user confirmed "metadata likely EV." So `benchmark` is **EV** (entity-versioned; `record_version` the drift discriminator; the `portfolio` / instrument-reference EV precedent; symmetric RLS; NOT append-only). The **membership**, by contrast, is captured **as-of** and re-versions at each reconstitution (and corrects as-known) — exactly the captured-market-data shape — so `benchmark_constituent` is **FR / bitemporal** (`FullReproducibleMixin`; capture → effective-dated supersede → as-known correction → `reconstruct_membership_as_of(valid_at, known_at)` on both axes; NOT in `APPEND_ONLY_TABLES`; content-immutability service-enforced; the `price_point`/`curve` protocol). The membership is captured/superseded/corrected **as a set** per `(benchmark, effective_date)` (the `curve` node-set atomicity — a vendor publishes/restates the whole constituent set, not one name in isolation; a set-level write keeps "which weights belong together as-of T" unambiguous and lets the weight-sum DQ — when later enabled — measure a coherent set).

**This is a NOVEL composite (review-watch — architect/data-arch):** an **EV metadata header + FR detail** is not a shipped combination (the shipped header+detail pairs share a class — `dataset_snapshot`/`component` both IA; `curve`/`curve_point` are FR/IA). It fuses the EV reference-metadata pattern (header) with the FR captured-membership protocol (detail). The correction/supersede semantics live on the **constituent set**, not the EV header: `supersede_membership(benchmark, effective_date, …)` closes the prior open set's `valid_to` and writes a fresh set; `correct_membership(…)` closes the prior set's `system_to` and writes a corrected set over the same valid period — the EV `benchmark` header is untouched by a membership op (it versions only when the *definition* changes, via `MARKET.BENCHMARK_UPDATE`). **Reproducibility asymmetry (review fold — architect/data-arch LOW):** because the EV header is system-time-versioned only (`record_version` is the drift discriminator; no `valid_to`/`system_to` as-of), "reproduce the benchmark as-of T" is **asymmetric by design** — the FR membership is bitemporally reconstructable (`reconstruct_membership_as_of`), while the EV definition metadata is current-state + `record_version`-drift-detected (a header restatement is drift-flagged, not system-time-reconstructable). This is exactly how `portfolio`/`instrument` EV behaves; a future snapshot pin of a benchmark definition follows the EV-component convention (`record_version` + `captured_content`, NULL `system_from` — `canonical_data_model_standard.md`:162). No v1 design change. *(OQ-P2-6-1 records the uniform all-FR alternative — make `benchmark` FR too, for fully-symmetric as-of metadata reproducibility — rejected for v1 as the definition is slowly-changing reference data the canonical model already classes EV.)*

### OD-P2-6-C — benchmark grain (header logical key, v1)
The v1 EV current-identity key is **`(tenant_id, benchmark_code, benchmark_source)`** — one current `benchmark` per `(code, vendor)`; `benchmark_source` is **in the key** (a benchmark sourced from two vendors coexists — the `price_source`/`curve_source` precedent). All promoted key columns are **DB-level `NOT NULL`** (the `price_point` lesson). The EV `record_version` carries the definition's version history.

### OD-P2-6-D — constituent grain (v1)
The v1 current-head partial-unique is **`(tenant_id, benchmark_id, instrument_id, effective_date) WHERE valid_to IS NULL AND system_to IS NULL`** — exactly one open weight per (benchmark version × instrument × effective date); the FR axes version the membership for the fixed key. Columns: `benchmark_id` (NOT-NULL FK → `benchmark.id`), `instrument_id` (NOT-NULL FK via `resolve_instrument`, OD-P2-6-H), `effective_date` (separate immutable logical-key `Date`, OD-P2-6-E), `weight` (`Numeric(20,12)`, OD-P2-6-G), `constituent_currency` (nullable captured String(3), OD-P2-6-I), `tenant_id` (RLS — server-stamped from the parent header). A given `(benchmark, effective_date)` membership is the **set** of constituent rows sharing that pair; the binder writes/closes the whole set as one governed unit.

### OD-P2-6-E — `effective_date` convention → a separate immutable logical key
`effective_date` is a **separate immutable logical-key `Date`** — the membership/reconstitution date the constituent set is FOR — carried forward verbatim by supersede/correct, **never mutated**, DISTINCT from the FR `valid_from` axis (the `curve_date`/`valuation_date`/`price_date` precedent). Many `effective_date`s per `(benchmark)` coexist as separate open membership sets; `reconstruct_membership_as_of` selects the set true-at the as-of.

### OD-P2-6-F — benchmark identifiers → code-in-key + captured attributes; no resolution engine
`benchmark_code` (the tenant's logical code, e.g. `"SPX"`, `"AGG"`) and `benchmark_source` (the vendor label) are the **key**; `vendor_code` (the vendor's own identifier), `index_family` (e.g. `"S&P"`, `"Bloomberg"`), and `methodology_label` (a nullable inert metadata String — the `interpolation_method` precedent: captured, no engine consumes it) are captured **attributes**. v1 treats all of these as **opaque captured identifiers** — the platform stores what identifies the benchmark but does **NOT** resolve/cross-walk vendor codes or arbitrate identifier ambiguity (the `price_point` "no identifier-resolution engine" precedent). A scope-fence asserts no identifier-resolution symbol consumes them.

### OD-P2-6-G — weight convention → canonical decimal; sanity band; sum-completeness deferred
**`weight` is a canonical DECIMAL fraction** — never percent, never bps. A 5% index weight is stored `0.050000000000`; 12.5bp is `0.001250000000`. **`Numeric(20,12)`** (the `curve.point_value` rate-precision scale — sub-bp weight precision; NOT money scale). v1 DQ is a **non-negative sanity `RANGE` `[0, 1]`** per constituent (`{column:"weight", min:0, max:1}`; one constituent's weight is between 0 and 100%). **Weight-sum-to-≈1.0 completeness is DEFERRED** (OQ-P2-6-4): a set-level "Σ weights ≈ 1.0 within tolerance" check is meaningful only over an atomically-coherent set, and FR per-name corrections (a single instrument's weight superseded independently) make a transient Σ≠1.0 legitimate — so v1 captures weights with a per-row sanity band and leaves set-sum completeness to a later, set-grained DQ slice. **Signed/leveraged benchmark weights** (short legs, >100% gross) are a future extension — v1 is long-only captured weights `[0,1]`.

### OD-P2-6-H — relationship to instrument → NOT-NULL FK via resolve_instrument
`benchmark_constituent.instrument_id` is a **NOT-NULL FK** via the reused hybrid-aware `resolve_instrument` (the `price_point` precedent) — a constituent references a **mastered** instrument (cross-tenant fails closed). v1 requires the instrument resolved at capture (an unknown instrument → `InstrumentNotVisible` 404, fail closed). **Capturing an opaque constituent identifier for not-yet-mastered benchmark names is DEFERRED** (OQ-P2-6-3): vendor benchmark membership can include instruments a tenant doesn't yet master, and a future additive slice may add an opaque `constituent_ref` fallback — but v1 keeps the strict FK (consistency with `price_point`; a benchmark whose names are all mastered is the v1 contract), and a scope-fence asserts the FK + resolver path.

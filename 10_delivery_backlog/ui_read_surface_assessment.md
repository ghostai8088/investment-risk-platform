## Part 1 — Findings (file:line-grounded)

### F1 — The governed read surface is RUN-CENTRIC; screens (and agents) need ENTITY/TIME-CENTRIC reads. THE compounding gap.

Every governed-result read requires a `calculation_run_id` or `result_id` already in hand: all 13 result families read ONLY via `GET /{family}/runs/{run_id}` or `GET /{family}/{result_id}` (e.g. `api/risk.py:1519/:1548` for VaR; `:2160/:2189` backtests; `api/perf.py:407/:436`). **No endpoint anywhere filters a governed result by `portfolio_id`, `as_of`, or a date range; no "latest result for portfolio P" resolver; no metric time-series; no cross-family summary.** `portfolio_id`/as-of appear only on run-CREATION bodies and row OUTPUT DTOs, never as read filters — the run listings (`/risk/runs` `api/risk.py:346`, `/perf/runs` `api/perf.py:188`, `/exposure/runs` `api/exposure.py:208`) filter by `run_type`+`status` only, `limit`/`offset`, `{items}` with no total/cursor (`list_risk_runs`, `risk/queries.py:53-61`). Charting VaR over 30 as-ofs = find 30 run ids by paging a global list, then 30 fetches. **The captured families prove the platform knows the right shape**: `GET /positions/as-of?portfolio_id&instrument_id&valid_at&known_at` (`api/positions.py:242-247`), `GET /valuations?portfolio_id&instrument_id&valuation_date` (`api/valuations.py:227-231`), `GET /portfolios/{id}/holdings?valid_at&subtree&…` (`api/holdings.py:124-137`), `GET /portfolios/{id}/tree` (`api/portfolios.py:173`) — entity/time-centric reads exist for inputs but were never built for governed outputs, because for sixteen consecutive slices the only consumers were tests and a run-detail page that already held the id. **Each new governed family adds one more family in the wrong shape — this is the part that compounds** (CC-2 would have made it one wider). It is also NOT only a UI problem: thesis §2.3 commits to "every capability … exposed through governed APIs" and "every governance artifact … machine-readable" — an agent asking "portfolio P's tail risk this quarter" hits the same wall a screen does; completing the read surface discharges an obligation the strategy already made, for both consumers at once.

### F2 — The governance story (the differentiator) is the LEAST readable part of the API.

- Validation **findings and evidence are write-only**: accepted on POST (`api/models.py:271-272`) and returned by NO read endpoint — `ValidationSummaryOut` (`models.py:283-292`) omits both. A recorded finding is a finding no consumer can ever see.
- **No audit read endpoint exists at all** (no audit router in `main.py:40-63`; the audit trail is write-only over HTTP).
- **No snapshot listing** — `GET /snapshots/{id}` + `/verify` only (`api/snapshots.py:187/:206`); you must already hold the id.
- The model inventory list **omits `tier`** (`ModelSummary`, `models.py:76-82`); tier + validation_status live only on the per-model detail (`models.py:111-112`); no filter-by-tier. Lineage exposes a single edge by id (`api/lineage.py:39`); `GET /dq/results` is unfiltered (`api/dq.py:122`).
- Assumptions/limitations ARE exposed (rides on model detail, `models.py:189-206`) — the one governance surface already readable.

### F3 — FE types are hand-written mirrors, by an expired policy, and the drift has bitten three times.

`types.ts:1-6`: "Hand-written mirrors … (FE-1, OD-FE-1-G — no codegen; **the surface is five endpoints**)". The surface is now 12 registered families across a four-structure hand-synced registry (`FAMILIES` `:45-82`, `RUN_TYPE_TO_FAMILY` `:86-99`, `runDetailUrl` `:104-114`, `FAMILY_ROW_COLUMNS` `:119-261`), guarded only by `types.test.ts`. The recorded **FL-1 "FE drift trio"** (`delivery_roadmap.md:146`) is direct evidence: a dead-end family registration (the missing single-row proxy-weight GET — still absent), two missing VaR columns, and the ES `z×σ ≠ var_value` display — "the only place a governed number presents arithmetic that does not check out" was the UI, on a platform whose pitch is verifiability. FastAPI already serves `/openapi.json` (`main.py:39`, defaults untouched); generated types would close the whole drift class.

### F4 — Identity is a dev shim; nothing here can face a non-developer until SSO lands.

`deps.py:50-56` (`get_principal`) trusts `X-User-Id`/`X-Tenant-Id` headers — self-labeled "DEV PLACEHOLDER … Real identity comes from OIDC/SSO (AD-007)"; `session.ts:3-5`: "the identity is UNVERIFIED and this is NOT a security boundary." Enforcement BEHIND the shim is real (deny-by-default `require_permission`, `deps.py:78-91`; symmetric FORCE RLS) — the missing piece is verified identity only. Already themed at RTM-P9 (roadmap Part 3): "the dev header shim is replaced HERE at the latest, before anything internet-facing."

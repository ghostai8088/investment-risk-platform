## 1. Requirements included

P1A-3 implements **exactly one** Step-2 backbone requirement and **builds** one forward dependency:

- **REQ-DQR-001 — Data-quality rules engine** (backbone §7 CAP-13.1 "DQ rules"; RTM row 76; Phase **P1**; persona **P-DS** Data Steward (PERSONA-05/H-04, Platform/1L LoD; owns CAP-13 per the personas table; UJ-4 step 1 "DQ rules run on ingest"); controls **CTRL-027** (DQ rules run on ingest) + **CTRL-029** (stale/missing flagged, not silently filled); audit **BX-AUD**; entitlement **BX-ENT**; lineage column `—` because DQ validates data and is **not** itself a lineage-recorded data output — it BUILDS the rail). Backbone acceptance (row L231): *"Configurable DQ rules run on ingest; rules run on load; exceptions raised, not silent."*
- **DEP-DQF** — P1A-3 *is* the build of the data-quality forward dependency (RTM §4 rollup: first needed P1; required-by REQ-DQR-\*, REQ-SMR-\*, REQ-PUB-\*, REQ-PRV-\*, REQ-INT-\*). Confirms downstream consumers depend on a **stable DQ rule/result/gate contract**.

**Precise in-scope statement.** Establish the data-quality data model (`data_quality_rule` head + immutable `data_quality_result`), a pluggable `DQRule.evaluate(...) -> DQResult` interface with **1-2 generic rules only**, a result-capture utility, and the no-silent-failure gate, plus retrieve-the-results reads. Concretely the slice ships:

1. `data_quality_rule` (ENT-039, **EV**, tenant-scoped) with DR-P1-3 nullable, non-enforcing maker-checker hook columns and a controlled-vocab `rule_type` + declarative `params` config (§3); `SYSTEM_TENANT_ID` may hold global/template rules per AD-013 (mechanism available, not seeded).
2. `data_quality_result` (ENT-039, **IA**, immutable run output) — append-only, with a **NULLABLE** intra-tenant `data_source` reference (the thing validated) and a **NULLABLE no-FK** `ingestion_batch_id` placeholder reserved for P1A-4.
3. A pluggable `DQRule.evaluate(...) -> DQResult` interface dispatched by a small code registry keyed by `rule_type`, with **exactly 1-2 GENERIC rules** registered: `not_null` and `allowed_values`. NO domain-specific rules.
4. A result-capture utility `run_quality_check()` that evaluates a rule and persists a `data_quality_result` co-transactionally with the audit write.
5. `assert_passed_quality_checks()` — the no-silent-failure GATE utility (the analog of P1A-1's `assert_has_lineage` / P1A-2's `assert_registered_model_version`) that a **FUTURE P1A-4 ingestion** calls; P1A-3 implements **no ingestion/upload**.
6. A governed-write `POST /dq/rules` plus `GET /dq/results` read endpoints, and (resolved §4) `GET /dq/rules` + `GET /dq/rules/{id}` read symmetry.
7. **No new permission** — reuse the existing `dq.rule.manage` / `dq.result.view` (bootstrap.py:28-29) with **zero** role-template changes (§7).
8. One Alembic migration (`0006_data_quality_skeleton`, revises `0005_model_registry_skeleton`) creating both tables with FORCE RLS + tenant-isolation policy and append-only enforcement on the IA result table.

This realizes the headline **no-silent-failure** guarantee (CTRL-029 / QS-15 / QS-16 / QS-06 / BR-14): a failing rule MUST surface as a raised exception OR a persisted flagged result, never silently pass — at **skeleton level only**, on generic rules, with no reconciliation, override, or exception-management workflow. REQ-AUD-001 is **satisfied cross-cutting** (the rule-create path and the rule-run path emit taxonomy events) and is **not** scoped as a P1A-3 requirement of its own (verified by the shared audit-coverage enforcement test across P1A-1…4, CTRL-012).

**Backbone-shorthand reconciliation.** Backbone §7 CAP-13 (L231) tags Data as `data_quality_rule, dq_result (IA)` — shorthand that wrongly implies the `data_quality_rule` head is IA. P1A-3 intentionally refines this to `data_quality_rule = EV` (the rule mutates `params`/`severity`/`is_active` over time as the steward edits it) and `data_quality_result = IA` (immutable run output). This is a legitimate decomposition authorized by temporal_reproducibility_standard.md §2A (which explicitly lists "DQ and reconciliation results (ENT-039, ENT-040)" under IA), not a conflict — see §11/§18 for the backbone correction, exactly mirroring the P1A-1 `data_source=EV` and P1A-2 `model=EV` reconciliations.

## 3. Proposed database entities

Two new tables, one migration (`0006_data_quality_skeleton`, revises `0005_model_registry_skeleton`). Both are tenant-scoped and carry FORCE RLS + a `tenant_isolation_<table>` policy reusing the migration `0004`/`0005` `TENANT_SCOPED_TABLES` loop verbatim (USING **and** explicit WITH CHECK). ORM models follow the established `model/models.py` / `lineage/models.py` pattern: `class X(PrimaryKeyMixin, TenantMixin, <TemporalMixin>, Base)` with an explicit `__temporal_class__`. Named `pk_`/`ix_`/`uq_`/`fk_` constraints satisfy the `alembic check` drift gate + NAMING_CONVENTION. Grep confirms the slice is **fully additive** (zero existing `data_quality_*` / `DQRule` / `DQResult` artifacts).

**Genericity contract (load-bearing).** `rule_type` is a **controlled-vocabulary `String(50)` — NOT a Python `Enum`, NOT a DB `CHECK`/enum type** — so new generic rule families register by supplying a **new value + a code-registry evaluator entry, never a schema migration**. This mirrors the domain-agnostic polymorphic pattern already proven in-repo by `lineage_edge.source_type`, `audit_event.entity_type`, and `model.model_type`. Rule config lives in a `params` JSON column; the validated target is described by a **polymorphic no-FK** `(target_entity_type, target_entity_id)` descriptor on the result. **No table carries a foreign key to any domain/analytical table, to the model registry, or to `lineage_edge`.** This is the single point of architectural leverage that lets DQ extend to P1A-4 ingestion, Security Master, Portfolio, Market Data, Private Assets, and Risk Analytics with **zero schema change**.

```

NULLABLE so P1A-3 stands alone (result before a source is associated, and non-source-bound runs). |
| `ingestion_batch_id` | GUID | yes | **FUTURE placeholder, NO FK now** (P1A-4 owns `ingestion_batch`; a FK now would be a forward/circular dependency). Plain nullable GUID, logical reference, documented "reserved for P1A-4." |

No `created_by`/`updated_by` (IA — actor attribution lives in the `DATA.VALIDATE` audit event; `system_from` is the record timestamp). No second (valid) axis (TR-21). Add to migration `APPEND_ONLY_TABLES` (the existing `irp_prevent_mutation()` trigger, P0001) **AND** an ORM `before_update`/`before_delete` `_block_mutation` guard (mirror `lineage/models.py` + `model/models.py`).

> **Intra-context FK note.** `data_quality_result.rule_id` and `data_quality_result.data_source_id` are **real FKs** because both target tables live in the same bounded context (BC-12 data governance), all rows share `tenant_id` (stamped server-side), and this does **not** violate ARCH-P-01/DM-N-09. **Caveat (P1A-1/P1A-2 lesson):** an FK does **not** enforce same-tenant integrity across the RLS boundary — `run_quality_check()` MUST resolve the rule (and any `data_source`) through the RLS-scoped session and stamp the result's `tenant_id` server-side from the resolved rule (§8). The `ingestion_batch_id` placeholder is a plain column (no `ForeignKey`) in BOTH the ORM and the migration; declaring it as an FK now would break migration ordering / `alembic check` and forward-couple P1A-3 to P1A-4.

### 3.3 IA append-only enforcement (both layers)

Add `('data_quality_result',)` to the migration's `APPEND_ONLY_TABLES` so the existing `irp_prevent_mutation()` trigger (from `0001`) blocks UPDATE/DELETE at the DB layer (P0001), **and** add an ORM `event.listen(DataQualityResult, 'before_update'/'before_delete', _block_mutation)` guard raising `AppendOnlyViolation` (mirroring `model/models.py`) so IA is provable on both SQLite-local and Postgres-CI. `data_quality_rule` (EV) gets **neither** — a negative-control test (§13) proves the trigger is not over-applied to the mutable rule head. Making the flagged result immutable is itself part of the no-silent-failure guarantee: a `FAIL`/`WARN` flag cannot be silently cleared (CTRL-029).

## 4. Proposed API surfaces

**Resolution (OQ-P1A-3-API, §16): expose a governed `POST /dq/rules` write endpoint plus `GET /dq/results` and `GET /dq/rules`(/`{id}`) read endpoints; expose NO public rule-execution endpoint.**

| Surface | Method | Auth | Behavior |
|---|---|---|---|
| `POST /dq/rules` | write | `require_permission('dq.rule.manage')` under `get_tenant_session` | Create a `data_quality_rule` in one tenant-scoped transaction. Thin wrapper over the rule-create utility; server-stamps `tenant_id`. |
| `GET /dq/rules` | read | `require_permission('dq.rule.manage')` under `get_tenant_session` | RLS-scoped rule inventory list (caller's tenant only). A steward who manages rules must be able to list them; reuses an existing code (no new permission). |
| `GET /dq/rules/{id}` | read | `require_permission('dq.rule.manage')` under `get_tenant_session` | Rule detail; `uuid.UUID` path param; indistinguishable 404. |
| `GET /dq/results` | read | `require_permission('dq.result.view')` under `get_tenant_session` | RLS-scoped result list (caller's tenant only), deterministically ordered. |

**Why `GET /dq/rules` is in scope (resolving the open §6-plan question).** The Chief Architect, Data-Architecture, QA, and Scope lenses recommend including the rule reads for operability/read-symmetry (a steward who can manage rules must be able to read them back), mirroring P1A-2's `GET /models` + `GET /models/{id}`. The Security lens cautioned against inventing a permission. **Decision: include `GET /dq/rules` + `GET /dq/rules/{id}` gated on the EXISTING `dq.rule.manage`** (not `dq.result.view`, which is semantically a results gate, and NOT a new `dq.rule.view` code). Low cost, no new permission, same pattern as the model registry.

**Hard guardrails (Scope/Security lens):** REJECT any `POST /ingest*` / `POST /dq/run-on-upload` / multipart file-upload endpoint (P1A-4); REJECT any `POST /dq/reconcile` / `/dq/recon*` endpoint (REQ-DQR-002/P7); REJECT any `POST/PUT /dq/results/{id}/override|waive|acknowledge|resolve` or any exception-disposition endpoint (REQ-DQR-003/P7); REJECT any `/dq/dashboard` or DQ-visualization endpoint (P7); REJECT any **public `POST /dq/run`** rule-execution endpoint — execution is an in-process library call driven by P1A-4 ingestion; a public run endpoint would pre-build the engine and invite domain coupling and contract drift.

**Endpoint requirements (Security lens):**
- MUST depend on `get_tenant_session` (sets `app.current_tenant`), **not** `get_db`, so RLS scopes/stamps every read/write to the caller's tenant.
- `POST /dq/rules` **never** reads `tenant_id` from the body (server-stamps from principal/context); a forged/foreign `tenant_id` is ignored and backstopped by `WITH CHECK` → SQLSTATE 42501. Audits the rule-create code (§6) in the **same** transaction (fail-closed rollback).
- For reads, entitlement (403) is checked **first**, then the RLS-scoped lookup yields **404** for both "not found" and "exists in another tenant" — the two cases must be **indistinguishable** (fixed detail body, no existence/oracle leak; never 200 for a cross-tenant id). `GET /dq/results` and `GET /dq/rules` lists return only the caller's tenant's rows.
- Malformed `uuid` path param → uniform **422** before any DB hit; missing principal headers → **401**.
- The `observed_value`/`detail` payload on a result is protected by the same row-level tenant policy as every other column — cross-tenant data echo is impossible as long as the row's `tenant_id` is correct and `WITH CHECK` is enforced.

## 5. Worker / CLI impact

- **No standalone worker or CLI in P1A-3.** No DQ-batch job, no DQ-export CLI, no scheduler (those are reporting/ingestion concerns, P1A-4/P7). No scheduled-job/async test in this slice.
- **The `DQRule.evaluate(...) -> DQResult` interface + evaluator registry + `run_quality_check()` + `assert_passed_quality_checks()` MUST live in a NEW `irp_shared.dq` package** (shared-python), a **sibling to `irp_shared.lineage`, `irp_shared.model`, and `irp_shared.calc`**, **not** in `apps/backend`. Placing them in backend would couple the future P1A-4 ingestion worker and any batch job to the web app and break the single-contract guarantee — this is the P1A-1 R3 / P1A-2 R7 service-boundary trap applied to DQ.
- **Dependency direction is strictly one-way: P1A-4 ingestion → `irp_shared.dq`.** `irp_shared.dq` MUST import only `irp_shared.audit` (+ `db`/temporal); it MUST NOT import any ingestion/backend module, `irp_shared.lineage`, or `irp_shared.model`. An import-direction test guards the no-circular-dependency contract (the `data_source` FK target type is the only lineage-package symbol the result model touches, and only for the FK declaration).
- Concrete contract: (a) a `DQRule.evaluate(context) -> DQResult` pluggable interface with a small evaluator REGISTRY keyed by `rule_type` (exactly two generic evaluators registered: `not_null`, `allowed_values`); (b) `run_quality_check(session, *, rule, target_*, actor_id)` that evaluates and persists a `DataQualityResult` co-transactionally and emits `DATA.VALIDATE`; (c) `assert_passed_quality_checks(session, target_entity_type, target_entity_id, *, tenant_id=None)` → raises `QualityCheckFailedError` on any blocking-severity fail (the gate). The utilities take a **caller-managed `Session`**, run **co-transactionally** with the audit write, **stamp `tenant_id` server-side**, and MUST NOT COMMIT/ROLLBACK mid-call (the single-transaction invariant; a mid-call commit drops the transaction-local GUC and the next write fails closed). `assert_passed_quality_checks` is RLS-scoped by default with an optional explicit `tenant_id` param for SQLite logic-tests (mirror `assert_has_lineage` / `assert_registered_model_version`).
- The slice's primary downstream deliverable is the single contract (`run_quality_check` + `assert_passed_quality_checks`) that P1A-4 ingestion (and every future caller) will call. `evaluate()` is typed to **always return a structured `DQResult`** with an explicit `passed`/`severity` so failure is structurally un-swallowable.

## 6. Audit events — resolved

**Decision (resolves OQ-P1A-3-AUDIT): reuse the existing `DATA.VALIDATE` code for the rule run/result; mint two new DATA-family codes `DATA.DQ_RULE_DEFINE` (create) + `DATA.DQ_RULE_UPDATE` (update) for rule definition CRUD; do NOT overload `CONFIG.CHANGE`.**

The DATA category already exists (audit_event_taxonomy.md L61, EVT-020…): `DATA.INGEST`, `.VALIDATE`, `.CORRECTION`, `.RECONCILE`, `.PURGE`, `.SOURCE_REGISTER` (EVT-026), `.SOURCE_UPDATE` (EVT-027); `LINEAGE.RECORD` (EVT-028) is reserved. The next free DATA slots are **EVT-029** (`DATA.DQ_RULE_DEFINE`) and **EVT-030** (`DATA.DQ_RULE_UPDATE`) — confirm the exact EVT assignment with R-07 at slice exit.

| Governed write | Event code | Notes |
|---|---|---|
| `data_quality_rule` create | **NEW `DATA.DQ_RULE_DEFINE`** (EVT-029) | `record_event(session, event_type='DATA.DQ_RULE_DEFINE', tenant_id=<ctx>, actor_type='user'|'agent', actor_id=<principal>, source_module='dataquality', entity_type='data_quality_rule', entity_id=<rule.id>, action='create', after_value={code,name,rule_type,target_entity_type,severity,approval_status:null}, data_classification='DC-1')` |
| `data_quality_rule` update | **NEW `DATA.DQ_RULE_UPDATE`** (EVT-030) | `action='update'` with before/after diff. The rule is EV, so config edits are first-class versioned changes that MUST be audited (TR-06/TR-07), mirroring `DATA.SOURCE_UPDATE`. |
| `data_quality_result` capture (rule run, PASS or FAIL or ERROR) | **EXISTING `DATA.VALIDATE`** (audit_event_taxonomy.md DATA row) | `entity_type='data_quality_result'`, `entity_id=<result.id>`, `action='validate'`; `after_value={rule_id, target_ref, outcome, severity, data_source_id:nullable, ingestion_batch_id:null}`; **`outcome='success'` for PASS, `outcome='failure'` for a failing OR errored rule** — the execution error is itself audited, never dropped. |

**Decision & rationale (mirror P1A-1 option C / P1A-2 OQ-AUDIT).** Reuse `DATA.VALIDATE` for the run/result (it already exists for exactly this purpose). Mint **two dedicated DATA-family codes for rule definition** rather than overloading `CONFIG.CHANGE`. P1A-1 set the precedent: a config root that controls/data are bound to (there, `data_source`; here, the DQ rule) gets a dedicated code so CTRL-005/CTRL-027 completeness can be asserted directly, and `CONFIG.CHANGE` is not buried with deploy/provenance noise. Cost is two taxonomy rows; `event_type` is a free-form string (no code-level allowlist in `audit/service.py record_event` — confirmed) so no existing assertion breaks. Keep the codes under `DATA.*` (the DATA category already owns the DQ lifecycle via `.VALIDATE`); do NOT start a `DQ.*` family.

**No per-result-detail/per-flagged-field event proliferation:** one `DATA.VALIDATE` per rule run. The `lineage_edge` precedent (metadata of a governed write, no own event) does NOT apply to results — a `data_quality_result` is a first-class governed output and MUST emit `DATA.VALIDATE` on capture.

**Reserved-but-unused in P1A-3:** `DATA.RECONCILE` stays reserved for REQ-DQR-002/P7 (CTRL-028) — do **not** emit it. `DATA.CORRECTION` stays reserved for restatement/override (REQ-DQR-003/P7) — do **not** emit it. A negative scope-fence test asserts neither is emitted on any P1A-3 path.

**Fail-closed (AUD-04 / CTRL-005 / CTRL-012 / CTRL-032):** each `record_event` MUST occur in the **same tenant-scoped transaction** as the `data_quality_rule`/`data_quality_result` row (mirror `register_data_source`/`register_model`: add row → flush → `record_event` → return, no mid-call commit); if the audit insert is rejected (RLS 42501 or hash-chain failure) the registry/result row MUST roll back. `record_event` itself writes to the tenant-scoped + FORCE-RLS `audit_event`, inheriting the same fail-closed-without-context behavior. A DQ-execution **error** MUST be audited via `DATA.VALIDATE` with `outcome='failure'`, never swallowed — a bare `try/except` that drops a rule error is a CTRL-029 violation and must fail review.

**Correlation:** a DQ run triggered by a future P1A-4 ingestion should share the ingestion's `correlation_id` so an auditor can join `DATA.INGEST` → `DATA.VALIDATE` without extra events.

**AI-authorship (BR-16):** when a rule is defined or a run is initiated by an AI-agent principal, the event MUST carry `actor_type='agent'` + `agent_model`/`agent_model_version` + `on_behalf_of` (the audit schema already supports these); the utilities pass these through.

**Note:** `event_type` is a free-form string (no code-level allowlist), so a typo passes silently — the cross-cutting audit-coverage test (CTRL-012) MUST assert the **literal** `DATA.DQ_RULE_DEFINE`/`DATA.DQ_RULE_UPDATE`/`DATA.VALIDATE` codes and `entity_type`. Consider a shared event-code constants module (as `model/service.py` already does with `MODEL_REGISTER_EVENT`).

## 8. RLS / tenant-context behavior (built on P1A-0)

Both tables are tenant-scoped and rely entirely on the P1A-0 wiring (AD-015/AD-016): per-session `app.current_tenant` via `set_config(..., is_local=true)` (transaction-local) + durable pool `RESET`; `get_tenant_session` dependency; `run_in_tenant` for worker paths. All P1A-3 DB surfaces MUST run under this context; missing context **fails closed**.

Confirmed behaviors (mirroring `test_lineage_pg.py` / `test_model_registry_pg.py` under the constrained non-superuser `irp_app` role):
- **Policy form (identical to migrations `0004`/`0005`):** `ENABLE` + `FORCE` ROW LEVEL SECURITY (FORCE so the app role is also subject to RLS, BR-17/AD-013) + `tenant_isolation_<t>` with `USING (tenant_id::text = current_setting('app.current_tenant', true))` **and** explicit `WITH CHECK (...)` (resolved — OQ-P1A-3-SEC-1; mandatory because these are new write-bearing tables, the result's `observed_value`/`detail` payload can echo source values, and the SYSTEM_TENANT_ID global-rule option introduces read/write-scope divergence). `USING`-only would only HIDE cross-tenant rows; the `WITH CHECK` is what REJECTS a cross-tenant/forged-tenant INSERT.
- **No-context fail-closed:** with the GUC unset, `current_setting('app.current_tenant', true)` returns NULL → reads return empty, INSERTs rejected with **SQLSTATE 42501** on both tables (seed a real parent rule first for the result-table test so the rejection is RLS, not FK).
- **Tenant-mismatch write denied:** under context A, inserting `tenant_id=B` is rejected by `USING`/`WITH CHECK` (42501); `tenant_id` is server-stamped, never caller-supplied.
- **Parent→child same-tenant binding:** `run_quality_check()` MUST **resolve the rule through the RLS-scoped session** (a cross-tenant `rule_id` → zero rows → fail closed, mirroring `record_lineage`'s `DataSourceNotVisible` / `register_model_version`'s `ModelNotVisible`) and **stamp the result's `tenant_id` server-side from the resolved rule** — never from caller input. When a `data_source_id` is supplied, it MUST likewise be resolved through the RLS-scoped session so a cross-tenant `data_source` id fails closed rather than creating a dangling cross-tenant reference; cross-tenant child inserts are also backstopped by `WITH CHECK`.
- **Single-transaction invariant:** the utilities MUST NOT COMMIT/ROLLBACK mid-call in a request scope.

**Ops-role isolation (AD-015):** the BYPASSRLS `irp_ops` role MUST have **no** grant (SELECT/INSERT/UPDATE/DELETE) on `data_quality_rule`/`data_quality_result`; no P1A-3 code path connects via the ops `DATABASE_URL`; the app DB role stays NOSUPERUSER NOBYPASSRLS. A regression test asserts `has_table_privilege('irp_ops', <table>, …)` is False (mirroring `test_ops_role_has_no_grant_on_lineage_tables`). The PG-CI fixture must `GRANT SELECT,INSERT` on `data_quality_rule` and `SELECT,INSERT,UPDATE,DELETE` on `data_quality_result` (UPDATE/DELETE so the append-only rejection is the **trigger** P0001, not a 42501 privilege denial) to `irp_app` (+ `audit_event` already granted; + `SELECT` on `data_source` if referenced).

**SYSTEM_TENANT_ID global/template rules (OQ-P1A-3-3 resolution):** **tenant-scoped by default**; the mechanism for global/template rules under the reserved `SYSTEM_TENANT_ID` (`00000000-0000-0000-0000-000000000001`, already in bootstrap.py) is **available with FORCE RLS retained** (no RLS-exempt table), but **no system-tenant rules are seeded or enabled in P1A-3** and **cross-tenant READ is NOT widened** (that would require `tenant_id = current OR tenant_id = SYSTEM_TENANT_ID` in the `USING` clause and is a deliberate, separately-tested later change). A global rule is writable ONLY under system-tenant context (enforced by `WITH CHECK`). A global rule's `DATA.DQ_RULE_DEFINE` event is written to the SYSTEM_TENANT_ID chain; a RESULT produced by running a global rule is audited under the RUNNING tenant's chain (the run happens in tenant context). This mirrors the resolved P1A-1 OQ-P1A-1-2 / P1A-2 OQ-P1A-2-1 and keeps strict tenant-equality; it is never a doorway to general global data.

## 9. Lineage behavior

Data quality is **NOT a data output and is NOT lineage-recorded** — `record_lineage()` is **not** called for `run_quality_check()` (confirmed: RTM row 76 lineage column = `—`). There is **no schema change to `lineage_edge`** and **no import dependency** between `irp_shared.dq` and `irp_shared.lineage`. If any reviewer proposes that P1A-3 call `record_lineage` for a DQ result, reject it — that conflates DQ with lineage capture and pulls forward P1A-4 integration.

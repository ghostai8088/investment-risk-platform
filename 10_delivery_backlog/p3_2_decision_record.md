## Part 2 — Decision detail (the non-obvious ones)

### OD-P3-2-A/F — captured-only v1 (the honest input boundary)
A factor return can be (i) **captured** from a vendor/provider (Bloomberg/MSCI/Axioma/etc.) or (ii) **computed** on-platform (from adjusted prices via a return definition, or estimated via regression). Computing requires the two **unbuilt** inputs — **adjusted/total-return prices** (P2-4 captured RAW only) — plus a **return/estimation methodology + a registered `model_version`**. None exist. So v1 is **captured-only**: factor returns enter as a governed captured market-data series, exactly like FX/curve/benchmark. This unblocks the entire downstream factor stack (P3-3/P3-4) on real inputs **without** first building a price-adjustment + return-computation engine. Computed returns are a clean, separately-governed later slice.

### OD-P3-2-B/C — the `factor` (EV) + `factor_return` (FR) shape
A factor **universe** needs a stable definition (what the factor IS: its source, family, type, scope, frequency) distinct from its **time series** (the dated returns). This is exactly the `benchmark` (EV definition) + `benchmark_constituent` (FR series) shape — the cleanest, already-precedented model. `factor` = EV (versioned; a definition amend bumps `record_version`, `REFERENCE.*` audited). `factor_return` = FR bitemporal (a vendor restatement supersedes/corrects a dated return without mutating history — the `fx_rate` protocol). This gives **reproducibility-by-capture**: a `reconstruct_factor_return_as_of` on both axes returns the return as-known-at any time, so a later vendor correction can't change a historical covariance/VaR that pinned the earlier version.

### OD-P3-2-H/I — an INPUT, not a governed derived number
The load-bearing distinction: factor returns are **captured inputs**, so — like all P2 market data — they bind **no `calculation_run`** and **no `model_version`**. The run/model/snapshot governance contract (OD-P3-1-F) attaches to **derived risk numbers** (factor exposures P3-3, covariance P3-4, VaR P3-5), which *consume* these inputs. Keeping factor returns model-less + run-less is correct and consistent; forcing a run/model onto a captured vendor series would be a sham (the OD-P3-1-C rule: `code_version`/`model_version` bind *computed*/model-driven outputs, not captured inputs).

### OD-P3-2-J/K — MARKET.*/REFERENCE.* + marketdata.*, not a new family
Factor returns are captured market data → they belong in the **existing** captured-data governance surfaces: `MARKET.FACTOR_RETURN_*` (the FR series, `fx_rate` precedent) + `REFERENCE.*` (the EV definition, `benchmark` precedent), entitled by the **existing** `marketdata.view`/`.ingest`. This is the **least-new-surface** choice and is consistent with every other captured input. A net-new `FACTOR.*` audit family or `factor.*` permission is **not** justified for a captured input (reserve that vocabulary for genuine factor *analytics* if ever needed). `RISK.*` is explicitly wrong here (it is for risk *results*).

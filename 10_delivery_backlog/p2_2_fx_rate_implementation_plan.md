## 1. Requirements included
- **Captured FX market data** (FR) per **QS-07/08/09 + OD-030 + OD-P2-E**: a vendor-published `fx_rate` time-series, queryable as-of on **both** bitemporal axes, with effective-dated supersede + as-known correction.
- **Explicit currency-pair direction** (`base`/`quote` + `rate`, QS-08) and a **controlled `rate_type`** (MID v1, QS-09).
- **A pure published-rate `convert` helper** — direct lookup, read-time reciprocal, and **ratified triangulation-through-the-configured-base** (QS-08/OD-030) — the arithmetic P2-3's multi-currency exposure rollup needs.
- **The market input the AD-014 first compute needs**: `convert(captured mark → base currency, as-of)`. P2-2 provides the rate + the conversion; **P2-3** consumes it inside a snapshot-bound, run-tracked exposure.
- **Governed write + provenance**: VENDOR `data_source` ORIGIN lineage per physical version; `MARKET.FX_*` audit; symmetric tenant-scoped RLS; positive-rate + required-field DQ.

## 2. Requirements excluded (hard)
- **NO currency-conversion analytics**, **NO exposure calculation**, **NO `exposure_aggregate`**, **NO `calculation_run` wiring** (readiness only; the consuming run is P2-3).
- **NO model/curve/return/vol-implied or model-derived rates** — `convert` is *defined arithmetic over published rates only*. **NO bid/ask spread modeling** (MID only). **NO interpolation** of missing dates/pairs (a missing rate fails closed).
- **NO `dataset_snapshot` changes** (readiness notes only). **NO price history (P2-4) / curves (P2-5) / benchmark (P2-6) / factor model / risk / VaR / ES / pricing model / valuation model.**
- **NO market-data ingestion *pipeline*** (vendor SFTP/API adapters → REQ-INT-002/003/P9). P2-2 ships a **governed canonical-write API** (capture/supersede/correct), not a feed ingester. (Bulk staging→canonical mapping reuse, if any, is a small additive step — recommended **deferred** to keep P2-2 tight; §21 OD-P2-2-C.)
- **NO broad DQ-protocol rewrite, NO broad lineage rewrite, NO `audit/service.py` change, NO BYPASSRLS app path, NO hybrid/SYSTEM_TENANT** (symmetric only, unless a future AD-013-R2).
- **NO P2-3/P2-4/P2-5/P2-6 or P3+ work.** No REQ closes at P2-2 (it is an enabling market-data prerequisite for REQ-PPM-004 exposure).

## 3. Proposed entities / modules
- **`fx_rate`** (ENT-024, FR) — the single new table; **migration `0017_fx_rate`** (the next after `0016_dataset_snapshot`). The **`rate` column is `Numeric(28, 12)`** (NOT the money scale 6) — FX rates routinely need >6 dp (small-unit pairs, and read-time reciprocal/triangulation precision, §9); there is **no canonical FX scale to inherit**, so P2-2 fixes it explicitly here (QS-05 "rate convention must be explicit"). This concrete scale is what makes the §10/§20 snapshot byte-stability guarantee verifiable; P2-3's FX snapshot pin adds a matching `_SCALE_FX_RATE = 12` to `snapshot/serialize.py`.
- **New `irp_shared/marketdata/` package** (the first shared market-data package; FX is tenant #1):
  - `models.py` — `FxRate` (FR ORM, registered in `irp_shared.models`).
  - `events.py` — the `MARKET.FX_*` audit constants + `FxRateActor` + caller-side emitters to the **FROZEN** `record_event`.
  - `service.py` — the FR binder: `capture_fx_rate` / `supersede_fx_rate` / `correct_fx_rate` / `reconstruct_fx_rate_as_of` / `resolve_fx_rate`; VENDOR `data_source` lineage; the required-field + positive-rate DQ gate.
  - `convert.py` — `convert(amount, from_ccy, to_ccy, valid_at, known_at=None, base=DEFAULT_BASE)` + `resolve_rate_path` (direct / reciprocal / triangulated); pure published-rate arithmetic; fail-closed.
  - `__init__.py` — one-way exports. Imports `{reference, lineage, dq, audit, db}`; **imports NO `calc`, NO `snapshot`, NO `exposure`**; nothing imports `marketdata` (import-direction test).
- **New backend `api/marketdata.py`** (or `api/fx.py`) — the thin endpoints (§12).
- **One new generic DQ evaluator** `range` in `dq/rules.py` (the first new evaluator since the 2 shipped — §17).
- **Two additive entitlement codes** `marketdata.view` / `marketdata.ingest` in `entitlement/bootstrap.py` (seeded by migration `0002` on fresh migrate).
- **Reference dependency + a NEW `resolve_currency` (net-new P2-2 work):** `base_currency`/`quote_currency` are validated against the shipped **`currency`** vocabulary (P1B-1, the AD-013-R1 hybrid SYSTEM/tenant table). **No by-code currency resolver ships today** (only `reference/service.py dedupe_tenant_wins`), so P2-2 **adds** `resolve_currency(session, code, *, acting_tenant)` → `CurrencyNotVisible` (404) in `reference/`. **It MUST be hybrid-aware, NOT a verbatim copy of the symmetric by-id `resolve_*` (`tenant_id == acting_tenant`):** because `currency` is hybrid, the predicate admits **(own-tenant OR `SYSTEM_TENANT`)** codes (the RLS `USING(own OR SYSTEM)` arm — the `dedupe_tenant_wins` precedent) and rejects **foreign-tenant** ones. A symmetric `tenant_id == acting_tenant` copy would reject the SYSTEM `USD`/`EUR` base and **break triangulation** — an explicit anti-pattern this slice must avoid. **No change to the `currency` table or its hybrid policy** (closed 5-table hybrid set asserted unchanged).

## 4. Temporal classification
**FR — Full Reproducible / bitemporal** (`FullReproducibleMixin`: `valid_from`/`valid_to` + `system_from`/`system_to`; `__temporal_class__ = FULL_REPRODUCIBLE`, BR-19). Reuses the P1C-4 `valuation` protocol **verbatim**: **NOT append-only** (NOT in `APPEND_ONLY_TABLES`, no `irp_prevent_mutation` trigger, no ORM guard — close-out UPDATEs are required by the FR protocol; prior-version content immutability is **service-enforced + test-proven**). Mapping of OD-P2-E's "`rate_date` = valid-time as-of": `rate_date` is the **immutable logical-key business date** the rate is FOR (the `valuation_date` precedent — see §5/§7); the **FR valid-time axis** carries effective-dating (supersede) and the **FR system-time axis** carries vendor as-known corrections. You query by `(rate_date, valid_at, known_at)`.

## 5. FX pair convention
- The **logical key** of a rate is `(base_currency, quote_currency, rate_date, rate_type)` within a tenant.
- **`rate_date`** is a **separate immutable `Date` logical key** (NOT the FR `valid_from`), the business date the rate is FOR — exactly the `valuation_date` precedent. **Why separate (load-bearing):** a vendor re-quote/correction is FOR the *same* `rate_date`; if `rate_date` were `valid_from`, a supersede (new valid period) would change the date it is "FOR", breaking the semantics. The separate key lets many versions (corrections/re-quotes) of the rate FOR a given date coexist on the bitemporal axes.
- **Current-head partial-unique (5-part):** `(tenant_id, base_currency, quote_currency, rate_date, rate_type) WHERE valid_to IS NULL AND system_to IS NULL` — exactly one open head per pair+date+type; many pairs/dates/types coexist as separate heads (the `valuation` 4-part-key pattern, +`rate_type`).
- **Pair storage:** the pair is stored **as the vendor quotes it** (e.g. `EUR/USD`), with the explicit direction in `base`/`quote`. The reciprocal (`USD/EUR`) is **derived at read time** by `convert` (1/rate), **not** stored as a second row — avoiding two-rows-one-truth drift. (A pair MAY also be stored in both directions if a vendor publishes both; the DQ pair-consistency gate (§17) can flag `rate(A→B) × rate(B→A) ≉ 1` within tolerance, but no auto-reconciliation.)

## 6. Base / quote / direction convention
- `rate` ≔ **price of one `base_currency` in `quote_currency`** ("1 base = rate quote"). Self-describing per row; no global inversion convention to remember.
- **Same-currency** convert is the identity (rate ≡ 1, no row lookup).
- **Reciprocal**: `rate(quote→base) = 1 / rate(base→quote)` at read time.
- **Base/pivot currency** (for triangulation) is **configurable, USD default** (QS-07); it is a `convert`-call / portfolio property, **not** a column on `fx_rate`.

## 7. Rate-type convention
- `rate_type` is a **controlled-vocab plain String** (no enum/CHECK; app-side allow-list), **`MID` only in v1**. `BID`/`ASK`/`CLOSE`/`WMR` are **reserved vocabulary, not minted** (a future slice mints them with a one-line allow-list extension + a parity test). The DQ + `convert` paths assume MID; a non-MID `rate_type` is out-of-vocab → rejected (422 at the API, `FxRateValueError` at the service) with no DB write.

## 8. Source / provenance model
- **`rate_source`** — an **inert provenance label** column (nullable String; e.g. `"ECB"`, `"WMR_4PM_LON"`) surfaced verbatim; **NOT** an FK, **NOT** a market-data join key (the `valuation.mark_source` precedent).
- **Governed lineage** — a registered **VENDOR `data_source`** (`source_type` token e.g. `VENDOR_FX`) → **one ORIGIN `lineage_edge` per NEW physical version** (capture / new-open-row-of-supersede / correction each root one edge; the prior-head close-out roots NONE). **Reuses the shipped `record_lineage(source: DataSource, …)` UNCHANGED — no new lineage code** (unlike P2-1, which needed `record_internal_lineage`; FX's source IS a `data_source`). `ensure_vendor_source` is a thin resolve-or-register helper (the `ensure_manual_source` precedent), fail-closed; `assert_has_lineage` in tests.

## 9. Triangulation policy
- `convert(amount, from, to, valid_at, known_at=None, base=DEFAULT_BASE)`:
  1. `from == to` → return `amount` (identity).
  2. **Direct**: a head rate for `(from, to, rate_date, MID)` as-of `(valid_at, known_at)` → `amount × rate`.
  3. **Reciprocal**: a head rate for `(to, from, …)` → `amount × (1 / rate)`.
  4. **Triangulated** (neither directly quoted): `from → base → to` via two as-of lookups (each direct-or-reciprocal): `amount × rate(from→base) × rate(base→to)`. Pivot = the configured `base` (USD default).
  5. **No path** (a required leg missing as-of) → **`FxRateNotFound`** (fail closed) — never a silent `1.0`, never an interpolation.
- **`rate_date` selection (OD-P2-2-D APPROVED):** `convert` resolves each leg by **exact `rate_date == date(valid_at)`** in v1 (the resolver returns the head for that `(rate_date, valid_at, known_at)`); a missing exact-date rate **fails closed** (`FxRateNotFound`) — **"most-recent-on-or-before" is deferred** (it is a stale-rate policy and is not introduced unspecified).
- **The fence (§17):** `convert`/`marketdata` import or call **no** return/vol/curve/yield/spread/model/factor symbol and **no** P3 package; `ast.Mult` is **permitted** (it IS the published-rate arithmetic) — so the fence is **vocabulary/import + a positive correctness test**, NOT a no-Mult fence (the P2-3 exposure-fence shape, inverted-for-arithmetic).

## 10. Relationship to `dataset_snapshot`
- **Readiness only — no `snapshot`-package change.** At HEAD, `dataset_snapshot_component.component_kind` is an un-CHECKed `String(50)` whose app-side allow-list `SNAPSHOT_COMPONENT_KINDS = (PORTFOLIO, POSITION, VALUATION)` **forward-notes** `FX`/`PRICE`/`CURVE`/`REFERENCE` only in a source comment (per OD-P2-B) — **there is no `COMPONENT_KIND_FX` constant yet**. When **P2-3** builds a snapshot for a multi-currency exposure run, **its** binder extension mints `COMPONENT_KIND_FX` (adds it to the tuple), extends the binder walk + `_reresolve_content` dispatch, and pins the **FX head versions** by calling **`reconstruct_fx_rate_as_of`** (this slice's resolver) — capturing the same `(surrogate_id, valid_from, system_from)` + `captured_content` + `content_hash` the P2-1 binder uses for POSITION/VALUATION. **P2-2 binds no FX into any snapshot, adds no FX component/constant, and edits no `snapshot` code** — it only ships the resolver seam + the canonical-serialization-friendly row shape (the explicit `Numeric(28,12)` `rate` scale per §3/§5, ISO currency strings, `rate_date`) so the future P2-3 pin is byte-stable across the AD-011 SQLite/PG split (P2-3 adds a corresponding `_SCALE_FX_RATE` to `snapshot/serialize.py`).

## 11. Relationship to `calculation_run` — **NONE in P2-2**
P2-2 wires **no** `calculation_run`, adds **no** `environment_id`, produces **no** `exposure_aggregate`, and imports **no** `calc` symbol (AST/import fence). The FX rate is "bound to the consuming run" (QS-09) **only at P2-3**, when the exposure run pins the FX components and converts marks. P2-2's `convert` is a **pure read-time helper** usable directly (e.g. a `GET /fx/convert` read) and **by** the future P2-3 run — but P2-2 itself creates no run and no derived number.

## 12. APIs (thin layer over `irp_shared.marketdata`)
All under `api/marketdata.py`; tenant-scoped via `get_tenant_session`; deny-by-default guards; `uuid.UUID`/ISO-validated path/query params (422 + indistinguishable 404); single end-of-request commit; whole-unit rollback on a governed-write error.
- **`POST /fx`** — capture a rate (gated **`marketdata.ingest`**; 201). Body: `base_currency, quote_currency, rate_date, rate, rate_type=MID, valid_from?, rate_source?`. `tenant_id` server-stamped; currencies resolved via the new hybrid-aware `resolve_currency` (own-OR-SYSTEM; foreign-tenant/unknown → 404); positive-rate + required-field DQ gate (fail-closed → 409/422); VENDOR-source ORIGIN lineage; `MARKET.FX_CREATE` audit.
- **`POST /fx/{id}/supersede`** — effective-dated re-quote for the SAME `(pair, rate_date, rate_type)` (gated `marketdata.ingest`): close prior `valid_to` (`MARKET.FX_UPDATE`) → new open version (`MARKET.FX_CREATE`).
- **`POST /fx/{id}/correct`** — as-known restatement (gated `marketdata.ingest`): close prior `system_to` (`MARKET.FX_UPDATE`) → corrected version, SAME valid period + `rate_date`, with `restatement_reason` + `supersedes_id` (`MARKET.FX_CORRECTION`).
- **`GET /fx/as-of`** — single head rate for `(base, quote, rate_date, rate_type, valid_at, known_at?)` (gated **`marketdata.view`**; 404 if none).
- **`GET /fx`** — list/range over a `(pair, rate_type)` for a `rate_date` range, as-of `(valid_at, known_at?)`; `limit`/`offset`.
- **`GET /fx/convert`** — `amount, from, to, valid_at, known_at?, base?` → `{converted_amount, rate_path: [...], rate_type}` (gated `marketdata.view`). **Read-only — no audit, no lineage, no DQ write** (the holdings/verify read precedent); `FxRateNotFound` → 404/409.
- **No PUT/PATCH/DELETE** (FR is supersede/correct, never in-place edit/delete).

## 15. RLS behavior
- **Symmetric tenant-scoped** (`USING == WITH CHECK == current_setting('app.current_tenant', true)`, ENABLE+FORCE; migration `0017`, mirrors `0016`). **NEVER hybrid** — `fx_rate` is **per-tenant vendor-licensed** market data (MNPI-adjacent, AD-008); a shared-global FX set would EXPAND the closed hybrid set → an **AD-013-R2** governance event, out of scope (OD-P2-G). The migration carries the **"closed 5-table hybrid set asserted unchanged"** fence (the proprietary-RLS precedent).
- Cross-tenant `base_currency`/`quote_currency` resolution fails closed at the **service layer** via the **new hybrid-aware `resolve_currency`** (§3): it admits **(own-tenant OR `SYSTEM_TENANT`)** codes (so the SYSTEM `USD`/`EUR` base resolves) and rejects a **foreign tenant's** currency → `CurrencyNotVisible` (404). This is deliberately the **hybrid USING-arm predicate** (the `dedupe_tenant_wins` shape), **not** the symmetric `tenant_id == acting_tenant` predicate (which would wrongly reject SYSTEM currencies). No BYPASSRLS app path. PG-proven under FORCE-RLS as the constrained `irp_app` role (SYSTEM currency visible, foreign-tenant currency not).

## 16. Lineage behavior
- One **VENDOR `data_source` ORIGIN edge per NEW physical version** (capture / new-open-row-of-supersede / correction); the prior-head close-out roots none. **Reuses `record_lineage` UNCHANGED — no new lineage code, no `lineage_edge` migration.** `ensure_vendor_source` resolve-or-register (the `ensure_manual_source` precedent) → `record_lineage(source, target_entity_type='fx_rate', target_entity_id=row.id, edge_kind=ORIGIN)`; fail-closed; co-transactional (CTRL-032 — if `record_event`/`record_lineage` raises, the whole unit rolls back; negative test). ## 17. Data-quality behavior

## 1. Requirements included

P1A-4 implements **exactly one** Step-2 backbone requirement and is the **FINAL, first-COMPOSING** P1A slice — it builds **no** forward dependency of its own; it is a pure **consumer/composer** of the shipped P1A-0/1/3 rails:

- **REQ-INT-001 — File upload (CSV/Excel)** (backbone §7 CAP-18.1 "File upload (CSV/Excel) — Validated, sandboxed upload via anti-corruption layer", L271; RTM row 91; Phase **P1**; personas **P-DS** Data Steward (PERSONA-05/H-04, Platform LoD; owns CAP-18; drives UJ-4 step 1 "Ingest data via upload/adapter (REQ-INT-001/002) → DQ rules run (REQ-DQR-001)") + **P-ADM** Platform Administrator (PERSONA-08, CAP-17/18); control **CTRL-027** (DQ rules run on ingest); audit **BX-AUD**; entitlement **BX-ENT**; lineage **BX-LIN**; ModelGov `—`; dependencies "anti-corruption, FW-AUD, DEP-DQF"). Backbone acceptance (L271): *"Bad/malicious files rejected; canonical mapping + lineage recorded."*

P1A-4 is the **FIRST real governed-write endpoint** and the **FIRST on-ingest DQ run** (RTM row 76 already flagged "first real on-ingest run at P1A-4"). It is the first slice to actually invoke the P1A-1 lineage and P1A-3 DQ contracts on a real governed-write path.

**Precise in-scope statement.** Establish a generic file-staging data model (`ingestion_batch` head + immutable `ingestion_staged_record`), a synchronous upload endpoint with an anti-corruption validation layer, and an ingest orchestration utility that **composes** the shipped rails. Concretely the slice ships:

1. `ingestion_batch` (BC-16 Integration, NEW ENT id e.g. **ENT-047**, **IA-classed but status-mutable** — the CalculationRun precedent, tenant-scoped) with `data_source_id` FK, `filename`, `content_type`, `byte_size`, `status`, row/staged/failed counts, started/completed timestamps, and a `scan_status` AV placeholder (§3).
2. `ingestion_staged_record` (NEW ENT id e.g. **ENT-048**, **IA truly immutable**, tenant-scoped) — append-only `(batch_id` FK, `row_number`, `payload` JSON of raw parsed rows, **NO domain target)** (§3).
3. A synchronous `POST /ingest/upload` (multipart: file + `data_source_id`) gated `data.upload` under `get_tenant_session`, plus `GET /ingest/batches`(/`{id}`) reads (§4).
4. An ingest orchestration utility in a NEW **`irp_shared.ingestion`** package that runs validate → create batch → stage rows → DQ → lineage → audit in one tenant-scoped transaction (§5).
5. The anti-corruption layer (size limit / content-type+extension allowlist / filename-sanitize+path-traversal-reject / encoding / CSV formula-neutralize / malformed-reject) + the `scan_status` AV placeholder hook (§14).
6. **No new permission** — reuse the existing `data.upload` (bootstrap.py:23, granted `data_steward` + `platform_admin`) with **zero** role-template changes (§7).
7. **No new audit code** — reuse the existing `DATA.INGEST` (batch lifecycle) + `DATA.VALIDATE` (DQ runs); P1A-4 mints **zero** taxonomy codes — the cleanest audit posture of the four rails (§6).
8. One Alembic migration (`0007_generic_ingestion_staging`, revises `0006_data_quality_skeleton`) creating both tables with FORCE RLS + tenant-isolation policy (USING + WITH CHECK) and append-only enforcement on the IA staged-record table; plus the `python-multipart` backend dependency.

This realizes the end-to-end governed upload path **upload → anti-corruption validate → generic stage (raw JSON) → DQ (P1A-3) → lineage origin (P1A-1) → audit**, all tenant-scoped — the first executable on-ingest DQ run and the first real exercise of `assert_passed_quality_checks` / `assert_has_lineage` on a non-synthetic target. REQ-AUD-001 is **satisfied cross-cutting** (the upload/batch path emits taxonomy events) and is **not** scoped as a P1A-4 requirement of its own (verified by the shared CTRL-012 audit-coverage test across P1A-1…4).

**Backbone-shorthand reconciliation (load-bearing — mirror the sibling reconciliations).** Backbone §7 L271 REQ-INT-001 acceptance reads *"Bad/malicious files rejected; **canonical mapping** + lineage recorded"* and the functional column implies an anti-corruption layer that maps to canonical entities. P1A-4 **intentionally refines/defers the "canonical mapping" clause to P1B/P1C** — staged data is raw/parsed JSON rows pending future canonical mapping, **NOT** canonical domain data. The skeleton "done" is upload → validate → stage → DQ → lineage origin → audit, with **NO** mapping into instrument/portfolio/position/valuation tables. This is a legitimate phase-level decomposition (the backbone is feature/epic-level per §1) and mirrors P1A-1 (`data_source=EV` vs backbone shorthand), P1A-3 (`data_quality_rule=EV` + CAP-13 `(IA)` shorthand correction). The plan §16 acceptance satisfies ONLY the "lineage recorded" half; "canonical mapping" is explicitly deferred. ## 3. Proposed database entities

Two new tables, one migration (`0007_generic_ingestion_staging`, revises `0006_data_quality_skeleton`). Both are tenant-scoped and carry FORCE RLS + a `tenant_isolation_<table>` policy reusing the migration `0004`/`0005`/`0006` `TENANT_SCOPED_TABLES` loop verbatim (USING **and** explicit WITH CHECK). ORM models follow the established `calc/models.py` / `dq/models.py` pattern: `class X(PrimaryKeyMixin, TenantMixin, <TemporalMixin>, Base)` with an explicit `__temporal_class__`. Named `pk_`/`ix_`/`uq_`/`fk_` constraints satisfy the `alembic check` drift gate + NAMING_CONVENTION. Grep confirms the slice is **fully additive** (zero existing `ingestion_batch`/`ingestion_staged_record`/`scan_status`/`IngestionBatch` artifacts — only the placeholder references in `dq/models.py:93` and `bootstrap.py:23`).

**Genericity contract (load-bearing).** `ingestion_staged_record.payload` is a single generic JSON column of raw/parsed rows with **NO domain FK and NO domain columns** — it must NOT carry `instrument_id`/`portfolio_id`/`position_id`/`valuation_id` or any canonical reference. The staging table is a domain-agnostic envelope (`batch_id` FK, `row_number`, `payload` JSON), exactly like the `lineage_edge.source_type` / `dq_result.target_entity_type` polymorphism. `status` and `scan_status` are **controlled-vocabulary `String` columns — NOT Python `Enum`, NOT a DB `CHECK`/enum type** (mirroring `calc` `RunStatus` and `dq` `rule_type`). **The only FKs are the two intra-context FKs** (`ingestion_batch.data_source_id → data_source.id`, `ingestion_staged_record.batch_id → ingestion_batch.id`); no FK to any domain/analytical table, to the model registry, or to `lineage_edge`.

### 3.3 IA enforcement contrast (the load-bearing temporal split)

`ingestion_staged_record` gets **both** layers (P0001 trigger + ORM guard). `ingestion_batch` gets **NEITHER** — it is IA-declared but status-mutable. A negative-control test (§15) proves a raw `UPDATE` on `ingestion_batch.status` **succeeds** under context (no trigger) while a raw `UPDATE`/`DELETE` on `ingestion_staged_record` is **rejected by P0001** (with `irp_app` granted UPDATE/DELETE so it is the trigger, not a 42501 privilege denial). **CRITICAL: do NOT add `ingestion_batch` to `APPEND_ONLY_TABLES`** — the trigger would block every status transition on the first `UPDATE`.

## 4. Proposed API surfaces

Exactly three surfaces — one governed write, two reads. `POST /ingest/upload` is the ONLY write endpoint.

| Surface | Method | Auth | Behavior |
|---|---|---|---|
| `POST /ingest/upload` | write (multipart: `file` + `data_source_id`) | `require_permission('data.upload')` under `get_tenant_session` | Synchronous: anti-corruption validate (BEFORE any DB write) → create `ingestion_batch` → stage rows → DQ → lineage origin → finalize + audit, in one tenant-scoped transaction. Server-stamps `tenant_id`; never reads it from the body. |
| `GET /ingest/batches` | read | `require_permission('data.upload')` under `get_tenant_session` | RLS-scoped batch list (caller's tenant only), deterministically ordered. |
| `GET /ingest/batches/{id}` | read | `require_permission('data.upload')` under `get_tenant_session` | Batch detail; `uuid.UUID` path param; indistinguishable 404. |

**Read-gate resolution (OQ-P1A-4-READGATE, §18): gate the batch reads on the EXISTING `data.upload`, NOT `dq.result.view`.** Rationale (Product + Security + Data-Architecture + QA concurrence): the batch lifecycle/status/counts are an ingestion-domain read owned by the same P-DS/P-ADM uploader persona; reusing `data.upload` keeps the read symmetric with the write (the `dq.py` precedent gates `GET /dq/rules` on `dq.rule.manage`, not a new code) and — critically — `ingestion_staged_record.payload` echoes raw client data (DC-2-flaggable, possibly DC-3/DC-4), so it must NOT be exposed to the broader `dq.result.view` audience (1L/2L/3L). The DQ **results** produced on staged data remain readable via the existing P1A-3 `GET /dq/results` under `dq.result.view`, unchanged. No new permission, no role change.

**Hard guardrails (Scope/Security lens):** REJECT any `/ingest/map`, `/ingest/promote`, `/ingest/canonical` (canonical mapping, P1B/P1C); `/ingest/reconcile`, `/ingest/override` (P7); any adapter/feed/SFTP/poller endpoint (P9); any async/job-submit endpoint (sync skeleton); any public standalone `POST` to run DQ (DQ runs inline within upload). `GET /ingest/batches/{id}` must NOT expand into a staged-record browsing/aggregation/export surface, and do NOT add a `/ingest/batches/{id}/lineage` convenience route (that single-hop traversal drifts toward REQ-LIN-002/P7).

**Endpoint requirements (Security lens):**
- MUST depend on `get_tenant_session` (sets `app.current_tenant`), **not** `get_db`, so RLS scopes/stamps every read/write to the caller's tenant. Identical dependency wiring to `apps/backend/api/dq.py:create_rule`.
- The anti-corruption layer MUST validate the file **before** any DB write — validation order: size → extension/content-type allowlist → filename sanitize/path-traversal reject → decode/encoding → parse → per-cell formula-neutralize → stage. A rejected file leaves ZERO orphan DB rows except the auditable `RECEIVED`→`REJECTED` batch transition (§6).
- `POST /ingest/upload` **never** reads `tenant_id` from the body (server-stamps from principal/context); a forged `tenant_id` is ignored and backstopped by `WITH CHECK` → SQLSTATE 42501. `data_source_id` from the body MUST be resolved through the RLS-scoped session (cross-tenant/unknown → `DataSourceNotVisible`/indistinguishable 4xx, never a dangling cross-tenant lineage edge).
- The single-transaction invariant holds (no mid-request commit; commit once at end, else the next autobegun txn runs with no tenant context and RLS fails closed).
- **Size limit (OQ-P1A-4-3):** enforce a server-side hard byte cap (recommend a config-driven **10 MiB** for the CSV skeleton) by counting bytes **while reading/spooling** (do NOT trust `Content-Length`, which can be absent or lied); abort with **413** the moment the cap is exceeded, writing zero staged rows. Also early-reject a `Content-Length` that already exceeds the cap.
- For reads, entitlement (403) is checked **first**, then the RLS-scoped lookup yields **404** for both "not found" and "exists in another tenant" — indistinguishable (fixed detail body, no oracle leak). Malformed `uuid` path param → uniform **422** before any DB hit; missing principal headers → **401**.
- Needs the **`python-multipart`** dependency (FastAPI `UploadFile`/`Form`) — currently **absent** from `apps/backend/pyproject.toml` (confirmed); a hard prerequisite (§17).

## 5. Worker / CLI impact

- **Synchronous for the skeleton (OQ-P1A-4-2).** The entire upload → validate → stage → DQ → lineage → audit flow runs in-request in **one tenant-scoped transaction**. NO Celery/worker, NO background batch processor, NO ingestion CLI, NO scheduled poller, NO SFTP/feed-puller (that is REQ-INT-002 adapter territory, P9). REQ-INT-001 acceptance is satisfied end-to-end synchronously; async batch processing is a later option (no requirement mandates it now) that would add worker surface outside the slice. Keeping it synchronous also avoids a second trust boundary (a queue/worker re-reading attacker-controlled staged bytes outside the request's tenant context).
- **The ingest orchestration utility MUST live in a NEW `irp_shared.ingestion` package** (shared-python), a **sibling to `irp_shared.lineage`/`irp_shared.dq`/`irp_shared.model`/`irp_shared.calc`** — **NOT** in `apps/backend`. The FastAPI route is a thin wrapper; the orchestration (validate → create batch → stage → DQ → lineage → audit) lives in the shared package so a future async worker/CLI path gets identical behavior and audit. This is the P1A-1 R3 / P1A-3 R7 service-boundary trap applied to ingestion.
- **Dependency direction is strictly one-way: `irp_shared.ingestion → {irp_shared.lineage, irp_shared.dq, irp_shared.audit}` (+ db/temporal).** `irp_shared.ingestion` MUST NOT import any backend module, and `irp_shared.lineage`/`irp_shared.dq`/`irp_shared.model` MUST NOT import `irp_shared.ingestion`. It MUST NOT import `irp_shared.model` at all (no model coupling, §11). An import-direction test guards the acyclic DAG (mirrors P1A-3 AC-9).
- **Sandboxed parsing (THR-05).** The file MUST be parsed in-process with the standard-library `csv` module on the decoded text — no shell-out, no spreadsheet macro engine, no formula evaluation. Forward security requirement: if async lands later, the worker MUST set tenant context per job (`run_in_tenant`) before touching staged rows.

## 6. Audit events — resolved

**Decision (resolves OQ-P1A-4-AUDIT-SHAPE): reuse the existing `DATA.INGEST` and `DATA.VALIDATE` codes only — mint ZERO new audit codes.** Emit one `DATA.INGEST` at batch **create** plus a `DATA.INGEST` **status-change** event on **every** transition (the `CALC.RUN_CREATE` + `CALC.RUN_STATUS_CHANGE` precedent); `DATA.VALIDATE` (P1A-3, unchanged) carries the DQ runs on the staged data. This is the cleanest audit posture of the four rails (P1A-3 had to mint two codes; P1A-4 mints none).

The DATA category already exists (audit_event_taxonomy.md, EVT-020 block): `DATA.INGEST`, `.VALIDATE`, `.CORRECTION`, `.RECONCILE`, `.PURGE`, etc. Both codes carry the entire ingest lifecycle:

| Governed write | Event code | Notes |
|---|---|---|
| `ingestion_batch` create | **EXISTING `DATA.INGEST`** | `record_event(session, event_type='DATA.INGEST', tenant_id=<ctx>, actor_type='user'\|'agent', actor_id=<principal>, source_module='ingestion', entity_type='ingestion_batch', entity_id=batch.id, action='create', outcome='success', after_value={status:'RECEIVED', filename:<sanitized>, content_type, byte_size, data_source_id, row_count:null}, correlation_id=<request corr id>, data_classification='DC-1')`. Metadata only — NEVER raw cell payload or full client paths. |
| `ingestion_batch` status-change (each transition) | **EXISTING `DATA.INGEST`** | `action='status_change'`, `before_value={status:<old>}`, `after_value={status:<new>, staged_count, failed_count, reason:<code on REJECTED/FAILED>}`, `outcome=('failure' if new in {FAILED,REJECTED} else 'success')`. `actor_type='system'` when engine-driven (mirror `update_run_status`). Set `completed_at` on `TERMINAL_STATUSES={STAGED,COMPLETED,COMPLETED_WITH_WARNINGS,FAILED,REJECTED}`. |
| DQ run on staged data | **EXISTING `DATA.VALIDATE`** | Emitted by `run_quality_check` (P1A-3, unchanged): `outcome='success'` on PASS, `outcome='failure'` on FAIL/ERROR. P1A-4 adds nothing except threading the ingest `correlation_id` so an auditor joins `DATA.INGEST → DATA.VALIDATE`, and setting `DataQualityResult.ingestion_batch_id = batch.id` (no schema change). |

**Status-transition shape rationale.** `ingestion_batch` is the same IA-classed/status-mutable/NOT-in-`APPEND_ONLY_TABLES` pattern as `CalculationRun` (calc/service.py `CALC.RUN_CREATE` + `CALC.RUN_STATUS_CHANGE`): the row carries only a current-status projection while the **authoritative, append-only history lives in the hash-chained audit log**. A single terminal event would lose the `RECEIVED→…→FAILED/REJECTED` path and its per-transition timestamps — exactly the THR-05/THR-06 forensics signal an auditor needs to reconstruct "when was the file received vs when did the DQ gate fail vs when was the malicious file rejected." Each transition is chained at its own system time (gap-detectable).

**Anti-corruption REJECT path (resolves the reject-audit edge case).** Create the `ingestion_batch` row FIRST (`status=RECEIVED`, emit `DATA.INGEST` create capturing sanitized `filename`/`size`/`content_type`), THEN run anti-corruption validation; a validation failure becomes a `DATA.INGEST` `status_change` to `REJECTED` with `after_value.reason=<code>` (`oversized`|`bad_content_type`|`path_traversal`|`formula_injection`|`malformed`|`bad_encoding`). This guarantees **every** upload attempt — including rejected malicious ones — leaves an auditable, tenant-scoped, hash-chained forensic trail and avoids an un-auditable early return. Only a `get_tenant_session`/`require_permission` 401/403 precedes the row (not an ingest event; `AUTH.DENIED` deferred per DR-P1A0-4).

**No per-staged-row event proliferation:** ONE `DATA.INGEST` per batch lifecycle, NOT one per staged row. The `ingestion_staged_record` rows are immutable children of an already-audited governed write (the `lineage_edge` precedent — metadata of an audited write, no own event); `staged_count`/`failed_count` in the status-change `after_value` give the auditor the counts. A negative scope-fence test asserts no per-staged-row event is emitted.

**Lineage origin is auditable without its own event:** the `data_source → ingestion_batch` ORIGIN edge is co-transactional metadata of the already-audited `DATA.INGEST` batch-create; `record_lineage` emits no event of its own (the lineage precedent). Do NOT mint `LINEAGE.RECORD` (reserved P7) for the ingestion edge.

**Fail-closed (AUD-04 / CTRL-005 / CTRL-012 / CTRL-032):** each `record_event` MUST occur in the **same tenant-scoped transaction** as the batch/staged-record/status write (add row → `flush()` → `record_event` → return, no mid-call commit). If the audit insert is rejected (RLS 42501 or hash-chain failure) the governed ingest write MUST roll back. Because each status-change `record_event` is co-transactional with that status mutation, the status update + its audit event are atomic.

**Reserved-but-unused:** do NOT emit `DATA.CORRECTION` (restatement/override, P7), `DATA.RECONCILE` (P7), or `DATA.PURGE` on any P1A-4 path — a negative scope-fence test asserts none fires.

**Correlation:** reuse the ingest `correlation_id` for the `DATA.INGEST` events and thread it into `run_quality_check` so `DATA.INGEST → DATA.VALIDATE` joins without extra events.

**AI-authorship (BR-16):** when an agent principal uploads, the `DATA.INGEST` events carry `actor_type='agent'` + `agent_model`/`agent_model_version` + `on_behalf_of`.

**Note:** `event_type` is a free-form string (no code-level allowlist in `audit/service.py record_event` — confirmed), so a typo passes silently. The cross-cutting CTRL-012 audit-coverage test MUST assert the **literal** `DATA.INGEST` code and `entity_type='ingestion_batch'`. Recommend a shared `INGEST_EVENT='DATA.INGEST'` constants module (as `calc`/`dq` already do).

**Ops-role isolation (AD-015):** the BYPASSRLS `irp_ops` role MUST have **no** grant on `ingestion_batch`/`ingestion_staged_record`; no P1A-4 code path connects via the ops `DATABASE_URL`; the app DB role stays NOSUPERUSER NOBYPASSRLS. ## 10. Data quality behavior

**Compose P1A-3 exactly, no parallel engine.** P1A-4 imports `run_quality_check` + `assert_passed_quality_checks` + `DataQualityError` + `QualityCheckFailedError` from `irp_shared.dq.service`; it registers **NO** new evaluator in `dq.rules.REGISTRY` (the two generic evaluators `not_null`/`allowed_values` stay), defines **NO** domain-specific rules, and builds **NO** parallel DQ engine. An import/registry-fence test asserts `REGISTRY` still contains exactly `{NOT_NULL, ALLOWED_VALUES}` after P1A-4 loads.

**The flow (preserve P1A-3 semantics EXACTLY).** After staging, for each applicable active rule call:

```
run_quality_check(session, rule=<rule>, dataset=<staged rows as Sequence[Mapping]>,
                  actor_id=<principal>, target_entity_type='ingestion_batch',
                  target_entity_id=batch.id, data_source_id=batch.data_source_id,
                  correlation_id=<ingest correlation_id>, ingestion_batch_id=batch.id)
```

then `assert_passed_quality_checks(session, 'ingestion_batch', batch.id)` as the on-ingest gate. The dataset is the same parsed/formula-neutralized JSON rows persisted to `ingestion_staged_record` so the check evaluates exactly what was staged. This is the **FIRST real exercise** of the gate (P1A-3 tested it only synthetically) — where **CTRL-027 "rules run on ingest" becomes Implemented** rather than Designed.

**Rule selection (resolve OQ-P1A-4-RULESELECT).** Select rules by querying `data_quality_rule WHERE is_active=true AND target_entity_type='staging.row'` under the RLS-scoped session (the canonical staging target string already cited in the P1A-3 plan §3.1). For each, call `run_quality_check` with the staged dataset and record the result against `target_entity_type='ingestion_batch'`/`target_entity_id=batch.id` so the `assert_passed_quality_checks` gate (keyed on `ingestion_batch:batch.id`) finds them. This honors the genericity contract (controlled-vocab string, no FK, extend-by-row) and lets P1B/P1C add rules without code. Reject caller-passed rule-id lists (an uploader could skip inconvenient checks — a control bypass) and hardcoded rule sets (breaks genericity). **Fail-closed edge case:** `assert_passed_quality_checks` raises `QualityCheckFailedError('no quality checks recorded')` when zero results exist for the target — accept this no-silent-pass behavior (an empty active-rule set → batch REJECTED); seed ≥1 generic rule in the test fixtures so the happy path yields a PASS.

**Severity semantics (preserve P1A-3 exactly; CTRL-029).** `severity=ERROR` failure → `run_quality_check` persists a flagged `DataQualityResult` (`passed=false`/`outcome=FAIL`) + emits `DATA.VALIDATE` `outcome='failure'`, then RAISES `DataQualityError` → batch transitions to `FAILED`/`REJECTED`. `severity=WARNING` failure → `outcome=WARN`, no raise → batch proceeds to `COMPLETED_WITH_WARNINGS` (resolve OQ-P1A-4-WARNSTATUS: a distinct controlled-vocab status, with `failed`/`staged` counts populated, so the no-silent-fill posture is visible at the batch surface). An evaluator error always propagates as `DataQualityError` (audited `outcome='failure'`) and is treated as a REJECTED batch — never swallowed into a PASS (a bare `try/except` that drops it is a CTRL-029 violation that must fail review). P1A-4 reacts ONLY to `DataQualityError` (reject) vs clean return (complete) and reads `outcome` for WARN-flagging — it does NOT re-derive the raise-vs-flag policy.

**Populate the placeholder (resolve OQ-P1A-4-DQ-BATCHID).** `DataQualityResult.ingestion_batch_id` already exists as a **nullable no-FK GUID placeholder** (`dq/models.py:93`) reserved for exactly this. Populate it with **NO schema change** by adding an optional `ingestion_batch_id: str | None = None` kwarg to `run_quality_check` (a backward-compatible additive change to the P1A-3 signature) that sets `DataQualityResult.ingestion_batch_id` **before flush** and echoes it into the `DATA.VALIDATE` `after_value` (P1A-3 already reserved `ingestion_batch_id:null` there). Set-before-flush populates the placeholder on BOTH the PASS and the ERROR/raise paths (the result is built before the raise) and respects IA immutability (never mutate-after-flush). Keep the column **NO-FK** so `irp_shared.dq` does not forward-couple to `ingestion` (preserves the acyclic boundary).

**Durable-record-on-reject (resolve OQ-P1A-4-DURABLE-REJECT — the load-bearing DQ concern).** `run_quality_check` **flushes** the `DataQualityResult` AND the `DATA.VALIDATE` audit BEFORE it raises `DataQualityError` (proven by `test_error_severity_failure_persists_and_raises`). So on `DataQualityError`, P1A-4 MUST **NOT roll back** the outer transaction and MUST NOT wrap the gate in a `begin_nested` SAVEPOINT that rolls back (a savepoint rollback would DISCARD the very flushed result + audit we must keep — the exact P1A-3 caveat). Instead: catch the error, leave the flushed result/audit in the live transaction, set `ingestion_batch.status=REJECTED`/`FAILED`, emit the terminal `DATA.INGEST` status event, and **COMMIT** — so the REJECTED batch, its staged rows, the FAIL `DataQualityResult`, and both audit events are durably persisted together in one gapless audited transaction. The HTTP response is a 4xx/422 surfacing the failure — never a 200 with silently-dropped rows. (This is distinct from the CTRL-032 fail-CLOSED rule, which applies to an **audit-insert** failure: that rolls back the governed write.)

## 14. File/security controls (THR-05/THR-06)

The decisive net-new lens over the siblings: the upload anti-corruption layer, **ALL enforced at skeleton level BEFORE any DB write** (realizing CTRL-027 anti-corruption-on-ingest + the THR-05/THR-06 controls). Validation order: size → extension/content-type allowlist → path-traversal filename sanitize → encoding decode → CSV parse → per-cell formula-neutralize → stage.

1. **SIZE LIMIT (THR-05):** hard server-side byte cap (config-driven, recommend 10 MiB) enforced while reading (do NOT trust `Content-Length`); oversized → 413, zero DB writes.
2. **CONTENT-TYPE + EXTENSION ALLOWLIST (THR-05):** CSV only for the skeleton (`text/csv` + `.csv`); reject everything else → 4xx. Validate the decoded content is parseable CSV, not just the declared type (a `.csv` with a disallowed type, or a disallowed extension with `text/csv`, is rejected — the mismatch case).
3. **PATH-TRAVERSAL FILENAME SANITIZATION (THR-06):** reject/strip filenames containing path separators, `..`, null bytes, absolute paths, or control chars; never use the client filename as a filesystem path. In-DB JSON means we never write it to disk (structurally defeats THR-06 traversal), but still sanitize before storing/auditing the name — store only the sanitized basename.
4. **ENCODING HANDLING (THR-05/06):** decode with an explicit codec (UTF-8, strict or documented errors policy); a file that fails to decode (invalid bytes, unexpected encoding, embedded null bytes) is rejected as malformed, not silently mojibake'd or partially staged.
5. **CSV FORMULA-INJECTION NEUTRALIZATION (THR-06) — resolve OQ-P1A-4-FORMULA: NEUTRALIZE, do not reject.** Every cell value whose first char is one of `= + - @ TAB(0x09) CR(0x0d)` is neutralized (prefix with a single quote / escape) BEFORE it lands in `ingestion_staged_record.payload`, so a downstream spreadsheet export cannot execute it, while the data is preserved for canonical mapping. Document that downstream export (P7/8) must ALSO escape on render — THR-06 is a two-ended control; neutralizing at rest makes the stored evidence safe regardless. (Neutralize, not whole-file-reject, avoids a usability regression on legitimate leading-symbol data.)
6. **MALFORMED-CONTENT REJECTION (THR-05):** ragged rows, unparseable CSV, unterminated quotes, zero-row/header-only files → rejected with a reason code, no partial staging.
7. **AV/MALWARE — PLACEHOLDER HOOK ONLY (OD-042) — resolve OQ-P1A-4-AVHOOK.** `scan_status` field (controlled-vocab, defaulting to a **NON-clean** `PENDING`/`SKIPPED` so production cannot silently ship without AV) + a no-op pluggable `scan(file_bytes) -> status` seam the endpoint invokes before staging. A later AV integration swaps the seam implementation and lets production gate on `CLEAN` with ZERO table/endpoint/signature changes — addable WITHOUT redesign. Do NOT integrate real AV now (THR-05 later-hardening).
8. **TENANT-DATA-LEAKAGE (THR-04):** the staged payload is RLS-tenant-scoped with server-stamped `tenant_id` + `WITH CHECK` — the cross-tenant payload-invisibility test (§15) is the headline RLS proof.

THR-07/THR-08 (poisoned/injection-via-mapped-fields, canonical mapping, reconciliation) are partially mitigated by the anti-corruption layer + DQ but fully land with canonical mapping in P1B/P1C — their "canonical mapping"/"reconciliation" controls are OUT for P1A-4; the slice's anti-corruption layer covers parse-safety/validation/formula-neutralization ONLY. Do not let THR-07/08 alignment language smuggle canonical mapping into scope.

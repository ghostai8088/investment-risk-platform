## Part 2 — Design decisions

### OD-API-1-A — The read doctrine (additive, reproducibility-safe)
Entity/time reads are ADDITIVE; the run-id reads and TR-09 reproducibility-by-pin are untouched. Every new read: joins `CalculationRun`, filters `tenant_id` + `status=='COMPLETED'`, applies the entity equality filter(s), applies `as_of` as `CalculationRun.system_from <= as_of` (`None` = now), and totally orders `system_from DESC, run_id DESC, <grain> ASC`. Unknown/foreign entity id ⇒ **silent-empty** for the list (the positions/valuations precedent); the `/latest` resolver ⇒ **404** when the entity has no COMPLETED run (the pacing precedent). NO new writes, NO migration.

### OD-API-1-B — Per-family reads, by class (RATIFIED: pure read-only; Class C → API-1b)
- **Class A (8 families):** the two-endpoint pacing shape, gated the family's existing `.view` (risk/perf/exposure). `list` = flat rows filtered by the native entity column(s); `latest` = the newest COMPLETED run *for the entity*. **The single-book families** (portfolio_return, benchmark_relative, desmoothed, var_backtest, es_backtest, proxy_weight) are verbatim-clean. **exposure_aggregate + factor_exposure_result are SUBTREE-SPANNING** (verifier F3): their reads ROW-FILTER to the queried portfolio's rows (never return sibling rows), and `/latest` resolves to the newest COMPLETED run *containing a row for the entity* — documented as "may be an ancestor's subtree run; the returned rows are the entity's own." NOT a whole-run return.
- **Class B (3 families):** covariance, scenario, **sensitivity** (the verifier-F2 correction — curve-node-keyed, no portfolio). A `GET /{family}/latest[?filter]` **latest-RUN** resolver (newest COMPLETED run + all its rows) — the matrix/vector/curve is the run, so "current" = the latest run, not a row-entity latest. Filter: covariance by nothing; scenario by `scenario_definition_id`; sensitivity by the curve selector.
- **Class C (var, active_risk) — DEFERRED to API-1b (OD-API-1-H).** The verifier (F1, HIGH) REFUTED the read-only premise: the provenance factor-exposure run is subtree-scoped and **no root-portfolio is recorded on `calculation_run` or `dataset_snapshot`** (confirmed: `DatasetSnapshot` has label/purpose/as_of/predicate/hash — no portfolio/scope column), so "latest VaR for portfolio P" cannot be resolved read-only without misattribution. Ratified (user, 2026-07-20): keep API-1 pure read-only; API-1b adds the one run-scope column + the flagship reads.

### OD-API-1-C — F2 governance reads
- **Validation findings/evidence on GET (OQ-API-1-6):** a per-validation detail read `GET /models/{model_id}/validations/{validation_id}` returning findings + evidence (the summary list stays lean; the detail carries the heavy sub-objects) — gated `model.inventory.view` (the existing validation-read gate; no new code).
- **Audit read:** a NEW read-only router `api/audit.py` — `GET /audit/events?entity_type&entity_id&event_type&since&until` (paginated), gated **`lineage.view`** (RATIFIED reuse — no new R-07 mint). **Rationale correction (verifier doctrine F4):** `lineage.view` is NOT 3L-only — it is held by `auditor_3l` AND the makers `risk_analyst_1l` + `data_steward` (+ `risk_manager_2l` + `platform_admin`); only `ops` lacks it. So this reuse grants **tenant-wide, metadata-only audit-read to makers too**, not just oversight tiers. This is the ACCEPTED consequence of "reuse, don't mint" (within-tenant, DC-2-metadata-only, RLS-bounded; the user ratified it knowing the holder set); a dedicated oversight-only `audit.view` is the recorded alternative if a future policy wants audit-read narrowed. **`audit/service.py` stays FROZEN** — API-1 adds a READ query (a new `audit/queries.py`, never the frozen service) + router. **Verifier CONFIRMED (OQ-API-1-7):** `audit_event` is under symmetric tenant FORCE RLS (test-proven under a non-superuser role, no cross-tenant leak), NOT in the closed hybrid set; payloads are DC-2 metadata-only (no bulk data / secrets); pagination bounds trail size.
- **Snapshot listing:** `GET /snapshots?purpose&as_of_valuation_date&portfolio_id?` gated the existing snapshot-read permission (the same gate as `GET /snapshots/{id}`); silent-empty.
- **`tier` on inventory:** add `tier` + `validation_status` to `ModelSummary` (the list DTO) — a pure additive-field change, gated `model.inventory.view`.

### OD-API-1-D — Parity by-id GETs
Add `GET /risk/scenario-results/{result_id}` and `GET /risk/proxy-weight-estimates/{result_id}` (the two families missing a by-result-id read), gated `risk.view` — closing the house-pattern asymmetry.

### OD-API-1-E — The demo-completeness rider (OQ-W8C-4, pays OQ-W7C-5)
A demo **stage 10** (filename **zero-padded** — `stage10` sorts before `stage2` lexically, the CC-2-recorded caveat) exercising the FIVE zero-run codes on the living tenant: `risk.sensitivity.analytic`, `risk.active_risk.parametric`, `risk.scenario.factor_shock`, `perf.benchmark_relative`, and a proxy-mode `risk.factor_exposure.proxy` run. NO new model codes (all five already registered) — the counts move **runs-only** (96 → 96+N COMPLETED; codes stay 20, records stay 35 unless a run legitimately needs an AWC — none expected, the DS-2 census-honesty pattern). This makes API-1's new reads render NON-EMPTY for those families.

### OD-API-1-F — Scope fence
API-1 is **backend read endpoints + read queries + the demo stage ONLY.** ZERO FE changes (FE-2 consumes these). NO migration. NO write/mutating endpoints. NO new permission (audit read reuses `lineage.view`; everything else reuses `model.inventory.view`/`risk.view`/`perf.view`/`exposure.view`/the snapshot-read gate). **`audit/service.py` FROZEN**; the closed 5-table hybrid set untouched; RLS/write posture unchanged. Demo stages 1–9 byte-untouched (stage 10 additive). The run-id reads stay.

### OD-API-1-G — Read-query home
Each family's list-by-entity + latest-resolver live in that family's own `service.py` (the pacing precedent — `list_pacing_projections`/`latest_pacing_projection`), reusing the shared `CalculationRun`-join + `as_of` + ordering pattern; a small shared helper may factor the join/ordering if ≥3 byte-identical copies appear (the clean-code bar). The audit read gets a NEW `audit/queries.py` (read-only; NEVER the frozen `audit/service.py`); the snapshot listing extends `snapshot/service.py`.

### OD-API-1-H — API-1b: the flagship VaR/active-risk entity reads (the named fast-follow)
The verifier F1 HIGH established that "latest VaR/active-risk for portfolio P" cannot be resolved read-only (subtree-scoped runs; no recorded root portfolio). API-1b (the immediate next slice) pays it correctly: **ONE additive column `calculation_run.scope_portfolio_id`** (nullable; the ROOT portfolio a run was scoped to) stamped by the exposure/factor-exposure/var/active-risk binders at run creation (a small binder touch; historical runs stay NULL and are honestly unresolvable — disclosed, not back-fabricated), then the Class-C `list`/`latest` entity reads resolve `var_result`/`active_risk_result` → their run's `scope_portfolio_id == P` (unambiguous for hierarchical books). `active_risk`'s native `benchmark_id` filter also lands here. Read + one migration + a bounded binder change; the run-id reads and TR-09 untouched. **This is the single most-wanted UI/agent read** ("portfolio P's tail risk"), deferred ONLY because it needs the scope column — not dropped.

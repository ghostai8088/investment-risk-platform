## Part 1 — Grounding (file:line; the FE-3b auth-flow census 2026-07-21)

**The token contract the flow must satisfy** (`config.py`): `oidc_tenant_claim="tenant_id"`, `oidc_subject_claim="sub"`, `oidc_audience` (= `irp-backend`), `oidc_issuer`, RS256; required claims `exp/iss/sub/tenant_id`; MFA off by default. The Keycloak `irp-frontend` client's access token already carries all of these for the two seeded users.

**The three concrete infra gaps FE-3b closes** (all realm/compose, NOT backend logic):
1. **`demo-auditor` is absent from the realm** (`grep demo-auditor irp-local-realm.json` = 0) — yet it is the ONE read-only principal the FE-3 walk needs (the registrar/validator are SoD-split for *write* duties and cannot read the whole walk; FE-3 added `demo-auditor` to the demo seed precisely for this). Add a `demo-auditor` realm user (username `demo-auditor` → `sub`=`external_subject`; `tenant_id` attr = `DEMO_TENANT_ID`).
2. **No password credentials** on any demo user (`credentials: []`) — a browser login needs them. Add local-dev password placeholders (BR-10: clearly-marked, env-overridable, never a real secret).
3. **The backend's OIDC env is not wired in the compose oidc profile — and a naive wiring 503s every login (verifier CLAIM 2).** The browser reaches Keycloak at `localhost:8080`, so the token `iss` = `http://localhost:8080/realms/irp-local`; the backend's `OIDC_ISSUER` must be that exact string for the `jwt.decode(issuer=…)` check. But the backend runs INSIDE the compose network, and with `OIDC_JWKS_URI` unset it discovers JWKS by `urlopen`ing `{issuer}/.well-known/...` = `localhost:8080` — which from the backend container is the container ITSELF, not Keycloak → fail-closed 503 on every request. **The fix (config already supports it, `config.py:36` + `auth.py:142` `jwks_uri or _discover(...)`): set `OIDC_JWKS_URI=http://keycloak:8080/realms/irp-local/protocol/openid-connect/certs` on the compose backend** — keys fetched via the compose service name, issuer STRING kept as `localhost:8080` to match the token (the certs endpoint embeds no issuer, so the cross-host fetch is sound). So the oidc-profile env is `AUTH_MODE=oidc`, `OIDC_ISSUER=http://localhost:8080/realms/irp-local`, `OIDC_AUDIENCE=irp-backend`, **and `OIDC_JWKS_URI`=the keycloak-service certs URL** (the piece a naive wiring omits).

**The swap footprint — RUNTIME-shallow but TYPECHECK-wide (verifier CLAIM 3 corrected).** Identity becomes wire bytes in EXACTLY ONE place: `client.ts:41-47`; the only other reads of `session.userId/tenantId` are `AppShell.tsx:33` (display) and `session.ts` (parse). So the **logic** change is ~6 files: **`session.ts`** (the union type + load/save), **`client.ts`** (branch: dev headers vs `Authorization: Bearer`, keeping `method:"GET"` hard-coded), **`App.tsx`** (the `/callback` route ABOVE the session gate + OIDC logout), **`AppShell.tsx:33`** (the identity chrome renders `userId @ tenantId` today — a Bearer session has no such fields; show a decoded-`sub` or a generic label), a **new login view**, and a **callback handler**. **BUT the discriminated union `{kind:'dev',userId,tenantId} | {kind:'oidc',accessToken,…}` is NOT assignable to the `session: DevSession` parameter annotated in 16 non-test files** (the oidc arm lacks `userId/tenantId`), so `make fe-check` typecheck fails at every pass-through site until the annotation is mechanically widened `DevSession → Session` across those 16 signatures. This is a **type-only, low-risk, mechanical widening** (the runtime pass-through is unchanged) — NOT ~10-files-untouched; the record states the true footprint so the plan budgets for it.

**No `VITE_` env convention exists** (`grep import.meta.env apps/frontend` = 0; no `apps/frontend/.env.example`). FE-3b establishes it (typed `ImportMetaEnv` in `vite-env.d.ts` + a FE `.env.example`).

**PKCE zero-dep feasibility** — `crypto.subtle` (SHA-256), `crypto.getRandomValues`, `fetch`, `URLSearchParams`, `TextEncoder` are all baseline (no polyfill); the hand-rolled surface is ~30–50 lines (verifier/challenge + base64url + the token exchange + the callback). `crypto.subtle` requires a secure context — `http://localhost` qualifies; HTTPS deploys qualify; a non-localhost HTTP deploy would not (a footnote, not a blocker).

## Part 2 — Design decisions

### OD-FE-3b-A — The dependency posture (the central fork → OQ-FE-3b-1)
OD-FE-1-F stands: "any further runtime dependency is a decision, not a drive-by" — one has ever been granted (`react-router-dom`, for routing). FE-3b's flow is small (Part 1) and the codebase doctrine is "plain fetch + hand-written." **Recommendation: hand-roll PKCE with Web Crypto (zero new runtime dep).** The alternative (a vetted `oidc-client-ts`) is battle-tested but the first-ever non-routing runtime dep — surfaced as OQ-FE-3b-1.

### OD-FE-3b-B — The discriminated-union session (swap only the injection)
`session.ts` gains a union `Session = DevSession | OidcSession` (`kind` discriminant). `client.ts` branches: `kind==='dev'` → the existing `X-User-Id`/`X-Tenant-Id`; `kind==='oidc'` → `Authorization: Bearer <accessToken>`. **`method:"GET"` stays hard-coded** — the read-only fence is about the METHOD, not the header; a Bearer GET is still a read. The prop keeps flowing as `session` through the existing chain, but its ANNOTATION widens `DevSession → Session` across the 16 pass-through signatures (a type-only mechanical change; verifier CLAIM 3) — the runtime pass-through is unchanged. **Rejected: a superset union exposing `userId`(=`sub`)/`tenantId` to stay literally assignable** — it would feed `AppShell` a raw `sub` and contradict "a Bearer session has no such fields"; the honest widening is cleaner.

### OD-FE-3b-C — The auth-code + PKCE flow (hand-rolled path)
`login()` → generate a `code_verifier` (`getRandomValues`→base64url) + `code_challenge` (`subtle.digest` SHA-256→base64url) + a random `state`; stash verifier+state in `sessionStorage`; redirect to `{issuer}/protocol/openid-connect/auth?...&code_challenge_method=S256`. A **`/callback` route declared ABOVE the session gate** in `App.tsx` (it lands pre-auth): validate `state` (CSRF) against the stored value BEFORE the exchange, **strip the `?code=…` query synchronously (`history.replaceState`) before any subresource loads** (referrer-leak hygiene; verifier CLAIM-4 nit), exchange the code at the token endpoint (`fetch` POST form-encoded, public client — no secret), **delete the single-use `code_verifier`+`state` from `sessionStorage` after the exchange** (verifier CLAIM-4 nit), store the access token + `exp`, and route into the app. Re-auth on expiry (OQ-FE-3b-2).

### OD-FE-3b-D — Token storage (fork → OQ-FE-3b-2)
**Recommendation: `sessionStorage` (matching the current `DevSession` idiom — "closing the tab drops the identity") + re-auth-on-expiry** (no refresh-token / silent-renewal machinery for a demo login). Disclosed caveat: an access token in `sessionStorage` is XSS-exposed (acceptable for this SPA class; noted, not hidden). Alternative: in-memory + silent renewal (more secure, lost on refresh, more code) — OQ-FE-3b-2.

### OD-FE-3b-E — The FE env convention (net-new)
Add a typed `ImportMetaEnv` (`vite-env.d.ts`) + `apps/frontend/.env.example` with `VITE_AUTH_MODE` (`dev_header`|`oidc`, mirroring the backend `auth_mode`), `VITE_OIDC_ISSUER`, `VITE_OIDC_CLIENT_ID` (=`irp-frontend`), `VITE_OIDC_REDIRECT_URI`. `VITE_` vars inline at BUILD time (fork → OQ-FE-3b-4: build-arg-inlined vs a runtime `config.json`).

### OD-FE-3b-F — dev_header coexistence + honesty
`VITE_AUTH_MODE==='dev_header'` → render `SessionForm` + the permanent `DevBanner` ("identity is unverified; not a security boundary") UNCHANGED (the demo still runs locally). `VITE_AUTH_MODE==='oidc'` → render a "Sign in" button that kicks the redirect, **HIDE `DevBanner`** (a verified Bearer session IS a security boundary — showing "unverified" over it would be a lie), and make "End session" perform an OIDC logout (Keycloak `end_session_endpoint`), not just a `sessionStorage` clear. The backend's fail-closed posture is untouched.

### OD-FE-3b-G — The realm + compose wiring (Part 1's three gaps)
Add the `demo-auditor` realm user (+ `tenant_id` attr = `DEMO_TENANT_ID`) and BR-10-marked local password credentials on the demo users; wire the compose `backend` service's oidc-profile env (`AUTH_MODE=oidc`, `OIDC_ISSUER`, `OIDC_AUDIENCE=irp-backend`) + the issuer-consistency runbook. **Demo-end-to-end scope = OQ-FE-3b-3.**

### OD-FE-3b-H — Scope fence
FE (the flow + the union session + the env convention + the login/callback views) + `infra/keycloak/` realm additions + `docker-compose.yml`/`.env.example` oidc-profile env + FE tests. **NO backend logic change** (the resource server is DONE — do not touch `deps.py`/`auth.py`/`config.py` behavior); NO migration; NO new governed number/code/permission/role/ENT; `audit/service.py` untouched; the GET-only fence + the decimal-strings contract stay byte-behavior-identical; the ~10 pass-through consumers untouched. **Out of scope (flag, related): the prod nginx image has no `/risk`→backend proxy** (a deployed FE would 404 on path-relative reads — predates FE-3b, affects the whole deployed FE, a future deploy slice). Counts UNCHANGED.

## Part 3 — Open decisions (OQ-FE-3b-1…4) — ALL RATIFIED 2026-07-21 (OQ-1 = A hand-roll; OQ-2/3/4 = as recommended)

- **OQ-FE-3b-1 — The dependency fork (the central Tier-3 decision).** **(A, recommended)** hand-roll PKCE with Web Crypto — zero new runtime dep, ~30–50 lines idiomatic to the "plain fetch + hand-written" doctrine, honours OD-FE-1-F. **(B)** add a vetted library (`oidc-client-ts`) — battle-tested security code, but the FIRST new runtime dep since `react-router-dom` and the first ever for a non-routing concern (requires the OD-FE-1-F exception). Recommendation **A**; the flow is small enough that owning it is lower total risk than a new supply-chain surface for a read-only viewer.
- **OQ-FE-3b-2 — Token storage + renewal.** **(A, recommended)** `sessionStorage` + re-auth-on-expiry (matches the `DevSession` idiom + demo simplicity; XSS caveat disclosed). **(B)** in-memory + silent renewal (more secure, lost on refresh, more code). Recommendation **A**.
- **OQ-FE-3b-3 — Demo end-to-end scope (verifier CLAIM-2-corrected).** **(A, recommended)** wire the FULL oidc-profile demo: the `demo-auditor` realm user + passwords + the compose backend oidc env **including `OIDC_JWKS_URI=http://keycloak:8080/realms/irp-local/protocol/openid-connect/certs`** (the piece a naive `ISSUER/AUDIENCE`-only wiring omits — without it every login 503s, Part 1 gap 3), so `docker compose --profile oidc up` yields a real login end-to-end (the POINT of FE-3b). **(B)** FE flow + realm additions only, deferring the compose backend wiring to a documented runbook. Recommendation **A** (bounded — env + a runbook, no new logic; the JWKS-URI split is the one non-obvious piece and the verifier pinned it).
- **OQ-FE-3b-4 — Build-time vs runtime FE config.** **(A, recommended)** build-arg-inlined `VITE_` config for now (local-dev values are FIXED by the realm — client_id/redirect/PKCE are pre-decided). **(B)** a runtime `config.json` fetched at boot (deploy-flexible). Recommendation **A**; note **B** as the deploy-time follow-up (pairs with the out-of-scope nginx-proxy gap).

## Part 5 — Verifier pass — RAN 2026-07-21 (adversarial; 3 HOLD, 2 COMPLICATED folded — NO redesign)

An adversarial verifier attacked the 5 load-bearing claims against the actual backend/realm/compose/FE code. Verdicts:

- **CLAIM 1 — the token resolves end-to-end: HOLDS** (given the plan seeds the realm user correctly). The three mappers are declared at the `irp-frontend` **client** level, so a NEW `demo-auditor` user inherits them (`sub`=username, `tenant_id` claim, `aud⊇irp-backend`). The demo `app_user` is seeded ACTIVE (`is_active` defaults True) with `external_subject="demo-auditor"`, `tenant_id=DEMO_TENANT_ID` (= the byte-exact realm attr `8c3193a6-…986a9`), so `_principal_from_token`'s `(tenant_id, external_subject, is_active)` lookup resolves. PyJWT `verify_aud` passes on membership, so Keycloak's extra `account` audience is harmless. **LOW (review note):** `directAccessGrantsEnabled:true` on the public client leaves ROPC/password-grant open — fine for a local demo, consider disabling.
- **CLAIM 2 — Docker issuer-consistency: COMPLICATED → folded (the make-or-break infra fix).** A `ISSUER/AUDIENCE`-only wiring 503s every login (in-compose JWKS discovery to `localhost:8080` hits the backend container, not Keycloak). Folded: OQ-FE-3b-3=A + Part 1 gap 3 now name **`OIDC_JWKS_URI=http://keycloak:8080/…/certs`** (config-supported via `auth.py:142` `jwks_uri or _discover`), keeping `OIDC_ISSUER=localhost:8080` to match the token. No code change — a plan/env correction.
- **CLAIM 3 — the swap footprint: COMPLICATED → folded.** Runtime centralization holds (`client.ts` is the only wire-bytes reader), but the union is not assignable to the `session: DevSession` annotation in **16** files → `make fe-check` typecheck fails until a mechanical `DevSession → Session` widening. Folded: OD-FE-3b-B + Part 1 restate the true footprint (~6 logic files + a type-only 16-signature widening); the superset-union alternative rejected.
- **CLAIM 4 — PKCE/state design: HOLDS.** `crypto.subtle` valid at `http://localhost`; `state` validated before the exchange; token never in the URL. 2 nits folded into OD-FE-3b-C (delete the single-use `code_verifier`+`state` after the exchange; strip the `code` from the URL synchronously before any subresource loads).
- **CLAIM 5 — DevBanner honesty + GET-only fence: HOLDS.** `DevBanner` renders unconditionally today (so the oidc-mode gating is a real, required honesty change); `client.ts` hard-codes `method:"GET"` and the `Authorization` header opens no non-GET path.

**Net: the token-contract crux HELD; the 2 COMPLICATED findings were an env correction (the JWKS-URI split) and an honest footprint restatement — no redesign of the flow, the union, or the realm.** The record is ready for the ratification gate.

## Part 0 — What LIM-1 is (and is NOT)

- **IS** the first governed **write-side workflow**: a `limit_definition` (a governed, versioned, 2L-maker-managed control config) + a deterministic breach EVALUATION that runs on the per-tenant operational tick + an IA `breach` record that is self-evidencing (echoes the threshold + observed value + comparison it was detected under).
- **IS a REALIZATION, not a mint.** ENT-031 `limit_definition`, ENT-033 `breach` (+ ENT-032 `limit_utilization`, ENT-034 `breach_action` as follow-ons) were reserved on paper since genesis; the `LIMIT.*`/`BREACH.*` audit codes sit in the genesis EVT-060/070 decades, RESERVED-only. LIM-1's activation MECHANIC is identical to SCH-1's SCHEDULE activation (caller-side constants → the FROZEN `record_event`) — though LIM-1's case is actually cleaner: SCHEDULE was *minted* fresh at EVT-260, whereas LIMIT/BREACH are *genesis-reserved* codes now first emitted.
- **IS NOT a governed number.** A breach evaluation is a pure predicate over an already-pinned governed result; it opens no `calculation_run`, pins no `dataset_snapshot`, binds no `model_version`. The `breach` row REFERENCES (FKs) the `calculation_run` it evaluated. Counts UNCHANGED 23/38/109; no new `run_type`; no methodology/model registration. It is the THIRD control-plane object after `schedule`/`scheduled_run`.
- **IS NOT the breach lifecycle.** v1 = DETECT + record + read. The ASSIGN → 1L_RESPONSE → 2L_REVIEW → ESCALATE → CLOSE workflow (ENT-034 `breach_action` + the `DEP-WFL` state-machine engine, which is greenfield) is **MG-2** (literally "carry a breach to term") — do NOT half-build it (OQ-4).
- **IS NOT a notification system.** No notification/alert/outbox/webhook/email substrate exists (greenfield). v1's only "notification" surfaces are the `BREACH.DETECT` audit event + the durable `breach` row + the Rule-7 read surface; external egress is a recorded v2 (Fable audit demand #3).

**The evaluation (a PHASE of the per-tenant operational tick — Fable demand #2):**

```
poll_tenant_breaches(session, now):                         # non-BYPASSRLS, run in run_in_tenant alongside poll_tenant_schedules
  for limit in active_limits(session):                      # tenant-scoped read of ACTIVE limits
      savepoint = session.begin_nested()                    # per-limit isolation (the SCH-1 per-schedule pattern)
      try:
          rows = latest_result_for(session, limit)          # calc/reads: latest COMPLETED result for (run_type, scope, metric)
          if not rows: continue                              # no fresh number yet — nothing to evaluate

          if _breaches(observed, limit.threshold_value, limit.comparison):
              _append_breach(session, limit, run_id, observed, now)     # BREACH.DETECT + the echo row
          savepoint.commit()
      except Exception: savepoint.rollback(); ...            # per-limit isolation, catch-all (the SCH-1 starvation fold)
```

- **Discovery is `calculation_run`-driven, NEVER `scheduled_run` (Fable demand #1).** `latest_result_for` delegates to the shipped resolvers: VaR → `latest_var_for_portfolio` (`var_service.py:1475-1524`, `run_type=VAR`, filters `scope_portfolio_id` + `metric_type`); active-risk → `latest_active_risk_for_portfolio` (`active_risk_service.py:727-770`) — both built on `calc/reads.py` `list_governed_results`/`latest_run_rows` (`calc/reads.py:35-89`; tenant + COMPLETED + run_type + scope filters, `system_from DESC` newest). **A MANUALLY-run VaR is limit-checked identically** (a limit regime that watched only scheduled runs would be a hole). Idempotency `(limit_id, calculation_run_id)` means each run is evaluated once per limit — a re-tick over the same latest run is a no-op (the SCH-1 `(schedule, tick)` dedup pattern).
- **The metric value per family (OD-C, the unit landmine):** VaR → `var_result.var_value` (`risk/models.py:258`, `PreciseDecimal(28,6)`, CURRENCY, positive=loss; for ES `metric_type` rows `var_value` holds the ES); active-risk → `active_risk_result.te_value` (`risk/models.py:342`, `PreciseDecimal(20,12)`, a daily-return FRACTION, NOT currency). The selector carries `threshold_unit` so a currency threshold is never compared against a fraction.

## Part 2 — Rule-6 external benchmark grounding (research RAN 2026-07-23)

1. **FSB — Principles for an Effective Risk Appetite Framework (Nov 2013).** Risk appetite CASCADES (board statement → aggregate limits → allocated desk/portfolio limits); limits must be MONITORED and breaches escalated. THE supervisory basis for LIM-1's existence — a platform with 20 governed numbers and no limit cannot answer "are we within appetite?"
2. **The three-lines-of-defense breach lifecycle (IIA; BCBS governance).** A breach is OWNED (1L), CHALLENGED (2L), OVERSEEN (3L), ESCALATED on severity/persistence. The platform's OWN reserved taxonomy encodes it exactly: `BREACH.DETECT → .ASSIGN → .1L_RESPONSE → .2L_REVIEW → .ESCALATE → .CLOSE`. v1 activates the DETECTION leg + a governed record; the rest is MG-2 (OQ-4) — do NOT half-build a workflow.
3. **Hard vs soft limits / utilization (Basel RAF practice).** A HARD limit is binding (a breach is an incident); a SOFT/advisory trigger is a warning. `REQ-LIM-001` (10.3) mandates "soft/hard limits." v1 carries a `limit_kind` (HARD/SOFT); a two-level warning+breach threshold (amber→red utilization — the BT-1 Basel traffic-light sibling) is the recorded v2 (OD-D).
4. **Determinism / point-in-time reproducibility (AD-014 / TR-09).** "Was limit L breached by run R" is a deterministic function of the PINNED result R and the limit-in-force. The breach record FKs the evaluated `calculation_run_id` AND echoes the threshold + observed value + comparison at detection — self-evidencing, reproducible without re-reading mutable state (OD-F; the `var_backtest_result` echo precedent).
5. **Limit governance = maker-checker, 2L (SR 15-18 / OCC Heightened Standards; the LIMIT.* codes; the personas).** A limit is a 2L RISK-MANAGER function — `personas_and_user_journeys.md:51`: limit change → maker P-RM (2L), checker P-CRO/2nd-2L; `BX-SOD` lists limits under maker-checker. So `limit.manage` → `risk_manager_2l` (NOT the 1L analyst — this DIVERGES from SCH-1's `schedule.manage`); breach SoD (`REQ-BRC-002`): 1L responds, 2L reviews, 1L cannot approve own closure (OD-J).
6. **Build-vs-buy (boring-base doctrine).** Vendor limit systems exist (Bloomberg PORT, Axioma, MSCI); the HOUSE shape is a governed `limit` table + a deterministic evaluation over pinned results + an IA breach record — auditable, reproducible, tenant-scoped, ZERO new infra, riding the SCH-1 per-tenant tick.

## Part 3 — Design decisions

### OD-LIM-1-A — Realize ENT-031 `limit_definition` (EV) + ENT-033 `breach` (IA); migration `0050`
Per Part 1. `limit_definition` = EV entity-versioned-in-place (the `Schedule`/`ScenarioDefinition` precedent); `breach` = IA append-only with the `(limit_definition_id, calculation_run_id)` unique idempotency key + the P0001 trigger. Both FORCE-RLS tenant-scoped (proprietary — symmetric RLS, NOT hybrid). ENT-032 `limit_utilization` + ENT-034 `breach_action` are the reserved follow-ons (utilization %, the breach-lifecycle actions), NOT realized in LIM-1.

### OD-LIM-1-B — NOT a governed number; counts UNCHANGED; NO new `run_type` (census §9, confirmed)
A `limit_definition` is a config object (binds nothing); a `breach` is a deterministic evaluation over an already-governed `calculation_run` (references it, binds no snapshot/model of its own). Neither is a governed derived number nor a captured input — both are control-plane objects (the `schedule`/`scheduled_run` classification). Counts stay 23/38/109. LIM-1 mints NO `run_type` and NO `model_version`; it activates `LIMIT.*`/`BREACH.*` audit codes + `limit.*`/`breach.*` permissions.

### OD-LIM-1-C — The metric-selector = a HARDCODED `(run_type, metric_type, benchmark_id?)` → column map, with a FAIL-CLOSED unit assert (fork OQ-3; folds verifier 1c/2/3c)
A limit targets a governed number via a **hardcoded** (NOT user-choosable) map — the ONLY defense that disarms the unit landmine:
- `VAR` / `{VAR_PARAMETRIC | _HISTORICAL | _TOTAL | _UNIFIED | ES_PARAMETRIC | ES_HISTORICAL}` → `var_result.var_value` (**CURRENCY**, positive=loss; for the `ES_*` metric_types `var_value` holds the ES — verifier-confirmed the ES rows are written with `run_type=VAR`, `var_service.py:1109`/`var_hs_service.py:452`).
- `ACTIVE_RISK` / `TRACKING_ERROR` → `active_risk_result.te_value` (**FRACTION**, a daily active-return vol, NOT currency).

**Three fail-closed guards (all MANDATORY acceptance criteria, not prose):** (1) the evaluator asserts `column_unit(run_type, metric_type) == limit.threshold_unit` and REFUSES a mismatch — a `CURRENCY` threshold can never be compared against a `~0.02` fraction (both `var_result` and `active_risk_result` carry a rival currency column — `sigma`, `portfolio_value` — so the right column is the map's choice, enforced); (2) the discovery read passes `metric_type=limit.metric_type` NON-None + asserts `rows[0].metric_type == limit.metric_type` (the resolver's `metric_type` filter is OPTIONAL/default-None — `var_service.py:1504-1524` — so passing it is load-bearing: without it `latest_run_rows` returns the grain-ASC-first row, i.e. the wrong metric_type); (3) **`ACTIVE_RISK` REQUIRES a `benchmark_id`** — `active_risk_result.benchmark_id` is NOT-NULL and a portfolio can compute TE vs MULTIPLE benchmarks, so the selector is `(run_type, metric_type, benchmark_id)` for benchmark-relative families and the resolver is called with `benchmark_id=limit.benchmark_id` (else successive ticks compare against DIFFERENT benchmarks' runs — the verifier's HIGH). There is NO separate `VAR_UNIFIED` run_type (the unified VaR is `metric_type=VAR_PARAMETRIC_UNIFIED` on `run_type=VAR`). Multi-row families (sensitivity/scenario/pacing) need a grain selector → recorded v2.

### OD-LIM-1-D — The breach predicate: `breach_direction` names the BREACH condition (anti-inversion) + `limit_kind` HARD/SOFT (fork OQ-3; folds the BLOCKING verifier 1a)
**The single most safety-critical predicate on the write-side control — pinned explicitly to kill the inversion trap.** The `limit_definition` carries `breach_direction` naming the BREACH condition DIRECTLY (not a compliant-relation the implementer must negate): `ABOVE` ⟹ breach ⟺ `observed > threshold` (a CEILING — VaR must not EXCEED X; the v1 default, since `var_value` positive=loss); `BELOW` ⟹ breach ⟺ `observed < threshold` (a FLOOR — e.g. a min information-ratio). **Boundary: STRICT inequality** — `observed == threshold` is COMPLIANT (exactly at the limit is not a breach). `_breaches(observed, threshold, direction)` is a two-line total function tested against this table from birth. (The rejected `LTE`/`GTE` "comparison" naming invited the exact backwards coding the verifier flagged — an inverted limit fires on every healthy book and stays silent on real breaches.) `limit_kind` (`HARD` binding → a breach is an incident; `SOFT` advisory → a recorded warning) satisfies `REQ-LIM-001` (10.3). A two-level warning+breach (amber→red) + two-sided directions + `limit_utilization` (ENT-032, value/limit %) are the recorded v2.

### OD-LIM-1-E — Scope = exact `scope_portfolio_id` match; subtree/firm rollup = v2 (fork OQ-2)
A limit scopes to a `portfolio_id` and binds by EXACT `calculation_run.scope_portfolio_id` match (the shipped `latest_var_for_portfolio` resolver, `var_service.py:1495`). There is NO `DESK`/`FIRM` node_type (only `PORTFOLIO/FUND/STRATEGY/ACCOUNT`, `portfolio/models.py:50`) and NO in-app subtree aggregation — a firm/desk-level limit can bind ONLY if a run was scoped to that firm/desk ROOT node (which SCH-1 can schedule). NULL `scope_portfolio_id` runs (pre-0046) are honestly unresolvable (skipped, not silently zero). Subtree/firm rollup is the recorded v2.

### OD-LIM-1-F — The breach row is SELF-DESCRIBING (echoes metric IDENTITY + comparison arithmetic); EV header + IA-echo, NOT FR-bitemporal (fork OQ-1, the crux; folds verifier 4)
A past breach must reproduce against the threshold-as-known-then (TR-09). Rather than an FR-bitemporal threshold on the header, the `breach` row echoes BOTH the metric **identity** (`target_run_type` + `metric_type` + `benchmark_id`) AND the comparison **arithmetic** (`observed_value` + `threshold_value` + `threshold_unit` + `breach_direction` + `limit_kind`) at detection time — so the breach is genuinely **self-describing from its OWN row** (the earlier draft echoed only the arithmetic, so the metric provenance was recoverable only via the header — the verifier's fold). The observed value's source is itself immutable (`var_result`/`active_risk_result` are IA append-only with `_block_mutation` + the P0001 trigger), so the echo is belt-and-suspenders on the value and load-bearing on the identity. **Threshold HISTORY ("every value this limit ever held, and when") is answered by the `LIMIT.CHANGE` audit trail** (before/after + `event_time` — the `SCHEDULE.UPDATE` precedent), NOT by the breach echo (which only captures the threshold at breach moments). The one thing FR-on-the-header would add that neither the echo nor the `LIMIT.CHANGE` trail gives is two-axis as-known CORRECTION of a mis-recorded threshold — genuinely low-value for a control-plane predicate that pins into no governed calc (unlike `ScenarioShock`, which is FR *because* it is snapshot-pinned into a governed number). This is the cheaper, self-describing path; FR-bitemporal is the alternative (OQ-1).

### OD-LIM-1-G — Breach evaluation is a PHASE of the ONE per-tenant operational tick (Fable demand #2)
A new `poll_tenant_breaches(session, now)` runs inside `run_in_tenant` (`db/tenant.py:137-144` — arbitrary `work` callable, non-BYPASSRLS, one terminal commit) as a SECOND phase alongside `poll_tenant_schedules`, each limit in its own SAVEPOINT (the SCH-1 per-schedule isolation + catch-all failure recording — verifier-confirmed the two phases compose cleanly: SAVEPOINT rollback preserves the enclosing transaction, so phase-1 rollbacks cannot poison phase-2, both landing under the one terminal commit). It is NOT a `Schedule` row (a breach eval binds no model, is not a `calculation_run`, and expanding `SCHEDULABLE_RUN_TYPES` would collide with the schedule header's NOT-NULL `model_version_id`). **Ordering INVARIANT (verifier fold):** schedules-phase runs BEFORE breaches-phase, because `dispatch_one` runs `run_var` inline (`scheduling/service.py:204`) so a just-scheduled VaR reaches a COMPLETED `calculation_run` visible to the breach read in the SAME tick/transaction — same-tick detection. This is a LATENCY property, not a correctness one (idempotency on `(limit_id, calculation_run_id)` means a reversed order just detects on the next tick — one tick later, never double/miss); recorded so a future refactor does not silently reorder and add a tick of breach latency. **Ratify the single-per-tenant-operational-tick rule** (schedules-phase + breaches-phase under one `run_scheduler_for_tenant`→`run_operational_tick_for_tenant` entry — a clean rename, verifier-confirmed only two in-file refs, the SCH-1 CLI/tests untouched) BEFORE a second CLI entrypoint is minted — infra wiring is still zero (the Fable-audit window).

### OD-LIM-1-H — Notifications: record-only v1 (Fable demand #3)
No notification/alert/outbox substrate exists. v1's breach "notification" is the `BREACH.DETECT` audit event + the durable `breach` row + the Rule-7 read surface. Any external egress (email/webhook) is greenfield infra — assembled per-tenant INSIDE the tick as IA attempt evidence when it lands (OQ-1=B forbids cross-tenant DB reads, not outbound egress), a recorded v2.

### OD-LIM-1-I — Identity-frozen invariant (Fable demand #4, carried from SCH-1)
A limit's IDENTITY fields — `target_run_type`, `metric_type`, `scope_portfolio_id`, `threshold_unit` — are FROZEN for its life; only `threshold_value`, `limit_kind`, `comparison`, `name`, `status` are editable in place (`_UPDATABLE`). A re-target (different metric/scope) is a NEW limit. This keeps a breach's echo meaningful against a stable limit identity and mirrors the SCH-1 "re-cadence = new schedule" standing invariant.

### OD-LIM-1-J — SoD: `limit.manage` is a 2L function (DIVERGES from SCH-1); R-07 mint + audit-code activation
`limit.manage` (define/edit) → `risk_manager_2l` + `platform_admin` — a 2L RISK-MANAGER function (personas + `BX-SOD`), NOT the 1L analyst who runs the numbers (author ≠ limit-setter, the VW-1 SOD-03 precedent; this is the notable divergence from `schedule.manage`'s 1L placement). `limit.view` + `breach.view` → broad incl. `auditor_3l` (governed control-plane objects are 3L-oversight scope, the `pacing.view` precedent). Added to `bootstrap.PERMISSIONS`+`ROLE_TEMPLATES`, seeded by migration `0002`. `LIMIT.DEFINE`/`.CHANGE` + `BREACH.DETECT` emitted caller-side to the FROZEN `record_event` from new `limit/events.py`/`breach/events.py` constants — an R-07 taxonomy ACTIVATION of the reserved EVT-060/070 rows (the SCH-1/SCHEDULE mechanic). `LIMIT.APPROVE` + the breach 1L/2L verbs stay RESERVED (OQ-4).

### OD-LIM-1-K — Hygiene: sweep the SCH-1 ENT-061/062 realization into the canonical data model
The census found `canonical_data_model_standard.md` still says "next free … ENT-061" though SCH-1 realized ENT-061 `schedule`/ENT-062 `scheduled_run`. LIM-1's `0050` doc pass sweeps that (adds the ENT-061/062 realized rows + marks ENT-031/033 realized-by-LIM-1) — a small doc-lag correction folded into this slice.

### OD-LIM-1-L — A stale / never-evaluated limit-health read (folds the verifier's silent-hole)
An ACTIVE limit whose scope+metric has NO matching COMPLETED run (or whose metric has gone cold — the latest run for that scope lacks the limit's `metric_type`) evaluates to nothing every tick: no breach, no row, no signal — indistinguishable FROM THE OUTSIDE from an in-appetite limit. On a platform whose premise is "can we answer 'are we within appetite?'", a limit that has NEVER been able to answer is a control gap. **Fix (no new mutable state):** a Rule-7 read-side "limit health" surface computes, per ACTIVE limit, its latest matching COMPLETED run + that run's age + whether it has ever been evaluated/breached — distinguishing IN-APPETITE (a recent matching run, no breach) from NEVER-EVALUABLE (no matching run) from STALE (a matching run older than expected). This is derived on demand from `calc/reads.py` (no per-tick write, no EV-header churn), and is the honest v1 answer to "why is this limit green?"

## Part 2 — Decision detail

### OD-P2-5-A — entity model → unified `curve` header + `curve_point` nodes
The canonical model lists **ENT-021 `yield_curve`** ("Curve nodes") and **ENT-023 `credit_spread`** ("Spread time-series") as separate lines, but a yield curve and a credit-spread curve are **structurally identical**: a curve identity (date/type/currency/source) + a set of `(tenor, value)` nodes. The platform's **genericity principle** (controlled-vocab type strings; new families extend by value, never a migration — `claude_operating_instructions.md`) says: **one unified `curve` header + `curve_point` nodes**, with `curve_type` discriminating `{TREASURY/GOVT/SWAP/OIS/CREDIT_SPREAD}`. This realizes **ENT-021** (`curve` = the curve identity; `curve_point` = the "Curve nodes") **and ENT-023** (`credit_spread` = `curve_type=CREDIT_SPREAD` over the SAME tables — no separate `credit_spread` table). The header+detail split mirrors `dataset_snapshot` (ENT-049 header) + `dataset_snapshot_component` (ENT-050 detail). Separate `yield_curve`/`spread_curve` tables would duplicate the identical schema + binder + RLS + audit per family — rejected. *(The governance amendment recording ENT-021/023 → `curve`/`curve_point` is Part 3; the exact ENT-id bookkeeping for `curve_point` is OQ-P2-5-A-id below.)*

### OD-P2-5-B — temporal class → header FR; points version-pinned immutable
A captured vendor curve is published **as-of a `curve_date`** and **re-versions as a whole** (a vendor restates the entire curve, not one node in isolation). So the cleanest, precedent-aligned model is: the **`curve` header is FR/bitemporal** (`FullReproducibleMixin`; capture → effective-dated supersede → as-known correction → `reconstruct_curve_as_of(valid_at, known_at)` on both axes; NOT in `APPEND_ONLY_TABLES`; content-immutability service-enforced), and **`curve_point` rows are IA append-only immutable children** pinned to one header physical version via `curve_id` (the ENT-050 `dataset_snapshot_component` pattern — `curve_point` is in `APPEND_ONLY_TABLES` with the `irp_prevent_mutation` P0001 trigger + ORM guard; a re-version creates a NEW header row + a fresh full point set; points are never updated/deleted). The **curve-as-a-whole is fully reproducible** — `reconstruct_curve_as_of` returns the header version true-at the as-of, and its `curve_id` reads the immutable pinned points. This keeps "which point values belong together as-of T" unambiguous (they share one header version).

**This is a NOVEL composite pattern (review fold — architect/product):** no single shipped table combines an FR bitemporal header with IA-append-only version-pinned children — it fuses the `fx_rate`/`price_point` FR capture protocol (header) with the `dataset_snapshot`→`dataset_snapshot_component` header-owns-immutable-children split (points). Because the header carries the bitemporality and the points are version-pinned, the **correction semantics are header-driven + explicit:** `correct_curve(prior_header, …)` closes the prior header's `system_to = now` (a `MARKET.CURVE_UPDATE`), then inserts a **corrected header version** over the **same valid period + same logical key** (`system_from = now`, `supersedes_id`/`restatement_reason` set) **AND a FRESH full `curve_point` set** pinned to the new header `curve_id` (a `MARKET.CURVE_CORRECTION` + a new VENDOR ORIGIN edge); the prior header version's content columns and its nodes are **NEVER mutated** (the nodes stay readable + immutable under the prior `curve_id`). A `supersede_curve` is the same shape on the `valid_to` axis. The header's mutable "content" is the metadata it carries beyond the key (e.g. `point_count`, and the OQ-P2-5-9 `interpolation_method` label if ratified); the *values* live in the version-pinned nodes — so a correction restating values is realized as a new header version + a new node set, never an in-place node edit. *(OQ-P2-5-1 records the heavier points-independently-FR alternative, where each node carries its own bitemporal axes + current-head uniqueness — rejected for v1 as the vendor re-versions atomically and it muddies the coherent-curve semantics.)*

### OD-P2-5-C — curve grain (header logical key, v1)
The v1 current-head partial-unique is **`(tenant_id, curve_type, currency_code, reference_key, curve_date, curve_source) WHERE valid_to IS NULL AND system_to IS NULL`** — exactly one open curve per that 6-tuple; the FR axes version the curve for the fixed key. `curve_source` is **in the key** (multi-vendor curves coexist — the `price_source` precedent). `reference_key` is **in the key** with a sentinel `"NONE"` for rate curves (OD-P2-5-K), so credit spreads (per-issuer/rating/sector) and rate curves share one uniform key. **All promoted key columns are DB-level `NOT NULL`** (the `price_point` lesson — a NULL key component cannot defeat the partial-unique on PG; the required-field DQ gate is the additional fail-closed check).

### OD-P2-5-D — curve_point grain
**`(curve_id, value_type, tenor_days)`** unique within a curve version — one open value per (curve version × value_type × normalized tenor). A single curve version may carry multiple `value_type`s (e.g. both `ZERO_RATE` and `DISCOUNT_FACTOR` at each tenor) and multiple tenors. Columns: `curve_id` (NOT-NULL FK → `curve.id`, the header physical version), `tenor_label` String, `tenor_days` Integer (the key; OD-P2-5-G), `value_type` String (OD-P2-5-H), `point_value` `Numeric(20,12)`, plus `tenant_id` (RLS — carried from the header). The `(curve_id, value_type, tenor_days)` uniqueness is a plain UNIQUE (points are immutable children, not a current-head partial-unique).

### OD-P2-5-E — `curve_date` convention → a separate immutable logical key
`curve_date` is a **separate immutable logical-key `Date`** — the business date the curve is FOR — carried forward verbatim by supersede/correct, **never mutated**, DISTINCT from the FR `valid_from` axis (the `valuation_date`/`rate_date`/`price_date` precedent). Many `curve_date`s per `(curve_type, currency, reference_key, source)` coexist as separate open heads.

### OD-P2-5-F — `curve_type` vocabulary (v1)
A controlled-vocab plain String (no enum/CHECK; app-side allow-list — the `price_type`/`rate_type` shape). v1: **TREASURY** (sovereign benchmark), **GOVT** (general government), **SWAP** (par/IRS), **OIS** (overnight-indexed) — the rate/yield families — and **CREDIT_SPREAD** (the spread family). An out-of-vocab `curve_type` → `CurveValueError` (422). `MUNI`/`BASIS`/`INFLATION` and `volatility_surface` are reserved.

### OD-P2-5-G — tenor convention → label + normalized days
Each node carries a canonical **`tenor_label`** String (`{N}{D|W|M|Y}`, e.g. `"1M"`, `"2Y"`, `"30Y"`) for human/audit readability **and** a normalized **`tenor_days` Integer** (the key component, strictly positive). The point key is on `tenor_days` (the normalized form) so `"12M"` and `"1Y"` cannot double-insert as distinct tenors. v1 validation (a DQ + a value check): `tenor_days > 0`; `tenor_label` matches the `{N}{D|W|M|Y}` pattern (an out-of-pattern label → `CurveValueError` 422). The label→days normalization is a **deterministic captured mapping** (D=1, W=7, M=30, Y=365 calendar-day convention, recorded), **not** an interpolation or day-count engine. **NO interpolation between tenors** (the curve stores exactly the vendor's nodes).

### OD-P2-5-H — value convention → canonical decimal; value_type; signed-per-type
**`point_value` is a canonical DECIMAL fraction** — never percent, never basis points. A 4.25% zero rate is stored `0.042500000000`; a 15bp credit spread is `0.001500000000`; a discount factor is `0.973…`. One canonical unit eliminates unit ambiguity (the platform never has to guess "is 4.25 a percent or a fraction?"). `value_type` is a controlled-vocab String describing what `point_value` means: **`ZERO_RATE`** (continuously/annually-compounded zero), **`PAR_RATE`** (par/swap rate), **`DISCOUNT_FACTOR`** (the discount factor at the tenor), **`SPREAD`** (a credit/Z-spread, decimal). Scale **`Numeric(20,12)`** — rate precision (12 dp; the `fx_rate` family's high-precision scale, NOT the money scale 6 — a zero rate / discount factor / spread needs sub-bp precision). **Sign is per `value_type`** (OD-P2-5-N): a `DISCOUNT_FACTOR` is strictly positive; `ZERO_RATE`/`PAR_RATE`/`SPREAD` may be **negative** (negative-rate regimes + tight/negative spreads are real) — so the value DQ gate is a **bounded sanity `RANGE`** keyed on `value_type`, not a blanket strictly-positive (a regression vs the simpler `price_point` gate, validated by the Data-Architecture/DQ lenses). **No interpolation, no bootstrapping, no discounting math** — `point_value` is captured verbatim. *(OQ-P2-5-2 scale; OQ-P2-5-4 the exact sanity bounds.)*

### OD-P2-5-I — currency → captured, no conversion
Each curve carries a **captured `currency_code`** (ISO alpha-3 String(3)), validated via the hybrid-aware `resolve_currency` (own OR SYSTEM — the `fx_rate`/`price_point` precedent), and **in the header key** (a SWAP curve in USD and one in EUR are two open heads). **NO FX/curve conversion in P2-5** — the curve is stored in its native currency; any cross-currency curve transform is a *later calculation*, never the capture path.

### OD-P2-5-J — source / provenance → label-in-key + VENDOR lineage; no feed pipeline
`curve_source` is a **controlled-vocab String label** (e.g. `"BLOOMBERG"`, `"REFINITIV"`, `"MARKIT"`) — a **key** component, so multi-vendor curves coexist. The **governed provenance** is a **VENDOR `data_source` ORIGIN lineage edge** per new physical curve version, via a new `ensure_vendor_source(VENDOR_CURVE)` (the `VENDOR_FX`/`VENDOR_PRICE` precedent; REUSES `record_lineage` UNCHANGED). The edge targets the **`curve` header** row (the points are children of that version, covered transitively). **NO market-data feed-ingestion pipeline / connector / scheduler** — captures arrive via the governed binder, one governed write at a time.

### OD-P2-5-K — credit-spread reference dimension → opaque `reference_key`, sentinel for rate curves
A credit-spread curve is meaningless without a reference dimension (it is a spread *for* an issuer / rating bucket / sector). v1 adds **`reference_key`**, a **NOT-NULL controlled String** in the header key: the **sentinel `"NONE"`** for rate curves (TREASURY/GOVT/SWAP/OIS — they have no reference entity), and an **opaque captured label** for `CREDIT_SPREAD` curves (e.g. `"ISSUER:<LEI>"`, `"RATING:BBB"`, `"SECTOR:FIN"`). v1 treats `reference_key` as an **opaque captured identifier** — the platform stores what identifies the spread curve but does **NOT** resolve it to an `issuer`/`rating_grade`/`legal_entity` FK (that formalization is a future additive slice; **OQ-P2-5-3**). This makes spread curves first-class without minting a new reference relationship, keeps the key uniform + NOT-NULL, and a scope-fence test asserts `reference_key` is a plain String (no FK, no resolver) in v1.

**A `curve_type` ↔ `reference_key` invariant is binder-enforced (review fold — data-arch #2):** the sentinel must not be a mere caller convention, or the 6-tuple key fragments (a TREASURY curve captured with `reference_key="RATING:BBB"` would split a rate curve's identity into two heads; a `CREDIT_SPREAD` with `"NONE"` would be a reference-less, meaningless spread). So `capture_curve`/`supersede_curve`/`correct_curve` enforce, **before any write** (a `CurveValueError` → 422, the same fail-closed path as the out-of-vocab `curve_type`/`value_type` checks; **NOT** via the FROZEN DQ Protocol): `curve_type ∈ {TREASURY,GOVT,SWAP,OIS}` ⇒ `reference_key == "NONE"`; `curve_type == CREDIT_SPREAD` ⇒ `reference_key != "NONE"` and non-empty. Negative tests cover both rejections. This turns the sentinel into a checked invariant and keeps the current-head key unambiguous.

### OD-P2-5-L — relationship to `dataset_snapshot` → readiness only
P2-5 makes `curve`/`curve_point` **available** to the reproducibility machinery but **pins nothing**: **NO `COMPONENT_KIND_CURVE` constant minted** (CURVE is noted-for-later only in the `SNAPSHOT_COMPONENT_KINDS` comment `# PRICE/CURVE/REFERENCE reserved later` — there is **no** named `COMPONENT_KIND_CURVE` constant today and P2-5 adds none; review fold — scope), **NO `build_snapshot`/`_reresolve_content` change, NO `curve_content` serializer.** This mirrors P2-2→P2-4 exactly: `COMPONENT_KIND_FX` was minted only at P2-3 (when exposure needed to pin FX); `price` was readiness-only at P2-4. A curve component is pinned **only** when a future calculation consumes a curve (a later derived-number slice). **NO `dataset_snapshot` schema redesign.** *(OQ-P2-5-7 confirms readiness-only vs pin-now.)*

### OD-P2-5-M — relationship to `calculation_run` → none
P2-5 wires **no** `calculation_run` and changes **no** `exposure_aggregate` / `dataset_snapshot` / `fx_rate` / `price_point` — captured curve data is an input, not a run. Binding a curve into a run (discounting, key-rate sensitivity, spread duration) is a **future calculation** (P3+), explicitly out of P2-5. The `calc`/`exposure`/`snapshot` packages are **untouched**.

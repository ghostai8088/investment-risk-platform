## Part 0 — What SCH-1 is (and is NOT)

- **IS:** a governed **control plane** for cadenced re-execution of already-shipped governed numbers. A `schedule` says "run family X for portfolio P, under model M in environment E, every N days from anchor A." A worker poll loop finds due schedules, dispatches each into its tenant, and appends a `scheduled_run` ledger row binding the tick to the governed `calculation_run` it produced.
- **IS NOT** a new governed number. It mints NO `model_version`, NO methodology validation record, NO new `run_type`. Every fire re-invokes an EXISTING family binder (`run_var`, …) which appends an ordinary immutable `CalculationRun` through the existing `execute_governed_run` scaffold. The scheduler is *provenance around* governed math, never governed math itself.
- **IS NOT a historical-series backfiller.** A scheduled fire RE-PINS FRESH over current data (a fresh number is inherently as-of-now). So the scheduler NEVER manufactures backdated numbers: each poll fires at most the CURRENT grid tick and COALESCES any missed intervals (Part 1 / OD-SCH-1-F). The daily VaR series FRTB wants accrues PROSPECTIVELY — one genuine as-of-that-day number per business day the schedule runs — it is not reconstructed by backfill.
- **IS NOT a BYPASSRLS business path.** The recommended dispatch (Option B) keeps the app **100% non-BYPASSRLS**: an infra-driven per-tenant invocation runs the whole poll+dispatch inside one tenant's RLS context. (The alternative Option A — an in-app ops-role cross-tenant enumeration — is a genuine doctrine expansion, not a free reuse; see OD-SCH-1-B.)

## Part 1 — The scheduling model (file:line grounded, SCH-1 census 2026-07-23)

**The tick loop (greenfield — none exists today; `main.run_once` at `apps/worker/src/irp_worker/main.py:12-14` is a pure `{"status":"idle"}` stub). Under the recommended Option B, `poll_tenant` runs entirely inside ONE tenant's non-BYPASSRLS `run_in_tenant` context; infra invokes it once per tenant:**

```
poll_tenant(session, now):                              # non-BYPASSRLS; RLS-scoped to this tenant
  for schedule in select_active_due(session, now):      # tenant-scoped read of this tenant's ACTIVE schedules
      try:
          dispatch_one(session, schedule, now)          # its own SAVEPOINT; a failure never starves siblings
      except Exception as exc:
          record_failed(session, schedule, now, exc)    # append a FAILED scheduled_run; CONTINUE the loop

current_tick(anchor, interval_days, now):               # PURE — the most recent grid point at/before now
  return anchor + floor((now - anchor) / interval_days) * interval_days

is_due(schedule, now, ledger):
  return schedule.status == 'ACTIVE' and now >= schedule.anchor_date \
     and current_tick(...) not in ledger[schedule.id]  # idempotency: has THIS grid tick already fired?
```

- **`select_active_due` — a TENANT-SCOPED read (Option B).** Under `run_in_tenant` (`apps/worker/src/irp_worker/jobs.py:20-24` → `db/tenant.py:137-142`; sets `app.current_tenant`, commits) the app role sees only this tenant's rows. It selects `schedule` rows with `status='ACTIVE'` whose `current_tick(anchor, interval_days, now)` has no matching `scheduled_run`. No cross-tenant read, no ops role. (Option A would instead do this cross-tenant under the BYPASSRLS ops role — see OD-SCH-1-B / OQ-1.)
- **The cadence bucket is `current_tick`, a PURE grid value — never a wall-clock read (INV-SCH-1).** `current_tick = anchor + floor((now−anchor)/interval)·interval`. This is the idempotency key component `scheduled_for`. It depends ONLY on `(anchor, interval, now)` — NOT on `MAX(scheduled_run.scheduled_for)`, so two concurrent polls compute the identical bucket and cannot diverge. A schedule N intervals overdue jumps straight to the current grid point and fires ONCE; missed grid points are simply never filled (a visible, honest gap in the ledger), NOT backfilled.
- **`dispatch_one` — the per-tenant governed execution (non-BYPASSRLS), each in its own SAVEPOINT.** (1) **idempotency guard** — if a `scheduled_run` for `(schedule_id, current_tick)` already exists, SKIP; the `UniqueConstraint(schedule_id, scheduled_for)` is the hard race backstop (a concurrent double-insert loses at COMMIT and the losing SAVEPOINT rolls back its phantom `CalculationRun` — the scaffold only `flush()`es, never commits: `calc/scaffold.py`, `db/tenant.py:138-142`). (2) **resolve upstream** — the latest COMPLETED **exposure** run for the SCOPE (`exposure/service.py:363 latest_exposure`, filtered by `scope_portfolio_id`) and the latest COMPLETED **covariance** run TENANT-GLOBAL by run_type (`risk/covariance_service.py:421 latest_covariances` — a covariance matrix is tenant-wide, takes NO scope arg); a thin adapter pulls `.calculation_run_id` off the newest `latest_run_rows(...)` row (ordered `system_from DESC, run_id DESC` — deterministic; an exact-`system_from` tie is broken by GUID, sub-second-unlikely). (3) **invoke the binder with the FULL contract** — `run_var(acting_tenant, actor=scheduler_actor, code_version, environment_id, model_version_id, exposure_run_id, covariance_run_id)` re-pins a FRESH `VAR_INPUT` snapshot over current data (`risk/var_service.py:729-739`, build args enforced non-None at `:820`). (4) **append the ledger row** — a `scheduled_run` binding `scheduled_for=current_tick` + `fired_at=now` + the resolved upstream run ids + the produced `calculation_run_id`.

## Part 2 — Rule-6 external benchmark grounding (research RAN 2026-07-23)

1. **BCBS 239 Principle 5 (Timeliness).** A bank must generate aggregate + up-to-date risk data on a TIMELY basis; the required frequency is driven by the risk's speed of change. THE supervisory basis for "governed numbers must be produced on a cadence, not only on manual demand." Directly motivates SCH-1's existence.
2. **SR 11-7 / OCC 2011-12 (Model Risk Management), "Ongoing Monitoring".** Models must be monitored on an ONGOING basis at a DEFINED CADENCE, not only at initial validation. VW-1 shipped the validation *event*; a scheduled re-run IS ongoing monitoring in the SR-11-7 sense. SCH-1 is what makes VW-1 monitoring and MG-2 remediation deadlines actually recurring rather than a manual promise.
3. **FRTB (BCBS d457, MAR) — DAILY desk-level backtesting.** The IMA requires DAILY VaR backtesting; the daily frequency is regulatory. **Honest scope (verifier fold):** SCH-1 serves this PROSPECTIVELY — a schedule run daily produces one genuine as-of-that-day fresh VaR per business day, and BT-2/BT-3 consume the series as it ACCRUES. SCH-1 does NOT reconstruct a historical daily series by backfill (a backfilled "series" of fresh re-pins would be N identical numbers wearing different date stamps — the no-backfill model, OD-SCH-1-F, exists precisely to forbid that).
4. **Job-scheduling reliability (SRE / distributed-systems).** A scheduled job WILL occasionally fire more than once (worker restart, overlap, retry) → execution must be idempotent OR the sink dedups on a stable key. Grounds the SEMANTICS decision: **at-least-once dispatch + a per-`(schedule, current_tick)` idempotency guard** (dedup-before-dispatch, with a `UniqueConstraint` hard backstop). Since the governed-run scaffold appends a NEW immutable `CalculationRun` per invocation, the guard lives at DISPATCH, before the binder is called.
5. **Reproducibility (the platform's own AD-014 / TR-09 pin doctrine).** A scheduler is inherently time-driven, so the "tick" is recorded AS DATA (`scheduled_for`/`fired_at`) and MUST NOT leak into the calculation's economic as-of. The census confirms NO injectable clock — `utcnow()` (`db/mixins.py:20-22`) is the only time source; reproducibility is the SNAPSHOT pin, not a clock. **INV-SCH-1 (elevated to ratified text on the verifier's flag):** `scheduled_for` MUST be the computed `current_tick` grid value, NEVER a wall-clock read — it is load-bearing for both idempotency (Finding 1) and the pure-function test firewall (Finding 4). The due-computation is a PURE function of `(anchor, interval, now, ledger)`; only the top-level poll reads `utcnow()` and threads it down — so the scheduler is fully testable WITHOUT a clock abstraction.
6. **Build-vs-buy (boring-base doctrine).** APScheduler / Celery-beat / Airflow / Temporal are the vendor options; a DB-backed `schedule` table + a worker poll loop is the HOUSE shape — deterministic, auditable, tenant-scoped, ZERO new infra/runtime dependency (consistent with the hand-rolled OIDC/PKCE flow and the zero-heavy-dep FE toolchain). v1 = interval cadence; calendar-aware business-day cadence (over the existing but logic-less `calendar`/`calendar_holiday` tables) is the recorded v2.

## Part 3 — Design decisions

### OD-SCH-1-A — Reuse the existing worker; add the poll loop to `main()` (fork OQ-SCH-1-2, low-stakes)
The census found `run_in_tenant`/`run_tenant_job` already exist as the AD-006-home per-tenant execution slot; only the loop is missing. Add it to `apps/worker/src/irp_worker/main.py` (the `audit_verify.py:47-48` inline `make_engine`/`make_session_factory` pattern — the worker has no `Settings` object and none is added). **Recommendation A: reuse the worker.**

### OD-SCH-1-B — The cross-tenant dispatch mechanism (**THE CRUX** — fork OQ-SCH-1-1; recommendation FLIPPED to B by the verifier pass)
There is no tenant registry and the app role cannot enumerate tenants (census §1). So "which tenants have due schedules" cannot be answered by the app role at all — SOMETHING must supply the tenant set. Two shapes:
- **B [rec] — infra-driven per-tenant invocation.** A deploy-layer mechanism (a per-tenant cron entry, or one cron iterating an infra-configured tenant list) invokes the worker once per tenant; the worker runs the WHOLE poll+dispatch inside that tenant's non-BYPASSRLS `run_in_tenant` context. The app stays **100% non-BYPASSRLS**; the standing `test_ops_role_has_no_grant_on_*` isolation doctrine is PRESERVED (SCH-1 adds its own such test). The tenant SET lives in infra config — the same place tenant provisioning, ops `DATABASE_URL`s, and secrets already live (tenant existence is already an infra/control-plane fact). Cost: the tenant list is an infra concern that must be kept in step with provisioning.
- **A — in-app ops-role cross-tenant enumeration.** `select_due` runs under the BYPASSRLS ops role (the `audit_verify` mechanism) reading the two scheduling tables cross-tenant. **The verifier REFUTED this as a settled reuse (Part 5): it is a genuine 3-part doctrine EXPANSION** — (i) AD-015's ratified scope is "audit verify / migration / diagnostics" (`11_decision_log/architecture_decision_log.md:23`), which does not cover dispatch; (ii) the `irp_ops` role is granted ONLY on `audit_event`/`audit_checkpoint` (`migrations/versions/0003_ops_role.py:36-37`), so `select_due` needs a first-ever `GRANT SELECT` on non-audit proprietary tables; (iii) that INVERTS the test-enforced invariant every P1A slice upholds (`test_ops_role_has_no_grant_on_model_tables` etc.). It also reads `scheduled_run.calculation_run_id` cross-tenant — governed-run provenance, not audit-integrity metadata. Option A is architecturally defensible (source-of-truth stays in the DB) but is a Tier-3 doctrine ratify that widens the BYPASSRLS surface for the first time.
- *(Considered and rejected — Option C, a SYSTEM-readable `tenant` registry the app could enumerate: it trades the ops-role expansion for a DIFFERENT closed-doctrine expansion — the "closed 5-table hybrid SYSTEM-readable set" — and adds a whole `tenant` primitive that is arguably its own slice. Net: it moves the expansion, doesn't remove it.)*
**Recommendation B** — it is the only option that keeps SCH-1's security posture identical to every prior slice (no BYPASSRLS surface change, no inverted test doctrine); the "tenant list in infra" cost is real but is where tenant existence already lives. **This is THE ratification fork — the user decides whether the app's non-BYPASSRLS purity (B) outweighs keeping the tenant source-of-truth in the DB (A).**

### OD-SCH-1-C — A new EV `schedule` (ENT-061) + IA `scheduled_run` (ENT-062), migration `0049`
Per Part 1. `schedule` = EV entity-versioned-in-place (config, edited via `record_version` bump); `scheduled_run` = IA append-only ledger with the `(schedule_id, scheduled_for)` unique idempotency key + the append-only trigger. Both FORCE-RLS tenant-scoped (proprietary data — symmetric RLS, NOT the closed 5-table hybrid set). The header carries `environment_id` (the verifier-flagged required governed-run pin). Next canonical ids ENT-061/062 (`04_data_model/canonical_data_model_standard.md:97`); next migration `0049`.

### OD-SCH-1-D — A scheduled run RE-PINS fresh via the full binder contract (fork OQ-SCH-1-3 scope)
The point of a cadence is FRESH numbers, so a fire uses the binder's BUILD-IN-REQUEST path (census §2): resolve the tenant's latest COMPLETED upstream runs and let the binder build a fresh input snapshot over current data. A schedule stores the *family + scope + model + environment*, NOT a fixed snapshot. **v1 scope = the VaR family only** (`run_var`; the flagship, and the number FRTB wants daily). The dispatch contract is fully specified in Part 1 (model/environment from the header; code_version from the build; actor synthesized; upstream resolved at tick — exposure scope-filtered, covariance tenant-global). Active-risk (needs a benchmark + effective date) and the other build-in-request families are the recorded v2 generalization (the mechanism is family-agnostic; only the upstream-resolution map differs). **Recommendation: re-pin fresh, VaR-only v1.**

### OD-SCH-1-E — At-least-once dispatch + a per-`(schedule, current_tick)` idempotency guard (Rule-6 Finding 4; INV-SCH-1)
A re-poll that finds an existing `(schedule_id, current_tick)` row SKIPS (`SKIPPED_DUPLICATE`); the `UniqueConstraint` is the hard race backstop (the loser's SAVEPOINT rolls back its phantom governed run — see Part 1). **`poll_tenant` isolates each schedule in its own try/except (verifier-flagged gap, folded): a unique-violation or any per-schedule error is swallowed as a recorded `FAILED`/`SKIPPED_DUPLICATE` outcome and the loop CONTINUES — one schedule's collision never starves the tenant's other due schedules.** The idempotency bucket is the pure `current_tick`, not a ledger `MAX` and not a wall-clock read (INV-SCH-1) — so it is collision-correct under concurrency.

### OD-SCH-1-F — No backfill: interval cadence, coalesce-to-current-tick v1; calendar-aware = v2 (fork OQ-SCH-1-3 cadence; folds verifier Findings 1B + 3B)
`cadence_kind=INTERVAL` (`interval_days` + `anchor_date`) v1. **The scheduler NEVER backfills.** Each poll fires at most the CURRENT grid tick (`current_tick`); missed grid points (worker downtime, or a pause window) are never filled — a visible, honest ledger gap. This folds BOTH blocking verifier defects at once: (1B) a schedule N intervals overdue does NOT emit N fresh-re-pinned clones wearing N historical date stamps (which would be a fraudulent daily series) — it fires ONE current as-of-now number; (3B) a resumed schedule computes `current_tick` from `now`, so it fires ONCE at the current interval — NOT a full-pause-window catch-up storm. Pause = `status='PAUSED'` (excluded from `select_active_due`); resume = `status='ACTIVE'`. Calendar-aware business-day cadence is deferred (the census confirmed NO business-day logic exists — `reference/models.py:106-113`; v2 must build `is_business_day`/`next_business_day` over the holiday-vocabulary substrate); `CALENDAR` is reserved in the `cadence_kind` vocab so v2 adds no migration. **Recommendation: no-backfill interval v1.**

### OD-SCH-1-G — R-07 permission mint `schedule.manage` + `schedule.view`
Creating/pausing a schedule is a NEW capability → a governed R-07 mint (census §3b): add `("schedule.manage", …)` + `("schedule.view", …)` to `entitlement/bootstrap.PERMISSIONS`, wire into `ROLE_TEMPLATES` (`.manage` to the run/ops-capable roles; `.view` broadly, including the auditor role — a schedule is control-plane config the auditor should see, the `perf.view`/`commitment.view` posture). Seeded by the existing migration `0002` live import on fresh provisioning; NO new role, NO entitlement DDL in `0049` (verifier CONFIRMED).

### OD-SCH-1-H — Schedule config changes emit a governed audit event = an R-07 audit-taxonomy mint (verifier fold: NOT trivial)
A schedule that DRIVES governed number production must itself be auditable (who scheduled the flagship VaR daily; when paused). The EV `schedule` create/update emits `SCHEDULE.CREATE`/`SCHEDULE.UPDATE` by CALLING the existing audit append (the `reference/calendar.py` mechanic) — this does NOT modify the FROZEN `audit/service.py` (event *types* are a free `String(100)` parameter, `audit/models.py:38`; no code-level whitelist rejects a new code). **BUT the verifier flagged that OD-H understated a governance gate:** there is NO `SCHEDULE` category in the governed `04_data_model/audit_event_taxonomy.md`, and every prior new code (`MODEL.TIER_ASSIGN`, the `PORTFOLIO.*` block) was a governed **R-07 taxonomy amendment** with an assigned EVT-index block — CLAUDE.md's invariant "no new audit code … outside the governed R-07 mint" applies. **Fold: SCH-1 mints a new `SCHEDULE` category (SCHEDULE.CREATE/UPDATE) as an R-07 taxonomy amendment recorded in `audit_event_taxonomy.md` at this ratification** — same governed path as MG-1/P1C-0. The scheduled RUNS need NO new audit code (they reuse `CALC.RUN_CREATE`/`CALC.RUN_STATUS_CHANGE` via the scaffold, tagged `initiated_by="scheduler:<schedule_id>"`).

### OD-SCH-1-I — Determinism: the tick is wall-clock evidence; the number's as-of is the pin (Rule-6 Finding 5; INV-SCH-1)
`fired_at` = `utcnow()` at dispatch, recorded on the append-only ledger as an operational fact. The governed VaR's economic as-of is the FRESH re-pinned snapshot, NEVER `fired_at`. `scheduled_for` = the pure `current_tick` (INV-SCH-1), never a wall-clock read — this is the load-bearing invariant that keeps both idempotency and the test firewall clean (verifier Finding 4, conditional-confirmed on exactly this). The due-computation is exercised in tests with explicit `now` values; no clock abstraction (none exists; none is invented).

### OD-SCH-1-J — A failing scheduled run records + continues; no auto-pause v1 (fork OQ-SCH-1-4)
A fire whose binder FAILS produces a governed FAILED `CalculationRun` (the scaffold writes FAILED + `failure_reason`) and a `scheduled_run` row with `outcome='FAILED'` linking it — durable evidence. v1 = **record + continue**: the schedule keeps firing on cadence; a persistently-failing schedule surfaces as a run of FAILED runs for LIM-1/MG-2 to alert on. NO auto-pause (it would MASK a systematic failure) and NO auto-retry storm (the next scheduled tick is the retry). **Recommendation: record + continue.**

### INV-SCH-1 (ratified invariant) — `scheduled_for := current_tick(anchor, interval, now)`, a pure grid value, NEVER a wall-clock read
Elevated to ratified text on the verifier's flag: it is simultaneously load-bearing for idempotency (Finding 1: two polls must compute the same bucket to collide) and the pure-function test firewall (Finding 4: an injected `now` must reconstruct the stored bucket). `fired_at`/`system_from`/`valid_from` are wall-clock; `scheduled_for` is grid. A test asserts a fire at an injected `now` writes exactly `current_tick`.

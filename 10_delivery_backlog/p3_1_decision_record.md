## Part 2 — Decision detail (the non-obvious ones)

### OD-P3-1-A — v1 sensitivity scope → curve-node analytic DV01 (the honest reconciliation)
P3-0 framed analytic sensitivities as computable "over the already-captured curve + positions, needs no new captured input." Grounding the shipped reality refines this: an *instrument/position-level* key-rate DV01 needs (i) per-instrument **cash-flow terms** (coupon schedule, maturity) — **not captured**; (ii) **curve interpolation** to discount cash flows at arbitrary dates — the curve module **deliberately has none** ("captured, never computed"); (iii) a **discounting/pricing engine** — explicitly **out of P3-1 scope** ("no valuation model"). Forcing instrument-level DV01 would balloon P3-1 into instrument-terms capture + an interpolation/pricing engine — violating the slice's exclusions. The **cleanly-reproducible, zero-new-data** number that still proves the entire contract is the **curve's own node-level analytic DV01** (the analytic building block of rate/spread key-rate risk): for a unit zero-coupon claim at node tenor `T`, `PV=DF(T)` and `∂PV/∂z = −T·DF`, so `DV01 = −T·DF·1bp` — closed-form, evaluated at the captured nodes, exactly reproducible. This is a genuine sensitivity, it is the literal foundation key-rate risk is assembled from, and it **honestly defers** instrument attribution (the user's "only if inputs support cleanly… or a smaller v1 subset" — the inputs do **not** cleanly support instrument-level, so the smaller curve-intrinsic subset is taken). **This is the #1 sign-off item (OQ-P3-1-1).**

### OD-P3-1-C — model-version-for-risk hardening (the load-bearing first)
Exposure (P2-3) was deterministic + model-less → `model_version_id` N/A. A sensitivity **applies a methodology** (conventions: bump, day-count, compounding — OD-P3-1-G) → it is **model-driven** (OD-P3-0-E) → it **must** bind a registered `model_version`. The binder therefore calls `assert_registered_model_version` in the **pre-create gate**: an unknown/unregistered version raises `UnregisteredModelError` and the run is **never created** (zero run/rows/audit) — making **CTRL-003 inventory-before-use executable for the first time**. This is the single most important governance hardening of P3-1: from here on, **no risk number escapes the model inventory**.

### OD-P3-1-E — snapshot curve pinning (curve-only, no migration)
`build_snapshot` today auto-selects kinds from a **portfolio scope**; a curve-intrinsic sensitivity has **no portfolio**, so P3-1 adds a `PURPOSE_SENSITIVITY_INPUT` build path keyed on a **curve set** (logical keys or ids) + an as-of, pinning each resolved curve version as one `COMPONENT_KIND_CURVE` component whose `captured_content` carries the **header immutable fields + the ordered `curve_point` node set** (one component per curve = the header+nodes unit; the `point_count`-on-header + immutable-nodes design makes this exact). `component_kind` is an unconstrained `String(50)` → **adding the kind is an app-constant change, no schema migration**. The verify path (`_reresolve_content`) re-resolves via `resolve_curve` + `list_curve_points` for hash stability.

### OD-P3-1-G — numerical conventions (why these)
**Analytic, not finite-difference:** a closed form (`−T·DF·1bp`) is exactly reproducible and needs no re-pricing/interpolation — the cleanest fit for "analytic sensitivities". **ACT/365F + continuous compounding** are the simplest unambiguous declared conventions that map `tenor_days`→`T` and `value_type`→`DF` with no captured convention column (the curve carries none); both are **declared in the methodology doc + `model_assumption`s** so a reader reproduces them. **`DISCOUNT_FACTOR` via implied `z=−ln(DF)/T`** lets the one closed form serve both rate value-types. **`PAR_RATE` deferred** (par→zero needs bootstrapping = curve construction, out of scope). **Evaluate at nodes only** keeps the no-interpolation invariant intact. **`quantize_HALF_UP(...,12)`** matches the captured `point_value Numeric(20,12)` precision and the exposure quantization precedent.

## Part 1 тАФ The `breach_action` model (file:line grounded, MG-2 census 2026-07-23)

**ENT-034 `breach_action`** (IA TRUE append-only тАФ the `ScheduledRun`/`model_validation` twin; the RTM prescribes IA, `requirements_backbone.md:220-221`): `PrimaryKeyMixin, TenantMixin, ImmutableAppendOnlyMixin, Base`; `__tablename__ = "breach_action"`. Columns:

- `breach_id GUID FK breach.id`, NOT NULL, indexed тАФ the breach this action transitions (re-resolved tenant-filtered pre-write; the P3-5 guard тАФ a `breach_id` from another tenant must be refused, mirroring LIM-1's `create_limit` FK guard).
- `action_type String(20)`, NOT NULL тАФ the transition verb тИИ `{ASSIGN, 1L_RESPONSE, 2L_REVIEW, ESCALATE, CLOSE}` (controlled vocab; `DETECTED` is the breach's own genesis, not an action).
- `from_state String(20)` + `to_state String(20)`, NOT NULL тАФ the recorded transition (both stored so the log is self-describing AND so the allowed-transition guard has the observed pre-state to check; the LIM-1 self-describing-echo doctrine).
- `actor_id String(255)`, NOT NULL тАФ the human principal who performed the action (the person-level SoD source; SYSTEM for auto-ESCALATE).
- `actor_line String(4)` nullable тАФ `1L`/`2L`/`SYS` line-of-defense tag (derived from the gating permission; convenience for reads).
- `assigned_to String(255)` nullable тАФ the 1L owner assigned (populated on ASSIGN).
- `response_due DateTime(tz)` nullable тАФ the deadline stamped at ASSIGN (a FIXED timestamp; the SCH-1-cadence deadline, compared `< tick_now`, NOT a `current_tick` grid тАФ SCH-1 recon finding).
- `narrative String(2000)` nullable тАФ the 1L remediation response / 2L review note / closure rationale.
- `review_outcome String(10)` nullable тАФ on 2L_REVIEW тИИ `{ACCEPT, REJECT}` (ACCEPT тЖТ REVIEWED; REJECT тЖТ back to ASSIGNED for re-response).
- `evidence_ref String(500)` nullable тАФ closure-evidence pointer, REQUIRED on CLOSE (REQ-BRC-003 "closure requires evidence").
- `occurred_at DateTime(tz)`, NOT NULL тАФ the ACTION's wall-clock (the tick's `now` for SYSTEM actions; request time for human actions) тАФ kept as evidence, but **NOT the recency-ordering key** (see `seq`).
- `seq Integer`, NOT NULL тАФ a **per-breach monotonic sequence**, the recency-ordering key (VERIFIER B-1 fold). The mixins provide only random `uuid4` `id` (`db/mixins.py:27-32`) and a `system_from` timestamp (`mixins.py:66-71`) тАФ NEITHER is a monotonic causal-order column, and MG-2 creates a genuine same-`occurred_at` tie (a SYSTEM auto-ESCALATE at the injected `tick_now` racing a human action at request-time == `tick_now`), so ordering by `occurred_at` + the random `id` tiebreaker would be nondeterministic. `seq` is **application-assigned under the parent-breach `FOR UPDATE` lock as `max(seq for this breach) + 1`** (starting at 1) тАФ race-free BECAUSE the lock serializes every append for a breach (and SQLite serializes all writes globally, so the invariant holds cross-tier without needing a PG-only `IDENTITY`/`FOR UPDATE`, which SQLite does not support). `UniqueConstraint(breach_id, seq)` pins it. Recency = `ORDER BY seq DESC LIMIT 1`. (Chosen over a DB `IDENTITY` precisely for SQLite-tier parity тАФ the CC-2 "PG-only behavior is invisible at the SQLite tier" lesson.)

**Append-only enforcement:** the ORM `_block_mutation` on `before_update`/`before_delete` (the LIM-1 `breach` pattern) + the migration-`0051` P0001 trigger. `breach_action` is TRUE append-only тАФ a correction is a new compensating action, never an edit. Consequently **`breach.status` (frozen at `DETECTED` by LIM-1, `limit/models.py:122-123`) is DEPRECATED-IN-PLACE for lifecycle purposes** (VERIFIER H-3 fold): it can never change, so MG-2's recency-derived state is the SOLE source of truth; `breach.status` is retained only as the detection-genesis marker and no reader may infer lifecycle from it. `limit_health` already recomputes from live values and does not read `breach.status` (`service.py:404-427`), so blast radius is nil today тАФ but the deprecation is stated so a future reader cannot reintroduce the stale-flag bug.

**The operative current state** (recency-derived тАФ the VW-1 pattern, NOT stored): `current_breach_state(session, breach_id, *, acting_tenant)` = the `to_state` of the latest `breach_action` for the breach `ORDER BY seq DESC LIMIT 1` (monotonic, deterministic), or `DETECTED` if no action exists yet. The read carries an **explicit `breach_action.tenant_id == acting_tenant` predicate** atop RLS (VERIFIER H-2 fold тАФ parity with `latest_validation` `validation.py:392-393` and `select_active_limits` `service.py:181-183`). `limit_health` (LIM-1's read surface) is extended to surface this state per breach.

**Per-breach write serialization (VERIFIER B-2/B-3/H-1 fold тАФ the linearizability backstop):** every transition (`_append_action`) takes a `SELECT тАж FOR UPDATE` on the PARENT `breach` row before reading current state and appending, so all transitions on one breach are strictly serialized even under concurrent per-tenant ticks (which SCH-1's own dedup machinery proves are a designed-for condition тАФ `scheduler.py:42-44`, `breaches.py:27-29`). This closes three read-then-write TOCTOU races the append-only log otherwise permits: (a) two concurrent ticks both escalating (B-2); (b) a stale-read `2L_REVIEW` resurrecting a concurrently-CLOSED breach (H-1); (c) an interleaved self-review slipping the SoD check (B-3). The lock is the primary guard; the constraint-backed escalate epoch (Part 3) and the all-responders SoD set (Part 2) are the belt-and-suspenders backstops.

**Migration `0051_breach_action`** (next free; latest `0050_limit_breach`): the `0050` boilerplate тАФ `TENANT_SCOPED_TABLES=(breach_action,)`, `APPEND_ONLY_TABLES=(breach_action,)`, the 63-char `_IDENTIFIERS` assert, the symmetric FORCE-RLS loop, the append-only trigger, `down_revision="0050_limit_breach"`. NO ops-role grant (the SCH-1/LIM-1 posture тАФ all reads/writes tenant-scoped). **If OQ-1=A**, `0051` also `ALTER`s `limit_definition` to add `created_by String(255)` (the drafter principal, for the approverтЙаdrafter SoD) тАФ nullable, backfilled NULL for pre-MG-2 rows (which are all already ACTIVE, so the approve gate never applies to them).

## Part 2 тАФ The state machine (the DEP-WFL realization)

**States:** `DETECTED` (genesis тАФ the LIM-1 terminal state) ┬╖ `ASSIGNED` ┬╖ `RESPONDED` ┬╖ `REVIEWED` ┬╖ `ESCALATED` ┬╖ `CLOSED` (terminal).

**The allowed-transition table** (a hardcoded frozenset `_ALLOWED: dict[tuple[from_state, action_type], to_state]` тАФ VW-1 has NO transition table to copy, this is the greenfield artifact; an illegal transition raises `BreachActionValueError` тЖТ 422, the fail-closed pre-write guard pattern):

| from_state | action_type | gate | to_state | notes |
|---|---|---|---|---|
| DETECTED | ASSIGN | `breach.review` (2L) | ASSIGNED | sets `assigned_to` (a 1L) + `response_due` |
| ASSIGNED | 1L_RESPONSE | `breach.respond` (1L) | RESPONDED | actor SHOULD be `assigned_to`; `narrative` required |
| ASSIGNED | ESCALATE | SYSTEM (deadline) or `breach.review` | ESCALATED | auto when `response_due < now`; idempotent |
| RESPONDED | 2L_REVIEW | `breach.review` (2L) | REVIEWED \| ASSIGNED | `review_outcome` ACCEPTтЖТREVIEWED, REJECTтЖТASSIGNED; **reviewer тЙа responder** (SOD-02) |
| RESPONDED | ESCALATE | SYSTEM or `breach.review` | ESCALATED | overdue review or manual |
| ESCALATED | 1L_RESPONSE | `breach.respond` (1L) | RESPONDED | recovery path |
| ESCALATED | 2L_REVIEW | `breach.review` (2L) | REVIEWED \| ASSIGNED | escalated breach can still be reviewed |
| REVIEWED | CLOSE | `breach.review` (2L) | CLOSED | `evidence_ref` required; **closer тЙа responder** (SOD-02) |
| CLOSED | тАФ | тАФ | тАФ | terminal; a re-breach is a NEW `breach` row (next tick's detection) |

**Person-level SoD (SOD-02, the platform's first same-actor refusal):** a shared helper `assert_not_prior_responder(session, breach_id, actor_id)` refuses a `2L_REVIEW` or `CLOSE` when `actor_id` is in the **SET of ALL prior `1L_RESPONSE` actors** on that breach тАФ NOT merely the latest one (VERIFIER B-3 fold: with a rejectтЖТre-respond cycle, X responds, 2L rejects, Y responds, then X reviews тАФ a latest-only check compares X against Y and wrongly allows it, though X was a responder; the all-responders set closes this). The comparison runs under the per-breach `FOR UPDATE` lock so it cannot be raced. This is the reusable primitive OQ-5's `limit.approve` also consumes (approver тЙа the `created_by` drafter). **`reviewer тЙа closer` is deliberately NOT required** (VERIFIER-confirmed defensible): both are 2L actions, so REQ-BRC-002 ("1L cannot approve own closure") is met by `closer тЙа any-1L-responder` alone; forcing two distinct 2L humans is a v2 stringency, recorded not shipped.

**The role-partition invariant (VERIFIER H6 fold, the SOD-03 precedent):** person-level SoD is the backstop, but the FIRST line of defense is that **no non-admin role may hold both `breach.respond` and `breach.review`** (exactly as no non-admin role holds both `model.inventory.register` and `model.validate`, `bootstrap.py:310-313,359-361`). A bootstrap conformance test pins this (the `test_model_validate_grants_as_ratified` precedent). Only `platform_admin` holds both тАФ and for the admin dual-hat, the person-level `assert_not_prior_responder` set-check IS the sole defense, which is exactly why it must be the all-responders set, not latest-only.

**AI-actor guard (BR-15):** every human transition asserts `actor_type == "user"` (the VW-1 `validation.py:243-247` guard) тАФ AI is maker-side only, never an approver/reviewer/closer. Auto-ESCALATE is the only non-user action and it is a SYSTEM escalation (never an approval).

## Part 3 тАФ Deadline enforcement (the third tick phase)

`poll_tenant_breach_deadlines(session, now, *, acting_tenant)` тАФ a THIRD phase of `run_operational_tick_for_tenant`, inserted after `poll_tenant_breaches` (`scheduler.py:140-141`). It mirrors the **phase-2 (`breaches.py`) single-layer shape** specifically (VERIFIER H4 fold тАФ NOT phase-1's two-layer `_record_failed` fresh-savepoint, which exists to write a FAILED ledger row that escalation has no analog for): `session.begin_nested()` per breach; a single benign-dedup on the ESCALATE idempotency constraint; everything else LOGGED not masked; nothing escapes the per-item handler, so `run_in_tenant`'s single terminal commit (`scheduler.py:143`) always lands every successful sibling escalation:

```python
for breach in _open_overdue_breaches(session, now, acting_tenant=acting_tenant):
    savepoint = session.begin_nested()
    try:
        _append_action(session, breach, ESCALATE, actor_id="SYSTEM", to_state=ESCALATED, occurred_at=now)   # takes FOR UPDATE on breach; BREACH.ESCALATE
        savepoint.commit()
    except IntegrityError as exc:
        savepoint.rollback()
        if not _is_escalate_dedup(exc): logging.getLogger(__name__).warning(...)     # constraint-specific (uq_breach_escalation), LIM-1 lesson
    except Exception: savepoint.rollback(); logging.getLogger(__name__).exception(...)
```

**Phase ordering (VERIFIER H2 correction):** placing phase 3 after phase 2 is NOT for same-tick escalation of a just-detected breach тАФ a breach DETECTED in phase 2 has state `DETECTED` and NO `response_due` (stamped only at the human ASSIGN, never in a tick), so it is never escalation-eligible in the same tick. `_open_overdue_breaches` only ever acts on breaches ASSIGNED in a prior committed request/tick; the ordering is immaterial to escalation correctness, phase 3 goes last simply as the read-of-prior-state sweep. `_work` (`scheduler.py:136-141`) MUST add an `"escalated"` key to its returned dict and `main()` (`scheduler.py:173-175`) surface an escalation count, so tick observability isn't dropped (VERIFIER H3 fold).

**Deadline SOURCE (OQ-4):** `response_due` is stamped at ASSIGN = `assigned_at + SLA(limit_kind)`, where `SLA` is a hardcoded `{HARD: <short>, SOFT: <longer>}` day-count map v1 (a per-limit configurable SLA column is a recorded v2). "Overdue" is a plain `response_due < now` comparison against the injected tick `now` (the `next_review_due < today` precedent, `api/models.py:225-227`) тАФ MG-2 mints NO grid math; `current_tick` is the wrong shape for a fixed deadline (SCH-1 recon).

**The ESCALATE idempotency backstop (VERIFIER B-2 fold тАФ ONE constraint-backed mechanism, not an unresolved "OR"):** a partial unique index **`uq_breach_escalation` on `(breach_id, response_due)`** among ESCALATE actions тАФ a breach escalates AT MOST ONCE per deadline epoch. `response_due` IS the natural epoch (the grid `scheduled_for` analog SCH-1's idempotency needs): a breach that stays overdue across many ticks re-selects, but the second insert hits `uq_breach_escalation` and is swallowed as benign dedup (`_is_escalate_dedup`, constraint-specific тАФ the SCH-1 no-storm discipline). A LEGITIMATE re-escalation after recovery is a DIFFERENT epoch: the recovery path (ESCALATED тЖТ `1L_RESPONSE` тЖТ 2L REJECT тЖТ `ASSIGN`) stamps a FRESH `response_due`, so its later overdue is a distinct `(breach_id, response_due)` and escalates once more тАФ the two mechanisms the draft left unreconciled now agree. The latest-state pre-check in `_open_overdue_breaches` (state тИИ {ASSIGNED, RESPONDED}, excludes ESCALATED) stays as the read-side optimization; the unique index + the per-breach `FOR UPDATE` lock (Part 1) are the correctness backstops against the concurrent-tick TOCTOU the draft's pre-check-alone left open.

## Part 0 ‚Äî What MG-2 is (and is NOT)

- **IS** the breach **remediation lifecycle**: the reserved **ENT-034 `breach_action`** (IA TRUE append-only) realized as a **transition-event log** ‚Äî one row per state transition, the operative CURRENT state DERIVED by recency (the latest action's `to_state`), NEVER a mutated status column. This is forced by two facts and one lesson: (1) `breach` is IA append-only (`limit/models.py:126-135` ‚Äî `before_update` raises `AppendOnlyViolation`), so a breach's lifecycle CANNOT be an in-place status flip; (2) VW-1's `model_validation` is the shipped precedent for exactly this shape (an IA judgment log, state = latest record, `model/validation.py:379-397`); (3) the LIM-1 headline lesson ‚Äî a fail-open control's state must be RECOMPUTED from the source of truth, never inferred from a stored flag.
- **IS** the platform's **FIRST person-level SoD** (approver/reviewer ‚âÝ author/responder, compared by principal id). Everything shipped to date is role-partition SoD only (SOD-03: 1L holds `model.inventory.register`, 2L holds `model.validate`, no non-admin role holds both ‚Äî `bootstrap.py`; but NO code anywhere compares actor principal ids). REQ-BRC-002 (SOD-02) requires the stronger form: "1L cannot approve own closure." MG-2 introduces the reusable same-actor refusal primitive.
- **IS** deadline enforcement with TEETH, riding the SCH-1 cadence: a THIRD phase of `run_operational_tick_for_tenant` (`scheduler.py:136-141`) ‚Äî `poll_tenant_breach_deadlines` ‚Äî auto-ESCALATES any breach whose response deadline has passed, mirroring the SCH-1/LIM-1 per-item SAVEPOINT isolation. This directly fixes the toothless-overdue asymmetry VW-1/MG-1 left open (overdue PERIODIC revalidation is display-only, `mg_1_decision_record.md` OD-F; the "earliest real overdue 2027-07-19" erosion trigger).
- **IS a REALIZATION, not a taxonomy mint.** Every audit code MG-2 emits is already reserved in the genesis EVT-070 decade (`audit_event_taxonomy.md:66`): `BREACH.ASSIGN`, `BREACH.1L_RESPONSE`, `BREACH.2L_REVIEW`, `BREACH.ESCALATE`, `BREACH.CLOSE`. MG-2 activates them caller-side against the FROZEN `record_event` (the LIM-1 pattern). ## Part 1 ‚Äî The `breach_action` model (file:line grounded, MG-2 census 2026-07-23)

**ENT-034 `breach_action`** (IA TRUE append-only ‚Äî the `ScheduledRun`/`model_validation` twin; the RTM prescribes IA, `requirements_backbone.md:220-221`): `PrimaryKeyMixin, TenantMixin, ImmutableAppendOnlyMixin, Base`; `__tablename__ = "breach_action"`. Columns:

- `breach_id GUID FK breach.id`, NOT NULL, indexed ‚Äî the breach this action transitions (re-resolved tenant-filtered pre-write; the P3-5 guard ‚Äî a `breach_id` from another tenant must be refused, mirroring LIM-1's `create_limit` FK guard).
- `action_type String(20)`, NOT NULL ‚Äî the transition verb ‚àà `{ASSIGN, 1L_RESPONSE, 2L_REVIEW, ESCALATE, CLOSE}` (controlled vocab; `DETECTED` is the breach's own genesis, not an action).
- `from_state String(20)` + `to_state String(20)`, NOT NULL ‚Äî the recorded transition (both stored so the log is self-describing AND so the allowed-transition guard has the observed pre-state to check; the LIM-1 self-describing-echo doctrine).
- `actor_id String(255)`, NOT NULL ‚Äî the human principal who performed the action (the person-level SoD source; SYSTEM for auto-ESCALATE).
- `actor_line String(4)` nullable ‚Äî `1L`/`2L`/`SYS` line-of-defense tag (derived from the gating permission; convenience for reads).
- `assigned_to String(255)` nullable ‚Äî the 1L owner assigned (populated on ASSIGN).
- `response_due DateTime(tz)` nullable ‚Äî the deadline stamped at ASSIGN (a FIXED timestamp; the SCH-1-cadence deadline, compared `< tick_now`, NOT a `current_tick` grid ‚Äî SCH-1 recon finding).
- `narrative String(2000)` nullable ‚Äî the 1L remediation response / 2L review note / closure rationale.
- `review_outcome String(10)` nullable ‚Äî on 2L_REVIEW ‚àà `{ACCEPT, REJECT}` (ACCEPT ‚Üí REVIEWED; REJECT ‚Üí back to ASSIGNED for re-response).
- `evidence_ref String(500)` nullable ‚Äî closure-evidence pointer, REQUIRED on CLOSE (REQ-BRC-003 "closure requires evidence").
- `occurred_at DateTime(tz)`, NOT NULL ‚Äî the ACTION's wall-clock (the tick's `now` for SYSTEM actions; request time for human actions) ‚Äî kept as evidence, but **NOT the recency-ordering key** (see `seq`).
- `seq Integer`, NOT NULL ‚Äî a **per-breach monotonic sequence**, the recency-ordering key (VERIFIER B-1 fold). The mixins provide only random `uuid4` `id` (`db/mixins.py:27-32`) and a `system_from` timestamp (`mixins.py:66-71`) ‚Äî NEITHER is a monotonic causal-order column, and MG-2 creates a genuine same-`occurred_at` tie (a SYSTEM auto-ESCALATE at the injected `tick_now` racing a human action at request-time == `tick_now`), so ordering by `occurred_at` + the random `id` tiebreaker would be nondeterministic. `seq` is **application-assigned under the parent-breach `FOR UPDATE` lock as `max(seq for this breach) + 1`** (starting at 1) ‚Äî race-free BECAUSE the lock serializes every append for a breach (and SQLite serializes all writes globally, so the invariant holds cross-tier without needing a PG-only `IDENTITY`/`FOR UPDATE`, which SQLite does not support). `UniqueConstraint(breach_id, seq)` pins it. Recency = `ORDER BY seq DESC LIMIT 1`. (Chosen over a DB `IDENTITY` precisely for SQLite-tier parity ‚Äî the CC-2 "PG-only behavior is invisible at the SQLite tier" lesson.)

**Append-only enforcement:** the ORM `_block_mutation` on `before_update`/`before_delete` (the LIM-1 `breach` pattern) + the migration-`0051` P0001 trigger. `breach_action` is TRUE append-only ‚Äî a correction is a new compensating action, never an edit. Consequently **`breach.status` (frozen at `DETECTED` by LIM-1, `limit/models.py:122-123`) is DEPRECATED-IN-PLACE for lifecycle purposes** (VERIFIER H-3 fold): it can never change, so MG-2's recency-derived state is the SOLE source of truth; `breach.status` is retained only as the detection-genesis marker and no reader may infer lifecycle from it. `limit_health` already recomputes from live values and does not read `breach.status` (`service.py:404-427`), so blast radius is nil today ‚Äî but the deprecation is stated so a future reader cannot reintroduce the stale-flag bug.

**The operative current state** (recency-derived ‚Äî the VW-1 pattern, NOT stored): `current_breach_state(session, breach_id, *, acting_tenant)` = the `to_state` of the latest `breach_action` for the breach `ORDER BY seq DESC LIMIT 1` (monotonic, deterministic), or `DETECTED` if no action exists yet. The read carries an **explicit `breach_action.tenant_id == acting_tenant` predicate** atop RLS (VERIFIER H-2 fold ‚Äî parity with `latest_validation` `validation.py:392-393` and `select_active_limits` `service.py:181-183`). `limit_health` (LIM-1's read surface) is extended to surface this state per breach.

**Per-breach write serialization (VERIFIER B-2/B-3/H-1 fold ‚Äî the linearizability backstop):** every transition (`_append_action`) takes a `SELECT ‚Ä¶ FOR UPDATE` on the PARENT `breach` row before reading current state and appending, so all transitions on one breach are strictly serialized even under concurrent per-tenant ticks (which SCH-1's own dedup machinery proves are a designed-for condition ‚Äî `scheduler.py:42-44`, `breaches.py:27-29`). This closes three read-then-write TOCTOU races the append-only log otherwise permits: (a) two concurrent ticks both escalating (B-2); (b) a stale-read `2L_REVIEW` resurrecting a concurrently-CLOSED breach (H-1); (c) an interleaved self-review slipping the SoD check (B-3). The lock is the primary guard; the constraint-backed escalate epoch (Part 3) and the all-responders SoD set (Part 2) are the belt-and-suspenders backstops.

**Migration `0051_breach_action`** (next free; latest `0050_limit_breach`): the `0050` boilerplate ‚Äî `TENANT_SCOPED_TABLES=(breach_action,)`, `APPEND_ONLY_TABLES=(breach_action,)`, the 63-char `_IDENTIFIERS` assert, the symmetric FORCE-RLS loop, the append-only trigger, `down_revision="0050_limit_breach"`. NO ops-role grant (the SCH-1/LIM-1 posture ‚Äî all reads/writes tenant-scoped). **If OQ-1=A**, `0051` also `ALTER`s `limit_definition` to add `created_by String(255)` (the drafter principal, for the approver‚âÝdrafter SoD) ‚Äî nullable, backfilled NULL for pre-MG-2 rows (which are all already ACTIVE, so the approve gate never applies to them).

## Part 2 ‚Äî The state machine (the DEP-WFL realization)

**States:** `DETECTED` (genesis ‚Äî the LIM-1 terminal state) ¬∑ `ASSIGNED` ¬∑ `RESPONDED` ¬∑ `REVIEWED` ¬∑ `ESCALATED` ¬∑ `CLOSED` (terminal).

**The allowed-transition table** (a hardcoded frozenset `_ALLOWED: dict[tuple[from_state, action_type], to_state]` ‚Äî VW-1 has NO transition table to copy, this is the greenfield artifact; an illegal transition raises `BreachActionValueError` ‚Üí 422, the fail-closed pre-write guard pattern):

| from_state | action_type | gate | to_state | notes |
|---|---|---|---|---|
| DETECTED | ASSIGN | `breach.review` (2L) | ASSIGNED | sets `assigned_to` (a 1L) + `response_due` |
| ASSIGNED | 1L_RESPONSE | `breach.respond` (1L) | RESPONDED | actor SHOULD be `assigned_to`; `narrative` required |
| ASSIGNED | ESCALATE | SYSTEM (deadline) | ESCALATED | auto when `response_due < now`; idempotent per epoch |
| RESPONDED | 2L_REVIEW | `breach.review` (2L) | REVIEWED \| ASSIGNED | requires a prior 1L response; ACCEPT‚ÜíREVIEWED, REJECT‚ÜíASSIGNED; **reviewer ‚âÝ any prior responder** (SOD-02) |
| RESPONDED | ESCALATE | SYSTEM (deadline) | ESCALATED | overdue 2L review |
| ESCALATED | 1L_RESPONSE | `breach.respond` (1L) | RESPONDED | recovery path |
| ESCALATED | 2L_REVIEW | `breach.review` (2L) | REVIEWED \| ASSIGNED | escalated breach can still be reviewed ‚Äî **but only if a 1L response exists** (an unresponded/escalated breach cannot be reviewed, VERIFIER-F1-HIGH1) |
| REVIEWED | CLOSE | `breach.review` (2L) | CLOSED | `evidence_ref` required; **closer ‚âÝ any prior responder** (SOD-02) |

*(v1 escalation is SYSTEM-only ‚Äî the auto deadline sweep; a manual 2L escalate verb is a recorded v2, not shipped. Idempotency epoch = the governing ASSIGN action's `seq` (`epoch_seq`), a true monotonic key ‚Äî VERIFIER-F1-MED1.)*
| CLOSED | ‚Äî | ‚Äî | ‚Äî | terminal; a re-breach is a NEW `breach` row (next tick's detection) |

**Person-level SoD (SOD-02, the platform's first same-actor refusal):** a shared helper `assert_not_prior_responder(session, breach_id, actor_id)` refuses a `2L_REVIEW` or `CLOSE` when `actor_id` is in the **SET of ALL prior `1L_RESPONSE` actors** on that breach ‚Äî NOT merely the latest one (VERIFIER B-3 fold: with a reject‚Üíre-respond cycle, X responds, 2L rejects, Y responds, then X reviews ‚Äî a latest-only check compares X against Y and wrongly allows it, though X was a responder; the all-responders set closes this). The comparison runs under the per-breach `FOR UPDATE` lock so it cannot be raced. This is the reusable primitive OQ-5's `limit.approve` also consumes (approver ‚âÝ the `created_by` drafter). **`reviewer ‚âÝ closer` is deliberately NOT required** (VERIFIER-confirmed defensible): both are 2L actions, so REQ-BRC-002 ("1L cannot approve own closure") is met by `closer ‚âÝ any-1L-responder` alone; forcing two distinct 2L humans is a v2 stringency, recorded not shipped.

**The role-partition invariant (VERIFIER H6 fold, the SOD-03 precedent):** person-level SoD is the backstop, but the FIRST line of defense is that **no non-admin role may hold both `breach.respond` and `breach.review`** (exactly as no non-admin role holds both `model.inventory.register` and `model.validate`, `bootstrap.py:310-313,359-361`). A bootstrap conformance test pins this (the `test_model_validate_grants_as_ratified` precedent). Only `platform_admin` holds both ‚Äî and for the admin dual-hat, the person-level `assert_not_prior_responder` set-check IS the sole defense, which is exactly why it must be the all-responders set, not latest-only.

**AI-actor guard (BR-15):** every human transition asserts `actor_type == "user"` (the VW-1 `validation.py:243-247` guard) ‚Äî AI is maker-side only, never an approver/reviewer/closer. Auto-ESCALATE is the only non-user action and it is a SYSTEM escalation (never an approval).

## Part 3 ‚Äî Deadline enforcement (the third tick phase)

`poll_tenant_breach_deadlines(session, now, *, acting_tenant)` ‚Äî a THIRD phase of `run_operational_tick_for_tenant`, inserted after `poll_tenant_breaches` (`scheduler.py:140-141`). It mirrors the **phase-2 (`breaches.py`) single-layer shape** specifically (VERIFIER H4 fold ‚Äî NOT phase-1's two-layer `_record_failed` fresh-savepoint, which exists to write a FAILED ledger row that escalation has no analog for): `session.begin_nested()` per breach; a single benign-dedup on the ESCALATE idempotency constraint; everything else LOGGED not masked; nothing escapes the per-item handler, so `run_in_tenant`'s single terminal commit (`scheduler.py:143`) always lands every successful sibling escalation:

```python
for breach in _open_overdue_breaches(session, now, acting_tenant=acting_tenant):
    savepoint = session.begin_nested()
    try:
        _append_action(session, breach, ESCALATE, actor_id="SYSTEM", to_state=ESCALATED, occurred_at=now)   # takes FOR UPDATE on breach; BREACH.ESCALATE
        savepoint.commit()
    except IntegrityError as exc:
        savepoint.rollback()
        if not _is_escalate_dedup(exc): logging.getLogger(__name__).warning(...)     # constraint-specific (uq_breach_escalation), LIM-1 lesson
    except Exception: savepoint.rollback(); logging.getLogger(__name__).exception(...)
```

**Phase ordering (VERIFIER H2 correction):** placing phase 3 after phase 2 is NOT for same-tick escalation of a just-detected breach ‚Äî a breach DETECTED in phase 2 has state `DETECTED` and NO `response_due` (stamped only at the human ASSIGN, never in a tick), so it is never escalation-eligible in the same tick. `_open_overdue_breaches` only ever acts on breaches ASSIGNED in a prior committed request/tick; the ordering is immaterial to escalation correctness, phase 3 goes last simply as the read-of-prior-state sweep. `_work` (`scheduler.py:136-141`) MUST add an `"escalated"` key to its returned dict and `main()` (`scheduler.py:173-175`) surface an escalation count, so tick observability isn't dropped (VERIFIER H3 fold).

**Deadline SOURCE (OQ-4):** `response_due` is stamped at ASSIGN = `assigned_at + SLA(limit_kind)`, where `SLA` is a hardcoded `{HARD: <short>, SOFT: <longer>}` day-count map v1 (a per-limit configurable SLA column is a recorded v2). "Overdue" is a plain `response_due < now` comparison against the injected tick `now` (the `next_review_due < today` precedent, `api/models.py:225-227`) ‚Äî MG-2 mints NO grid math; `current_tick` is the wrong shape for a fixed deadline (SCH-1 recon).

**The ESCALATE idempotency backstop (VERIFIER B-2 fold ‚Äî ONE constraint-backed mechanism, not an unresolved "OR"; the epoch KEY was later refined `response_due`‚Üí`epoch_seq` at the 4-finder fold, Part 6 F1-MED1):** a partial unique index **`uq_breach_escalation` on `(breach_id, epoch_seq)`** among ESCALATE actions (`epoch_seq` = the governing ASSIGN action's monotonic `seq`) ‚Äî a breach escalates AT MOST ONCE per epoch. `response_due` IS the natural epoch (the grid `scheduled_for` analog SCH-1's idempotency needs): a breach that stays overdue across many ticks re-selects, but the second insert hits `uq_breach_escalation` and is swallowed as benign dedup (`_is_escalate_dedup`, constraint-specific ‚Äî the SCH-1 no-storm discipline). A LEGITIMATE re-escalation after recovery is a DIFFERENT epoch: the recovery path (ESCALATED ‚Üí `1L_RESPONSE` ‚Üí 2L REJECT ‚Üí `ASSIGN`) stamps a FRESH `response_due`, so its later overdue is a distinct `(breach_id, response_due)` and escalates once more ‚Äî the two mechanisms the draft left unreconciled now agree. The latest-state pre-check in `_open_overdue_breaches` (state ‚àà {ASSIGNED, RESPONDED}, excludes ESCALATED) stays as the read-side optimization; the unique index + the per-breach `FOR UPDATE` lock (Part 1) are the correctness backstops against the concurrent-tick TOCTOU the draft's pre-check-alone left open.
